# Superpowers 软件开发方法论 v5.1.0

> 来源：https://github.com/obra/superpowers · MIT · Jesse Vincent & Prime Radiant 团队
> 这是一套为编码代理设计的完整方法论。不是建议，是强制工作流。

## 核心理念

| 原则 | 含义 |
|------|------|
| 测试驱动开发 (TDD) | 永远先写测试。没有失败的测试，绝不写生产代码。 |
| 系统化优于临时性 | 按流程走，不凭猜测。强制工作流不可跳过。 |
| 降低复杂度 | 简单是首要目标。DRY、YAGNI。 |
| 证据优于声明 | 先验证再宣布成功。 |

## 强制 7 步工作流

每次编码任务严格按此顺序执行：

1. **头脑风暴** — 写代码前先做设计推敲：探索意图、提 2-3 种方案、展示设计。**硬性门禁：设计未确认前不实施。**
2. **Git Worktree 隔离** — 新分支上隔离工作，建立干净测试基线。
3. **编写计划** — 拆解为 2-5 分钟小任务，每任务含精确文件路径、完整代码、验证步骤。
4. **子代理驱动开发** — 每任务派全新子代理（隔离上下文），配两阶段审查：需求合规性 → 代码质量。持续执行，不问"是否继续"。
5. **TDD** — RED → GREEN → REFACTOR。**铁律：没有失败的测试，绝不写生产代码。测试之前写的代码必须删除。**
6. **代码审查** — 任务间审查，关键问题阻塞进度。
7. **收尾** — 验证测试、合并/PR/保留/丢弃、清理 worktree。

## TDD 铁律

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
```

1. **RED** — 写失败的测试。验证它正确失败（不是因为拼写错误）。
2. **GREEN** — 写最小代码让测试通过。不添加功能、不重构其他代码。
3. **REFACTOR** — 仅在绿色后清理。消除重复、改进命名。不动行为。

**测试之前写的实现代码 → 删除。重新开始。不保留为"参考"。**

### 好测试标准

- 一个行为、清晰命名、真实代码（不用模拟除非不可避免）
- 命名描述行为："test_retries_failed_operations_3_times"
- 显示意图：展示期望的 API

## 子代理驱动开发

每任务全新子代理，不继承会话历史。持续执行，不问"是否继续"。

```
对于每个任务：
  派实施子代理（完整任务文本 + 上下文）
  → 派规格审查子代理（验证实现匹配需求）
  → 派代码质量审查子代理
  → 通过 → 继续下一个任务
```

**模型选择（硬约束）：**

| 任务类型 | 模型 |
|----------|------|
| 搜索、读文件、grep、glob、格式化 | 最便宜模型 |
| 单文件修改、简单重构、写测试、代码审查 | 标准模型 |
| 架构设计、多文件变更、安全审查、调试根因 | 最强模型 |

## 系统化调试四阶段

1. **根因调查** — 读错误消息、复现、检查变更、收集证据。铁律：未做根因调查不得修复。
2. **模式分析** — 找工作中的例子、对照参考比较、识别差异。
3. **假设与测试** — 形成单个假设、最小化测试。一次一个变量。
4. **实施** — 创建失败测试→修复→验证。**铁律：3 次修复失败后，停下来质疑架构。**

## 完成前验证

```
NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE
```

- 声称测试通过 → 必须看到测试命令输出显示 0 失败
- 声称 bug 已修复 → 必须测试原始症状
- 不能用"应该"、"大概"、"似乎"
- 信任代理报告 → 检查 VCS diff → 验证变更 → 才可声明

## 关键原则

- **技能强制性**：任何技能有 ≥1% 适用可能就必须调用
- **一口大小**：每任务 2-5 分钟，每步一个动作
- **零占位符**：计划中绝不写 "TBD"、"TODO"、"稍后实现"
- **最小改动**：只改需要改的，不动无关代码
- **文件按职责拆分**：一起变化的文件放一起，按职责而非技术层拆分
- **2 次试探上限**：同一问题试 2 次还不对，回到根因调查
- **3 次失败质疑架构**：不是继续修，是退一步看全局
- **测试反模式**：不测试模拟行为、不向生产类添加仅测试方法、不理解依赖不模拟
