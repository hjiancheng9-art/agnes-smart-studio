# 编辑精度契约 — 读后编 · 补丁优先 · 零扰动

融合 Codex apply_patch 的工程约束，确保 CRUX 的代码编辑达到同等精度。
此规则 default-active，自动注入所有 code_mode 会话。

## 一、读后编（Read-Before-Edit）

编辑任何文件前，必须先用 read_file 读取当前内容。不允许：
- 凭记忆或上下文猜测文件当前状态
- 在未确认文件存在性的情况下发出 edit_file / patch_file
- 连续编辑同一文件时不重新读取（文件可能已被上一轮修改改变）

> 反模式：LLM 假设"上次读过的文件没变"，连续 3 轮编辑同一文件不重读，第三轮必然漂移。

## 二、补丁优先于覆写（Patch over Write）

- **结构化补丁**：多文件、跨模块修改，用 patch_file（走 PatchEngine），失败自动回滚
- **单文件小改**：用 edit_file，传入精确的 old_str / new_str，不传整文件
- **禁止覆写**：绝不用 write_file 覆写核心模块（core/*.py），除非是新建文件
- **old_str 必须唯一**：传给 edit_file 的 old_str 必须在目标文件中只出现一次；如果重复，用更多上下文行锚定

## 三、零元数据扰动（No Metadata Churn）

- 不修改与任务无关的 import 顺序、空白行、注释格式
- 不自动格式化未触及的代码区域
- 不顺手将单引号改成双引号、不调整未改函数的参数排列
- 文件末尾的换行符状态保持不变（有就有，没有不加）

## 四、抽象克制（Abstraction Restraint）

- 只在消除真实重复或显著降低复杂度时才引入新抽象
- 不为"未来可能的需求"创建基类、接口、工具函数
- 新代码遵循文件内已有的模式（命名、错误处理、装饰器风格），不引入新范式
- 如果现有工具函数能完成任务，不从头实现

## 五、编辑半径最小化

- 一次 patch 只解决一个问题，不顺手改无关代码
- 变量重命名仅限于当前改动的逻辑范围内，不跨模块传播
- 删除代码前确认无引用（grep / 符号索引）

## 六、验证闭环

- Python 文件编辑后立即 ast.parse 校验语法
- JSON 文件编辑后立即 json.loads 校验结构
- 涉及 imports 的改动，确认模块路径在新位置下可达
- 编辑超过 20 行时，先展示 diff plan，确认后再执行

## 七、patch 格式规范

使用结构化补丁时遵循 Codex apply_patch 格式：
- `@@` 后跟多行上下文作为锚点，不只一行
- `-` 删除行，`+` 新增行，空格行保持不变
- 上下文包含足够的缩进层级，确保匹配唯一

> 约束：edit_file 的 old_str 若在该文件中出现超过 1 次，必须扩充上下文直到唯一。
