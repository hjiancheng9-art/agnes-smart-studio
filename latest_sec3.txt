3. ComfyUI API 最新用法
3.1 /prompt 仍是核心入口

官方 server routes 说明：前端提交 workflow 时会 POST 到 /prompt，它会验证 prompt 并加入执行队列；成功返回 prompt_id 和 number，失败返回 error 和 node_errors。
ComfyUI

推荐 payload：

JSON
{
  "prompt": {
    "3": {
      "class_type": "KSampler",
      "inputs": {}
    }
  },
  "client_id": "uuid",
  "prompt_id": "uuid"
}

ComfyUI 官方 websocket 示例也是这么提交：{"prompt": prompt, "client_id": client_id, "prompt_id": prompt_id}。
GitHub

3.2 WebSocket 必须接

不要只轮询 /history。

/ws 官方用于实时双向通信，可以接收执行进度、节点状态、错误、队列更新。消息类型包括：

status
execution_start
execution_cached
executing
progress
executed

官方 websocket 示例通过监听 executing，当 data.node is None 且 prompt_id 匹配时判断执行完成，然后再走 /history/{prompt_id} 获取结果。

3.4 Python SDK 状态

截至现在，本地 ComfyUI 没有一个必须采用的官方 Python SDK。官方主要给的是脚本示例和 comfy-cli。comfy-cli 是官方命令行工具，可安装、运行、扩展 ComfyUI，安装 custom nodes、models、运行 workflow、调用 partner models。
GitHub

第三方 Python 包有 comfyui-api-client，PyPI 说明它通过 WebSocket 和 HTTP 请求调用 ComfyUI，但生产系统建议你们自己封装一层 client，避免被第三方 SDK 版本绑定。
