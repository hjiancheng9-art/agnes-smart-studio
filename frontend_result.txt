2. brain.py 376 行：LLM 连接器和 agent.py 的关系
结论：brain.py 应该是底层适配器，不应该反向知道 agent.py。
理想关系应该是：
agent_flow.py / workflow_builder.py        ↓   BrainClient 接口        ↓brain.py 具体实现 OpenAI-compatible API
也就是说：
agent.py 不应该“拥有大脑逻辑”agent_flow.py 可以调用 brain.pybrain.py 不能调用 agent.py
如果现在是：
Python运行agent.py -> brain.py -> prompt -> workflow json
还可以接受。
但如果是：
Python运行brain.py 知道 ComfyUIAgentbrain.py 直接改 workflowbrain.py 直接调用 executorbrain.py 直接读取 config 全局变量
那就危险。
brain.py 应该负责
- API endpoint- model name- api key- timeout- retry- structured output- JSON repair- token/cost 记录- provider failover
brain.py 不应该负责
- ComfyUI 节点语义- 工作流拓扑修复- 模板选择- 模型库存判断- 文件下载- 启动 ComfyUI- 直接操作 workflow graph
推荐抽象：
Python运行class LLMClient:    def complete(self, messages, schema=None) -> LLMResult:        ...
然后业务层只依赖接口：
Python运行class WorkflowPlanner:    def __init__(self, llm: LLMClient):        ...
这样 CRUX 将来切模型、切 provider、接 Agnes、DeepSeek、OpenAI-compatible API，都不会污染 ComfyUI 业务逻辑。

3. executor.py 2000+ 行：API 通信、错误处理、超时、重试
结论：executor.py 也偏大，应该拆成 ComfyUI 通信层 + 执行状态机 + 产物管理。
ComfyUI 执行器通常会包含很多东西：
- /prompt 提交- /history 查询- /view 下载图片- /upload/image 上传图片- /queue 查询- /interrupt 中断- websocket 进度监听- 结果文件下载- 批量执行- seed / 参数替换- 失败诊断
所以 2000 行不是绝对不合理，但要看是否有明确分层。
executor.py 应拆成
comfy_api_client.py  只负责 HTTP / WebSocket 通信execution_runner.py  负责任务生命周期：submit -> wait -> collectexecution_state.py  定义状态：queued / running / completed / failed / timeout / cancelledartifact_collector.py  下载图片、视频、metadataexecution_errors.py  错误分类retry_policy.py  重试策略
必须有的错误分类
现在最怕的是所有错误都变成：
Exception: ComfyUI execution failed
应该分成：
ComfyUIUnavailableComfyUIPromptRejectedComfyUIWorkflowValidationErrorComfyUINodeMissingComfyUIModelMissingComfyUIExecutionTimeoutComfyUIQueueTimeoutComfyUIConnectionLostComfyUIOutputMissingComfyUIArtifactDownloadErrorComfyUIUserCancelled
超时机制建议
至少需要四层 timeout：
connect_timeout      连接 ComfyUI API 超时submit_timeout       提交 prompt 超时queue_timeout        排队等待超时execution_timeout    单次生成总执行超时download_timeout     下载产物超时
不要只有一个全局 timeout。
重试机制建议
可以重试：
- 连接失败- 502 / 503 / 504- websocket 临时断开- history 查询短暂为空- 下载产物临时失败
不应该重试：
- 缺节点- 缺模型- workflow JSON 错误- prompt 被 ComfyUI 拒绝- 显存不足但没有释放策略- 参数类型错误
否则会浪费时间，还会让上层 CRUX/V2 误判。

4. config.py 143 行：硬编码与安全性
结论：143 行不大，但 config.py 是高风险文件。
重点不是行数，而是里面有没有硬编码这些东西：

../../绝对路径覆盖写入系统目录覆盖已有项目文件
3. 自动安装 custom nodes
这是高风险能力。
如果系统支持：
自动下载节点自动 git clone自动 pip install自动安装模型
那必须有安全边界：
默认 dry-run需要 allowlist需要用户确认需要记录安装来源禁止静默执行 pip / shell
4. 下载模型
模型下载要防：
错误 URL恶意文件名路径穿越覆盖已有模型下载到错误目录没有 hash 校验
config.py 应该分三层
default_config.py       安全默认值env_config.py           环境变量runtime_config.py       运行时覆盖
而不是所有东西堆在一个 config.py。

7. 与 V2 集成：如何作为 V2 子执行器？
结论：不要把这个项目当 V2 内部模块，要把它当 V2 的 ComfyUI 子执行器服务。
你提到 V2 已经有：
comfyuiPreflightBlockers
这说明 V2 关心的是：
当前 ComfyUI 能不能执行？缺什么？会不会失败？失败原因能不能前置暴露？
所以这个项目应该给 V2 提供标准 preflight contract。
推荐 V2 调用方式
V2  ↓ComfyUI Agent Service  ↓/health/preflight/workflows/validate/workflows/build/jobs/jobs/{id}/artifacts/{id}
preflight 返回格式建议
JSON{  "ok": false,  "blockers": [    {      "code": "COMFYUI_OFFLINE",      "level": "P0",      "message": "ComfyUI API is not reachable",      "fix": "Start ComfyUI or check host/port"    },    {      "code": "MODEL_MISSING",      "level": "P0",      "model": "sd_xl_base_1.0.safetensors",      "expected_path": "models/checkpoints/"    },    {      "code": "CUSTOM_NODE_MISSING",      "level": "P0",      "node": "IPAdapterUnifiedLoader",      "extension": "ComfyUI_IPAdapter_plus"    }  ],  "warnings": [],  "capabilities": {    "txt2img": true,    "img2img": true,    "video": false,    "controlnet": true  }}
V2 的 comfyuiPreflightBlockers 就不要自己猜。
它应该消费这个服务返回的 blockers。
V2 不应该做的事
- 不要自己解析 ComfyUI workflow 细节- 不要自己扫 models 文件夹- 不要自己判断 custom node 是否存在- 不要 browser-control 操作 ComfyUI 页面- 不要直接 import agent.py 内部对象
V2 只负责：
- 任务规划- 剧本 / 分镜 / 镜头 / 素材结构- 调用 ComfyUI 子执行器- 接收 artifacts- 汇总状态
ComfyUI Agent 负责：
- 工作流构建- 工作流校验- 模型/节点/路径 preflight- 执行- 产物下载- 错误标准化

8. CRUX 总控如何调用：API 还是 browser-control？
结论：CRUX 应该优先 API / MCP 调用，不应该用 browser-control 作为主路径。
你的 CRUX 本身更像工程终端 / 总控系统，已经有 agent mode、skill、MCP、工具加载、RunSummary、Incident、Remediation、Replay 这类结构化能力。
所以 ComfyUI Agent 应该作为：
CRUX skill: comfyui-bridge
或者：
CRUX MCP tool server
而不是让 CRUX 去点网页。
推荐调用优先级
第一优先级：HTTP API第二优先级：MCP tool第三优先级：CLI wrapper最后兜底：browser-control
为什么不用 browser-control 做主链路？
browser-control 适合：
- 临时操作网页- 无 API 的系统- UI 冒烟测试- 截图验证
但不适合：
- 稳定执行工作流- 长任务生成- 错误分类- 并发- 产物下载- 可重放- CRUX RunSummary / Incident 归档
CRUX 需要的是结构化结果：
JSON{  "run_id": "comfy_20260706_001",  "status": "failed",  "stage": "preflight",  "blockers": [    {      "code": "MODEL_MISSING",      "message": "Missing checkpoint"    }  ],  "artifacts": [],  "logs": [],  "remediation": [    "Install model into models/checkpoints"  ]}
browser-control 很难稳定产出这种结构。
