1. 这轮链路唯一化做得对吗？

做得对，而且方向非常关键。

你这轮 P0 真正解决的是：主链路不再有多个事实来源。现在执行、修复、编译、agent 调度都收敛到了唯一入口，这比继续堆功能重要得多。

现在架构已经从：

多个 runner / executor / manager / repair / flow 并行存在

收敛成：

agent
  ├─ diagnose → DiagnosePipeline
  ├─ repair   → ExecutionRepair + RecoveryManager
  ├─ validate → ExecutionRepair + FlowCompiler
  └─ execute  → UnifiedExecutor → ComfyExecutor + ExecutionManager

这是正确的。

还要重点查 8 个遗漏点
A. 是否还有隐藏旧入口

全局 grep 这些关键词：

submit_workflow
workflow_runner
ExecutionManager(
ComfyExecutor(
run_agent_flow
agent_flow
recovery
AutoHealer
RecoveryManager

目标不是删除所有引用，而是确认：

业务代码 → 只走新模块
旧模块 → 只作为 wrapper 存在
测试代码 → 明确覆盖 wrapper 兼容性
B. wrapper 是否“只转发，不持有状态”

旧文件包装器最好不要自己 new manager、new executor、new recovery 状态。

错误写法：

Python
运行
# old wrapper 内部自己创建独立 manager
manager = ExecutionManager()

正确方向：

Python
运行
# wrapper 只调统一入口
from .execution import UnifiedExecutor

否则会出现“链路唯一化表面完成，运行状态仍然双轨”。

C. retry 是否可能重复提交

execute_with_retry() 要确认一件事：

网络错误 / poll 超时 / ComfyUI 未返回结果

这些场景下，重试是否会重复提交同一个 workflow，导致 ComfyUI 队列里生成多张图。

建议每次执行返回结构里固定带：

Python
运行
{
    "run_id": "...",
    "prompt_id": "...",
    "attempt": 1,
    "status": "success|failed|timeout|cancelled",
    "outputs": [...],
    "error": None,
}
D. run_id 和 ComfyUI prompt_id 是否打通

这是发布前必须做的。

你需要能从任意日志反查：

agent run_id
→ UnifiedExecutor attempt_id
→ ComfyUI prompt_id
→ output image path
→ repair/retry history

没有这个，实机出错时会很难定位。

E. cancel / timeout / cleanup 是否完整

统一执行器要明确支持：

用户取消
ComfyUI 队列取消
本地状态标记 cancelled
临时文件清理
poll loop 退出

否则 P0 表面统一，实际会留下 zombie run。

F. validate() 是否真的只校验，不执行副作用

你现在：

agent.validate() → ExecutionRepair + FlowCompiler

要确认 validate 不会偷偷触发 ComfyUI 执行、不自动修模型、不自动重启服务。

建议规则：

validate = 静态检查 + 可选轻量 API 探测
repair = 改配置/修 workflow/恢复服务
execute = 真提交任务
G. RecoveryManager 和 ExecutionRepair 的边界

建议边界固定为：

ExecutionRepair: 识别错误、修改 workflow、给修复方案
RecoveryManager: 重试策略、退避、服务恢复、断连处理
AutoHealer: 具体自愈动作，例如 restart ComfyUI

不要让 ExecutionRepair 直接 sleep/retry，也不要让 RecoveryManager 改 workflow 结构。

H. 测试是否覆盖“旧 API 兼容 + 新链路真调用”

17/17 很好，但现在还缺一个关键验证：

旧 import 调用 → 实际进入 UnifiedExecutor / FlowCompiler / ExecutionRepair

可以用 monkeypatch 或 spy 确认旧入口没有走旧逻辑。

2. 下一步先做什么？

我的建议顺序非常明确：

第一步：做一次真实端到端实机验证
第二步：把这次实机路径固化成集成测试
第三步：再进入 P1 问题

不要现在立刻做 P1。

原因是：你刚完成 P0 链路收敛，现在最大的未知不是架构设计，而是：

真实 ComfyUI API 往返是否稳定
模型路径是否正确
结果文件是否能被拿到
poll 逻辑是否可靠
retry 是否会重复提交
manager 状态是否和真实执行一致

所以优先级应该是：

P0.5：实机 smoke test

目标：

启动 ComfyUI → 提交最小 workflow → 等待结果 → 拿到图片路径 → agent 返回统一结构

这个通过之后，P0 才算真正闭环。

P0.6：集成测试扩大

把刚才实测的链路做成测试：

test_e2e_txt2img_success
test_e2e_comfyui_down
test_e2e_missing_model
test_e2e_invalid_workflow
test_e2e_timeout_retry
test_wrapper_compatibility

其中真实 ComfyUI 测试可以打标：

Python
运行
@pytest.mark.integration
@pytest.mark.requires_comfyui
P1：再做能力层问题

P1 可以放在后面，例如：

chat.py 拆分
FlowCompiler 能力增强
LLM 意图分类稳定化
更多错误模式学习
server API 生产化

但这些都应该建立在一条真实可跑的主链路之上。

3. 端到端实测建议测什么场景？

我建议做三层场景，不要一上来测复杂视频或复杂自定义节点。

第一层：最小生图闭环

这是必须先过的。

场景名
e2e_minimal_txt2img
目标

验证：

agent.execute()
→ UnifiedExecutor.execute_with_retry()
→ ComfyExecutor.submit()
→ ComfyUI queue
→ poll
→ history/result
→ output image path
建议参数
尺寸：512x512
steps：8 或 10
seed：固定 seed
batch：1
模型：本机已确认存在的 checkpoint
节点：只用 ComfyUI 标准节点
不要 ControlNet
不要 LoRA
不要自定义节点
不要视频
Prompt
a simple cinematic glowing crystal tower floating in deep space, clean background, soft blue light, high detail

负面词：

blurry, low quality, distorted, text, watermark

这不是为了出好图，是为了验证链路。

第二层：你的项目风格 smoke test

第一层过了，再测你真实项目方向。

场景名
e2e_divine_spire_txt2img
Prompt
a colossal floating divine light spire in deep cosmic void, sacred geometric ring platform, translucent energy architecture grown from light, photonic rivers flowing through the structure, electro blue and soft gold illumination, cinematic IMAX composition, ultra clean atmosphere, no fog, no humans, highly detailed, 8k, ray traced lighting

负面词：

human, character, vehicle, mechanical city, cyberpunk, low quality, blurry, overexposed, heavy fog, text, watermark
为什么测这个

它符合你当前产品真实目标：

自然语言/模板 → 神性建筑场景 → ComfyUI 生图 → 返回结果

但它仍然是单图，不会引入视频链路复杂性。

第三层：故障注入测试

真实出图过了之后，马上测失败路径。

A. ComfyUI 未启动

预期：

diagnose: comfyui_check failed
execute: 不应崩溃
repair: AutoHealer 可尝试重连/重启
返回结构化错误
B. 模型文件缺失

故意传一个不存在的 checkpoint。

预期：

model_check failed
severity = error
suggestion 包含缺失模型名和搜索路径
不进入 submit
C. workflow 拓扑非法

比如断开 KSampler 到 VAE Decode 的连接。

预期：

FlowCompiler / validate 捕获
不提交 ComfyUI
返回 workflow topology error
D. poll 超时

把 timeout 设得很短，例如 1 秒。

预期：

execute_with_retry 生效
attempt 记录正确
不会无限卡死
不会重复污染状态
我建议你的下一步任务名
P0.5 Real ComfyUI E2E Gate

完成标准：

1. 最小 txt2img 成功
2. 返回 output 文件路径
3. run_id / prompt_id / attempt 全链路可追踪
4. ComfyUI down 能结构化失败
5. missing model 能提前拦截
6. invalid workflow 不提交
7. timeout 不死锁
8. 旧 API wrapper 仍可调用，但实际进入新链路

一句话结论：

P0 链路唯一化做对了；现在不要急着继续 P1，先用真实 ComfyUI 跑通一次“启动→生图→取结果→失败恢复”的 P0.5 实机发布门。
