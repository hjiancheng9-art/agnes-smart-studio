需要，而且必须有。

否则它会变成一堆功能堆叠：会推荐、会跑、会训练 LoRA、会调参，但每次怎么判断、怎么拆解、怎么修复、怎么验收都不稳定。

ComfyUI 智能体方法论应该解决什么

它不是写给用户看的“宣传文档”，而是写给智能体自己的行动准则 + 决策流程 + 验收标准。

核心要回答：

1. 用户需求如何拆成任务？
2. 什么时候用现成模板？
3. 什么时候合成新工作流？
4. 什么时候多阶段 pipeline？
5. 参数怎么默认、怎么解释、怎么自动调？
6. 执行失败怎么诊断、修复、重试？
7. LoRA 训练什么时候建议做，什么时候不建议做？
8. 什么结果算成功，什么结果必须重新跑？
它应该有 6 套方法论
1. 需求拆解方法论

用户说：

做一个赛博雨夜图生视频，镜头慢慢推进，4秒

智能体要拆成：

任务类型：image_to_video
输入资产：需要图片
输出资产：video
关键参数：duration=4, motion=slow_push_in
风格参数：cyberpunk, rainy night, neon
执行方式：优先推荐图生视频 workflow
缺失项：如果没图，先生成关键帧
2. 工作流选择方法论

优先级应该是：

1. 已验证黄金工作流
2. 高置信推荐模板
3. motif 组合生成
4. LLM 规划 + GraphCompiler 合成
5. 失败时回退到更简单工作流

不要一上来就让 LLM 发明复杂工作流。

3. 参数调优方法论

参数不是随便给默认值。

应该有规则：

文生图：steps/cfg/sampler/seed
图生图：重点看 denoise
视频：重点看 duration/motion/fps/seed
LoRA：重点看 strength_model/strength_clip
高清修复：重点看 upscale_scale/denoise

并且每次调参都要记录：

原参数
调整原因
调整方向
预期效果
实际结果
4. 错误修复方法论

标准闭环：

错误捕获
↓
错误分类
↓
定位节点 / 参数 / 文件 / 模型
↓
生成修复 patch
↓
重新 validate
↓
重试
↓
记录修复经验

典型规则：

缺模型 → 替换同类模型或提示下载
缺节点 → 提示安装节点包
latent size mismatch → 统一尺寸 / 重建 latent
输出缺失 → 自动补 SaveImage / VideoCombine
显存不足 → 降分辨率 / batch / steps
5. LoRA 炼制方法论

LoRA 训练不能和普通 workflow 执行混在一起。

它应该有独立流程：

确认目标：角色 / 风格 / 物体 / 服装
↓
检查数据集：数量、质量、多样性
↓
生成 caption
↓
用户审核
↓
生成训练配置
↓
启动训练
↓
自动测试权重
↓
选择最佳 checkpoint
↓
部署到推理工作流

并且要有拒绝/提醒机制：

图片太少 → 提醒不稳定
图片太重复 → 提醒过拟合
caption 太乱 → 不建议直接训练
目标不清楚 → 先问清楚
6. 验收方法论

不能只看“执行成功”。

要分层验收：

技术成功：workflow 跑完，有产物
结构成功：节点连接合理，无缺模型/缺节点
质量成功：图像/视频符合用户目标
可复现成功：记录 seed、模型、参数、workflow_id
可继续成功：结果能进入下一阶段
最终建议：写成 4 个文件
AGENTS.md
- 智能体通用铁律
- 不直接裸生成 ComfyUI JSON
- 先规划、再编译、再验证、再执行

COMFYUI_METHODOLOGY.md
- 工作流生成/推荐/执行/修复方法论

LORA_METHODOLOGY.md
- LoRA 数据集、caption、训练、测试、部署方法论

FRONTEND_METHODOLOGY.md
- 前端如何暴露能力、隐藏复杂度、展示错误和进度
一句话结论

需要。

而且它应该成为系统的“操作系统级规则”。

没有方法论，智能体只是功能集合。
有方法论，智能体才会稳定地判断、生成、执行、修复和进化。
