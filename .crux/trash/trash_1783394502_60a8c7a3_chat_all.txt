Q3：EventLog 存储——SQLite 可以用，但不要热路径同步写
你的倾向 SQLite 是对的，但实现方式要避免：
每次工具调用 → 同步 insert SQLite → commit
这样会把工具热路径绑死在 I/O 上，尤其 agent_swarm 并发时会出现锁等待。
我的建议是：
内存队列 / ring buffer
→ 异步批量写 SQLite
→ 可选 JSONL 旁路备份
推荐结构
Hot Path:
tool_call 结束
→ event_queue.put_nowait(event)
→ 立即返回
Background Writer:
每 50-200ms 或累计 32-128 条
→ SQLite batch insert
→ commit
也就是：SQLite 是主存储，但写入由异步 writer 承担。
最小落地架构
EventEmitter
├─ in-memory ring buffer，用于驾驶舱实时显示
├─ async SQLite writer，用于持久化和 metrics 聚合
└─ optional JSONL mirror，用于崩溃恢复/人工排查
ring buffer 保最近：
最近 1000-10000 条事件
SQLite 保完整历史或按天清理。
SQLite 设置建议
SQL
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA busy_timeout = 250;
PRAGMA temp_store = MEMORY;
WAL + NORMAL 对 CRUX 这类本地 Agent 平台比较合适：写入性能好，崩溃风险可接受。

能检测这个直白版本。
AST 里它大概长这样：
Python

args=[Constant(value="os")]

所以 AST visitor 可以抓：

最小检测规则：
Python
import ast
DANGEROUS_NAMES = {
"__import__", "eval", "exec", "compile", "open",
"getattr", "setattr", "delattr", "globals", "locals", "vars"
DANGEROUS_MODULES = {
"os", "sys", "subprocess", "socket", "requests",
"urllib", "importlib", "ctypes", "shutil", "pathlib"
DANGEROUS_ATTRS = {
"system", "popen", "remove", "unlink", "rmdir",
"rmtree", "rename", "replace", "spawn", "fork",
"execv", "execve", "Popen", "run", "call"
class SkillSecurityVisitor(ast.NodeVisitor):
def __init__(self):
self.findings = []
def visit_Import(self, node):
for alias in node.names:
root = alias.name.split(".")[0]
if root in DANGEROUS_MODULES:
self.findings.append(("dangerous_import", alias.name, node.lineno))
self.generic_visit(node)
def visit_ImportFrom(self, node):
root = (node.module or "").split(".")[0]
if root in DANGEROUS_MODULES:
self.findings.append(("dangerous_import_from", node.module, node.lineno))
self.generic_visit(node)
def visit_Call(self, node):
if isinstance(node.func, ast.Name):
if node.func.id in DANGEROUS_NAMES:
self.findings.append(("dangerous_call", node.func.id, node.lineno))
if isinstance(node.func, ast.Attribute):
if node.func.attr in DANGEROUS_ATTRS:
self.findings.append(("dangerous_attr_call", node.func.attr, node.lineno))
self.generic_visit(node)
这个能抓到：
Python
__import__("os").system("rm -rf /")

或者：

所以边界是：
AST 可以做 suspicious scoring。
AST 不能证明代码安全。

新话题：TRM 成长引擎冷启动
你的直觉 ε-greedy 简单、可解释，这没错。
但对 Agent 工具路由，我不建议把 ε-greedy 作为主策略。
原因：
ε-greedy 会随机探索。
随机探索在普通推荐系统里可以接受。
但在 Agent 工具路由里，错误工具可能产生副作用。
例如：
本该 review，却探索到 execute
本该 patch preview，却探索到 patch_file
本该本地搜索，却探索到联网发送
本该只读，却探索到 shell
所以 TRM 不是普通 bandit，而是：
带安全约束的上下文 bandit。
三种策略对比
1. ε-greedy
机制：
1 - ε：选当前最优工具
ε：随机选一个候选工具
优点：
实现最简单
可解释
容易调参
缺点：
探索太盲
不利用不确定性
对高风险工具不安全
低频 intent 学得慢
适合 CRUX 的位置：
只适合 Top-K 安全候选内部的小比例探索。
不适合作为全局路由主策略。
例如：
只在 risk <= low 且候选工具同属 intent 时，
以 3%-5% 概率在 top-3 内探索。
2. UCB
机制：
score = mean_reward + c * uncertainty_bonus
优点：
鼓励探索样本少的工具
比 ε-greedy 不那么随机
可解释性较好
缺点：
对非平稳环境不够灵活
容易过度探索新工具
reward 定义不好时会被污染
上下文适配能力弱
适合 CRUX 的位置：
工具健康度探索
provider fallback 探索
同一能力不同供应商排序
比如：
同样是 search：
crux_search vs kimi_search vs codebuddy_search
UCB 很适合比较这些“能力相似、风险相近”的工具。
3. Thompson Sampling
机制：
为每个工具维护成功率分布。
每次从分布采样一个可能成功率。
选采样值最高的工具。
优点：
天然平衡探索/利用
比 ε-greedy 更少盲目随机
适合冷启动先验
可以表达不确定性
可以按 intent / namespace 建模
缺点：
实现比 ε-greedy 稍复杂
解释性略差
reward 设计要谨慎
我认为它最适合 CRUX 的 TRM 成长引擎。
但不是裸 Thompson Sampling，而是：
Constrained Contextual Thompson Sampling with Priors
带约束、带上下文、带先验的 Thompson Sampling。
推荐方案：规则路由 + 先验分数 + Thompson Sampling 微调
CRUX 工具路由不要完全交给 bandit。应该分层：
1. Policy gate
过滤掉风险不允许的工具
2. Intent / capability retrieval
关键词 + embedding 召回 top-K
3. Static prior score
根据 tool manifest / schema / route card 打初始分
4. Thompson Sampling
在安全候选中做成长排序
5. Deterministic override
对危险/确定场景强制固定工具
也就是说，bandit 只做：
候选工具之间的排序微调
不要让 bandit 决定：
是否允许危险工具
是否跨 intent
是否绕过确认
冷启动先验怎么设
新工具不能随机排。给它一个先验。
先验来自：
intent match
capability match
schema fit
关键词分数
embedding 分数
namespace 可信度
skill trust tier
相似工具历史表现
静态 eval 分数
例如每个工具维护：
JSON
"tool": "patch_file",
"intent": "execute",
"success_alpha": 18,
"success_beta": 4,
"latency_mu": 700,
"latency_sigma": 120,
"risk_penalty": 0.15,
"cost_penalty": 0.05
对于新工具：
如果和高质量旧工具相似：
alpha/beta 继承部分先验
如果是 marketplace 新工具：
保守先验
如果本地可信工具：
中性偏乐观先验
如果高风险工具：
强保守先验
示例：
trusted built-in tool:
Beta(8, 2)
new local trusted skill:
Beta(5, 3)
new marketplace skill:
Beta(2, 4)
suspicious skill:
Beta(1, 6)
这比 ε-greedy 冷启动安全得多。
Reward 不要只看 success
Agent 工具路由里的 reward 应该是复合的：

- wrong_tool_penalty

Q1：假修复检测怎么设计？
你的判断是对的：
不是每种工具都需要假修复检测。只对“会改变系统状态或宣称完成目标”的工具做。
