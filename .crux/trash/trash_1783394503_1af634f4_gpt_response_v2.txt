这三个反方全部成立，而且它们指向同一个升级方向：

CRUX 不能把启发式规则当成确定性裁决。Seed policy、manifest inference、quarantine 都应该从“硬规则”升级为“带置信度、上下文和生命周期的策略系统”。

反方1：Seed policy 静态分类不够深

成立。

你举的 CUDA OOM 是典型反例：

A. batch_size=32 爆显存 → 降 batch_size 可修
B. 模型本体超过显存 → 代码层不可修

所以 seed policy 不应该直接输出：

CUDA OOM => downgrade_to_diagnosis

而应该输出：

CUDA OOM => fixability_prior + requires_context_probe

也就是说，seed policy 只是 先验层，不是最终裁决层。

建议把当前 fake_fix_seed_policy.py 升级成三层：

L0 Seed Prior
- 根据 error_type / tool / signature 给出初始 fixability_score

L1 Context Probe
- 读取上下文：batch_size、resolution、model_size、VRAM、workflow_hash、是否有可调参数

L2 Bounded Experiment
- 只在高价值、低风险场景尝试一次小修复
- 例如 batch_size 32 → 16、开启 vae_tiling、降低 resolution

建议新增：

core/fixability_estimator.py

输出结构可以是：

JSON
{
  "fixability_score": 0.72,
  "action": "limited_retry",
  "max_attempts": 1,
  "requires_context_probe": true,
  "requires_new_evidence": true,
  "suggested_repair_class": "reduce_resource_pressure",
  "allowed_mutations": ["batch_size", "resolution", "vae_tiling"],
  "forbidden_mutations": ["change_model", "delete_node", "disable_validation"],
  "reason": "OOM appears parameter-driven rather than model-size-driven."
}

CUDA OOM 的策略应该细分成：

CUDA OOM + 有 batch_size / resolution / tiling 可调
→ limited_retry

CUDA OOM + 模型估算显存 > 当前显存上限
→ requires_user_action

CUDA OOM + 已尝试降 batch / tiling 仍失败
→ downgrade_to_diagnosis

CUDA OOM + 同 context 多次失败
→ quarantine/backoff

结论：

seed policy 应改名为 seed prior 更准确。真正裁决应该由 fixability estimator 完成。

反方2：自动推断 manifest 是 false comfort

成立，而且这个比反方1更急，因为它涉及权限边界。

你说的误判完全会发生：

description = "Learn how to import photos"

这里的 import 是自然语言，不是 Python import。

description = "take screenshot tutorial"

这里的 screenshot 也不等于技能需要 browser:screenshot 权限。

所以自动推断不能直接生成真实权限。应该拆成两类字段：

JSON
{
  "runtime": "prompt",
  "permissions": [],

  "inferred_runtime": "declarative",
  "inferred_permissions": ["browser:screenshot"],

  "legacy": true,
  "confirmed": false,
  "confidence": 0.61
}

加载时只承认：

runtime
permissions
confirmed

不承认：

inferred_runtime
inferred_permissions

inferred_* 只能作为人工 review 建议。

另外你现在的：

Python → inline

我建议立刻改掉。

默认应该是：

.py → python_subprocess

python_inline 只能用于：

builtin
CRUX core-owned
confirmed=true
manual trusted

否则 manifest 权限系统会被主进程执行绕穿。

legacy manifest 建议策略：

confirmed=false:
  - 不进入 TRM 自动高优先级路由
  - 不授予 inferred_permissions
  - 不允许 network/write/browser_submit
  - 可手动调用
  - 低风险 prompt/declarative 技能可降级运行

最终建议：

JSON
{
  "name": "legacy-browser-control",
  "version": "0.0.0-legacy",
  "runtime": "prompt",
  "permissions": [],
  "inferred_runtime": "declarative",
  "inferred_permissions": ["browser:screenshot", "browser:navigate"],
  "legacy": true,
  "confirmed": false,
  "trust": "local_legacy"
}

结论：

自动推断 manifest 只能产生“待确认建议”，不能产生“安全事实”。

反方3：Quarantine 缺少衰退

成立。

当前：

24h 内 3 次，第 4 次隔离

只能解决短时间重复失败，不能解决你说的：

每天重新浪费 3 次

应该加：

strike_count + exponential backoff + decay + context_fingerprint

建议 quarantine 记录结构：

JSON
{
  "error_signature": "sha256:...",
  "context_fingerprint": "sha256:...",
  "status": "quarantined",
  "strike_count": 2,
  "first_seen": "2026-07-07T10:00:00+09:00",
  "last_seen": "2026-07-08T10:00:00+09:00",
  "quarantined_until": "2026-07-15T10:00:00+09:00",
  "manual_override": false
}

Backoff 梯度建议：

第1次 quarantine: 24h
第2次 quarantine: 7d
第3次 quarantine: 30d
第4次 quarantine: manual_review_required

我不建议叫“永久”，建议叫：

manual_review_required

因为版本、环境、依赖、显存、模型、workflow 变化后，原本不可修的问题可能变得可修。

还要把两个 hash 分开：

error_signature:
  表示是哪类错误
  例如 self_heal + CUDA_OOM + ComfyUI

context_fingerprint:
  表示这个错误发生在什么上下文
  例如 model=sdxl, batch_size=32, vram=8GB, workflow_hash=...

否则会出现两个问题：

同类错误在不同上下文被错误隔离
同一上下文每天重复浪费重试

再加 decay：

30天未命中:
  strike_count -= 1

90天未命中:
  archived 或 strike_count = 0

同类错误成功修复:
  strike_count -= 1
  fixability_prior 上调

结论：

Quarantine 应该是带上下文指纹和退避衰退的状态机，而不是 24h 计数器。

最小改动顺序

我建议下一步按这个顺序改：

1. skill_manifest.py
   - Python 默认 python_subprocess
   - permissions 与 inferred_permissions 分离
   - runtime 与 inferred_runtime 分离
   - legacy 默认 confirmed=false

2. fake_fix_detector.py
   - 增加 strike_count
   - 增加 quarantined_until
   - 增加 context_fingerprint
   - 增加 backoff: 24h / 7d / 30d / manual_review_required

3. fake_fix_seed_policy.py
   - 从硬规则改成 fixability prior
   - 输出 fixability_score / requires_context_probe / allowed_mutations

4. 新增 core/fixability_estimator.py
   - L0 seed prior
   - L1 context probe
   - L2 bounded experiment 后置实现

5. EventLog
   - 记录 fixability_score
   - 记录 quarantine_backoff_level
   - 记录 manifest_confirmed
   - 记录 inferred_permission_used=false

最终裁决：

三个反方全部成立。CRUX 下一步应从“静态规则治理”升级为“上下文感知治理”：seed 是先验，manifest inference 是建议，quarantine 是带衰退的状态机。
