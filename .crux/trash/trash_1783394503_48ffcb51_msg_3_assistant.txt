非常好，EventLog 和 ADR 已经把 CRUX 从“能跑”推进到“可审计”。最后两个问题，我建议这样定版。

Q1：假修复检测怎么设计？

你的判断是对的：

不是每种工具都需要假修复检测。只对“会改变系统状态或宣称完成目标”的工具做。

web_search、trm_route、hover、diagnostics 这类只读工具，不需要“假修复检测”。它们需要的是：

结果可信度检测
结果为空检测
结果过期检测
结果与任务不相关检测

而不是 pre/post state。

工具分层

建议把工具分成 4 类：

A. Read-only tools
   search / status / diagnostics / hover / graph_query

B. Generative tools
   generate_code / generate_prompt / generate_image / generate_video

C. State-changing tools
   patch_file / create_file / delete_file / install_package / browser_submit

D. Goal-declaring tools
   execute_plan / self_heal / goal_evaluate / release_create / rollout

假修复检测主要覆盖：

C. State-changing tools
D. Goal-declaring tools
对不同工具的 pre/post 定义
1. patch_file

pre/post state 应该包括：

JSON
{
  "files_touched": [
    {
      "path": "src/foo.py",
      "pre_hash": "sha256:...",
      "post_hash": "sha256:...",
      "diff_hash": "sha256:..."
    }
  ],
  "syntax_pre": "...",
  "syntax_post": "...",
  "tests_pre": "...",
  "tests_post": "...",
  "diagnostics_pre": "...",
  "diagnostics_post": "..."
}

最低检测：

是否改了允许范围外的文件
是否删除/跳过测试
是否只改 snapshot / golden file
是否引入新的 syntax error
是否新增 broad except / pass / TODO / disable lint
是否改动与目标无关
2. self_heal

self_heal 的假修复检测要更严格。

必须记录：

原始失败是否复现
补丁后失败是否消失
相关测试是否新增或更新
旧测试是否仍通过
错误签名是否真的变化
是否只是吞掉异常
是否只是改变日志/提示而非根因

推荐完成条件：

JSON
{
  "reproduced_original_failure": true,
  "applied_patch": true,
  "target_failure_resolved": true,
  "regression_passed": true,
  "new_evidence_added": true,
  "false_fix_score": 0.12
}

如果没有 reproduced_original_failure，self_heal 不能宣称 completed，只能是：

status = mitigated | attempted | needs_review
3. execute_plan

execute_plan 的 pre/post 不应该盯单个工具，而应该盯 goal-level evidence。

JSON
{
  "goal_id": "...",
  "pre": {
    "repo_hash": "...",
    "failing_tests": ["..."],
    "open_diagnostics": 12,
    "known_errors": ["..."]
  },
  "post": {
    "repo_hash": "...",
    "failing_tests": [],
    "open_diagnostics": 2,
    "known_errors": []
  },
  "acceptance": {
    "finish_line_met": true,
    "boundaries_respected": true,
    "budget_respected": true
  }
}

execute_plan 的重点是：

有没有完成 finish_line
有没有越界
有没有消耗超预算
有没有把错误转移到别处
有没有留下未处理 fallback
4. generate_code

代码生成本身不一定改状态，所以不要强行假修复检测。
但如果它输出代码并被 patch_file 应用，则检测在 patch_file 层做。

如果只是生成文本：

schema_valid
syntax_valid
compilation_possible
references_resolved
5. web_search / trm_route

这类不要做 pre/post checksum。

它们应该记录：

JSON
{
  "query": "...",
  "selected_tool": "web_search",
  "candidate_tools": ["web_search", "wiki_search", "file_search"],
  "route_confidence": 0.82,
  "result_count": 8,
  "empty_result": false,
  "fallback_used": false
}

需要的是 route quality，而不是 false-fix。

建议新增一个字段：tool_effect_class

每个工具 manifest / contract 加：

JSON
{
  "effect_class": "readonly | generative | state_changing | goal_declaring",
  "requires_false_fix_scan": false
}

规则：

readonly:
  不做假修复检测

generative:
  做 schema/syntax/quality check

state_changing:
  做 pre/post state diff

goal_declaring:
  做 evidence-level fa
