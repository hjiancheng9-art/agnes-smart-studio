你的判断是对的：CWIM 不应该继续作为 CRUX 的总方法论。

它适合做 ComfyUI 领域方法论插件，不适合作为通用 Agent 的最高层操作系统。

更准确地说：

CWIM 是 Domain Methodology，不是 Agent Methodology。
CRUX 需要一套更高层的 Agent Studio Methodology，然后把 CWIM 挂在其中作为 ComfyUI 专用子方法论。

1. CWIM 对 CRUX 是框架还是束缚？

答案是：现在一半是框架，一半会逐渐变成束缚。

CWIM 里有两类原则。

A 类：可泛化原则

这些应该保留，并上升为 CRUX 通用原则：

参数不是数字，而是语义
所有推荐必须可解释
不要直接生成最终产物，先理解目标结构
工作流要可验证、可回滚、可诊断
推荐必须说明适用条件和失败边界

这些不是 ComfyUI 专用，而是 Agent 可靠性的基础。

例如：

参数不是数字，而是语义

泛化后就是：

任何工具参数都不是裸值，而是意图、约束、风险和上下文的投影。

这对文件操作、浏览器控制、代码修复、工具路由都成立。

B 类：领域专用原则

这些应该留在 CWIM 内部，不应污染通用 Agent：

LLM 不直接生成 ComfyUI JSON
LoRA 是项目不是文件
Workflow 是图不是 JSON
节点兼容性优先于表面结构
模型 / VAE / ControlNet / Sampler 是语义绑定

这些对 ComfyUI 极其正确，但对通用任务会变形。

例如：

Workflow 是图不是 JSON

在 ComfyUI 中很强。
但在通用 Agent 中，文件操作、问答、浏览器控制并不总是图。硬套会造成过度工程。

2. 你不该问“继续 CWIM 还是放弃 CWIM”

正确问题是：

CWIM 应该降级为领域方法包，CRUX 应该升级出通用方法论内核。

推荐结构：

CRUX Agent Studio Methodology
│
├─ Core Agent Methodology
│  ├─ 意图澄清
│  ├─ 工具治理
│  ├─ 证据化执行
│  ├─ 风险控制
│  ├─ 可解释性
│  ├─ 回滚与自愈
│  └─ 预算管理
│
├─ Code Methodology
│  ├─ 先复现后修复
│  ├─ no evidence no patch
│  ├─ 假修复检测
│  └─ 测试闭环
│
├─ ComfyUI Methodology / CWIM
│  ├─ Workflow 是图不是 JSON
│  ├─ 参数语义化
│  ├─ 模型兼容性
│  └─ LoRA 生命周期
│
├─ Browser Methodology
│  ├─ 只读优先
│  ├─ 提交动作需确认
│  ├─ DOM 证据
│  └─ 截图/状态回放

   ├─ 镜头语言
   └─ 输出质量评估

CWIM 不消失，只是从“宪法”变成“专业部门规程”。

3. 方法论对 Agent 是必需品还是奢侈品？

看 Agent 的能力等级。

对普通聊天模型：方法论是加分项

普通问答、闲聊、写作辅助，方法论不是必需品。

因为它没有强副作用：

不会改文件
不会调用浏览器
不会安装依赖
不会提交表单
不会生成昂贵媒体
不会自动执行多步计划

这时方法论太重，反而会显得啰嗦。

对 CRUX 这种通用 Agent：方法论是必需品

你不是普通聊天模型。你有：

50+ 工具
文件写入
浏览器控制
ComfyUI 执行
self-heal
execute_plan
技能市场
多 Agent
Goal Mode
patch_file
pip_install

这类系统没有方法论会变成：

能做很多事，但不知道什么时候不该做
能修很多东西，但不知道是不是真修好
能调用很多工具，但不知道为什么选这个
能自动执行，但用户不敢信任

所以对 CRUX 来说，方法论不是奢侈品，而是 运行时安全与质量的操作系统。

但有一个前提：

方法论必须可路由、可降级、可局部启用，不能是一篇大 prompt 常驻压制所有任务。

坏的方法论会变成束缚。
好的方法论会变成治理层。

4. CRUX 应该发展 Agent Studio Methodology

我建议你把通用方法论命名为：

ASM — Agent Studio Methodology

或者更工程化一点：

CAGM — CRUX Agent Governance Methodology

其中 CWIM 作为：

CWIM — ComfyUI Workflow Intelligence Methodology

被 ASM 调用。

5. ASM 应该有哪些核心原则？

我建议第一版不要太多，控制在 10 条以内。

ASM v1 十原则
1. 先定义完成契约，再选择工具
2. 工具参数必须来自语义约束，而不是裸值猜测
3. 读操作优先，写操作必须可回滚
4. 高风险动作必须有确认、沙箱或降级路径
5. 所有执行必须留下 EventLog 证据
6. 所有推荐必须解释选择理由、适用边界和替代方案
7. 自愈必须先复现，再修复，再验证
8. 失败重试必须引入新证据，禁止盲重试
9. 方法论按任务路由，不得全局硬套
10. 完成不是“工具成功”，而是“目标证据通过”

这 10 条比 CWIM 更适合 CRUX 通用 Agent。

6. CWIM 原则应该如何迁移？

可以这样映射。

CWIM 原则	保留方式	泛化后
LLM 不直接生成 ComfyUI JSON	降级为 ComfyUI 专用	Agent 不直接生成高风险最终产物，先生成 IR / plan / patch
参数不是数字而是语义	上升为核心原则	所有工具参数都是语义约束投影
推荐必须可解释	上升为核心原则	每次路由、推荐、降级都要有 reason
LoRA 是项目不是文件	保留在 CWIM	长生命周期资源不是单文件，而是项目状态
Workflow 是图不是 JSON	保留在 CWIM	复杂任务应有结构化中间表示
先诊断再生成	上升为核心原则	Observe → Diagnose → Act
模型兼容性优先	泛化	工具/技能/上下文兼容性优先
错误恢复必须闭环	上升为核心原则	Retry 必须有 evidence
用户意图优先于模板	上升为核心原则	工具服从目标，不让目标服从工具
产物必须可验证	上升为核心原则	完成由 evaluator 裁决

这说明 CWIM 的价值很高，但它不该是总框架。

7. 最关键：方法论要从 prompt 变成 runtime policy

你现在最容易踩的坑是：

把方法论写成一大段系统提示

这会导致：

简单问答也过度分析
文件操作套 ComfyUI 逻辑
浏览器控制套 workflow graph
编程任务被视觉生成原则污染
模型每次都背诵原则而不是执行任务

正确做法是：

Methodology Router

根据任务选择方法论。

例如：

JSON
{
  "intent": "execute",
  "domain": "code",
  "risk": "medium",
  "methodology": ["ASM.core", "ASM.code", "ASM.file"]
}

ComfyUI 任务：

JSON
{
  "intent": "generate",
  "domain": "comfyui",
  "risk": "medium",
  "methodology": ["ASM.core", "CWIM"]
}

浏览器任务：

JSON
{
  "intent": "execute",
  "domain": "browser",
  "risk": "high",
  "methodology": ["ASM.core", "ASM.browser"]
}

普通问答：

JSON
{
  "intent": "think",
  "domain": "general",
  "risk": "low",
  "methodology": ["ASM.light"]
}
8. 方法论的最佳形态

不要做成：

一篇巨大的 METHOD.md

应该做成四层：

Principles
- 永久原则，少而稳

Policies
- 什么时候允许/禁止某类动作

Checklists
- 执行前/执行后检查

Evaluators
- 用证据裁决是否完成

例如：

ASM.core.principles.md
ASM.core.policies.json
ASM.code.checklist.yaml
ASM.comfyui.cwim.md
ASM.browser.policy.json
ASM.evaluators.py

这样方法论不只是“提醒模型”，而是能进入运行时。

9. 我的最终判断

CWIM 对你不是错的，但它已经不够高阶。

你现在应该：

保留 CWIM
但不要继续让 CWIM 统治 CRUX

正确路线是：

ASM 作为总方法论
CWIM 作为 ComfyUI 领域方法论
Code / Browser / File / Skill / Creative 各自拥有子方法论
Methodology Router 负责按任务加载

最终一句话：

方法论对 CRUX 不是奢侈品，而是自主系统的安全操作系统；但 CWIM 只能做 ComfyUI 专科医生，不能做 CRUX 的总院长。

规格 2：Fixability Estimator 三阶段接口协议
1. 目标

Fixability Estimator 用于在 self-heal / execute_plan / patch_file 重试前判断：

这个错误是否可能由 CRUX 自动修复？
应该重试、降级诊断、要求用户操作，还是 quarantine？

它不直接修复，只输出决策。

2. 三阶段结构
阶段	名称	时延目标	作用	是否默认执行
L0	Seed Prior	<5ms	基于 error_type/signature 的先验判断	是
L1	Context Probe	100ms-1s	读取上下文，判断是否有可修复变量	是，若 L0 要求
L2	Bounded Experiment	5s-30s	小范围低风险试验	仅高价值任务
3. 核心枚举

1. 痛点一：优先插话功能
你需要区分三种“插话”

用户说“打断插入新指令”，其实有三种语义：

类型	示例	Agent 应该怎么做
interrupt	“停一下” / “别继续了”	暂停当前 run
correction	“不是改 A，是改 B”	修改当前 goal / plan
priority message	“先回答这个问题”	当前 run 暂停，新消息优先处理

不要把三者都实现成 KeyboardInterrupt。那样太粗。

推荐事件模型

新增一个高优先级控制队列：

Python
运行
ControlEvent:
    id: str
    type: "interrupt" | "pause" | "cancel" | "correction" | "priority_message"
    priority: int
    target_run_id: str | None
    payload: dict
    created_at: datetime

TUI 普通消息走：

message_queue

控制事件走：

control_queue

Agent 执行循环每一步都先检查：

Python
运行
if control_queue.has_event():
    handle_control_event()
执行循环应该变成这样
while goal.active:
    check_control_events()

    next_step = planner.next_step()

    before_tool_call:
        check_control_events()

    result = tool.call(...)

    after_tool_call:
        check_control_events()

    evaluator.update(result)

关键点：

优先插话不一定要打断正在执行的底层工具，但必须在每个 step / tool boundary 处可响应。

长工具调用怎么办？

比如：

ComfyUI submit_workflow
browser automation
pip install
long test run
agent_swarm

这些不是每 100ms 都能停。

所以需要工具声明：

JSON
{
  "tool": "pip_install",
  "cancellable": true,
  "interrupt_mode": "cooperative | kill_process | not_interruptible"
}

三类：

interrupt_mode	含义
cooperative	工具定期检查 cancel token
kill_process	可杀子进程
not_interruptible	不能中止，只能标记“完成后暂停”
用户体验建议

运行中输入框仍可输入。按 Enter 后出现选择：

[1] 插话优先处理
[2] 修改当前任务
[3] 中止当前任务后处理
[4] 追加到任务后续

快捷键可以设计成：

Esc          暂停当前 agent

注意：Ctrl+C 不建议直接退出程序。应该先解释为：

连续两次 Ctrl+C 再退出 TUI。

2. 痛点二：撤销发送功能

撤销发送要分两个阶段处理。

阶段 A：消息还没被 agent 消费

这是最干净的撤销。

流程：

用户 Enter
→ 消息进入 pending 状态
→ TUI 显示 2 秒撤销窗口
→ 2 秒后 commit 到 message_queue

状态机：

draft
→ pending
→ committed
→ consumed

只要还在 pending，就可以无副作用撤销。

UI 可以显示：

你：帮我删除整个项目
状态：待发送 2.0s  [Ctrl+Z 撤销]

如果用户按 Ctrl+Z：

pending message removed
不会进入 agent
不会写入 goal
不会触发 tool
阶段 B：消息已经被 agent 消费

这时不能叫“撤销发送”，应该叫：

撤回并中止当前响应

因为 agent 已经可能：

读了消息
开始规划
调用工具
修改文件

这时撤销要做的是：

1. 发送 cancel event
2. 暂停当前 run
3. 标记用户消息为 retracted
4. 如果产生副作用，提示是否 rollback

EventLog 里记录：

JSON
{
  "event_type": "message.retracted",
  "message_id": "...",
  "run_id": "...",
  "retract_mode": "before_consumed | after_consumed",
  "rollback_available": true
}

4. 新任务

坑 1：用 Ctrl+C 直接杀进程

不建议。会破坏：

应该先发 cancel_run。

坑 2：插话直接写入当前 prompt

如果 agent 正在执行多步任务，直接把插话塞进上下文会污染当前 plan。

应该作为：

坑 3：撤销只改 UI，不改后端状态

撤销必须同步更新：

message_store
run_context
EventLog

否则 UI 上撤销了，agent 记忆里还在。

坑 4：不可中断工具假装可中断

每个工具必须声明：

否则用户按取消后以为停了，实际上后台还在跑。
