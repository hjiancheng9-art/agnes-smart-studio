下面只回答你指定的三块：Q3 EventLog 存储、Q4 AST 边界、TRM 冷启动 bandit 策略。

Q3：EventLog 存储——SQLite 可以用，但不要热路径同步写

你的倾向 SQLite 是对的，但实现方式要避免：

每次工具调用 → 同步 insert SQLite → commit

这样会把工具热路径绑死在 I/O 上，尤其 agent_swarm 并发时会出现锁等待。

我的建议是：

内存队列 / ring buffer
→ 异步批量写 SQLite
→ 可选 JSONL 旁路备份
推荐结构
Hot Path:
tool_call 结束
→ event_queue.put_nowait(event)
→ 立即返回

Background Writer:
每 50-200ms 或累计 32-128 条
→ SQLite batch insert
→ commit

也就是：SQLite 是主存储，但写入由异步 writer 承担。

最小落地架构
EventEmitter
  ├─ in-memory ring buffer，用于驾驶舱实时显示
  ├─ async SQLite writer，用于持久化和 metrics 聚合
  └─ optional JSONL mirror，用于崩溃恢复/人工排查

ring buffer 保最近：

最近 1000-10000 条事件

SQLite 保完整历史或按天清理。

SQLite 设置建议
SQL
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA busy_timeout = 250;
PRAGMA temp_store = MEMORY;

WAL + NORMAL 对 CRUX 这类本地 Agent 平台比较合适：写入性能好，崩溃风险可接受。

表结构建议

你给的 schema 可以作为最小版，但我建议多加几个字段，否则后面 trace、goal、self-heal 很快不够用：

SQL
CREATE TABLE IF NOT EXISTS event_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    event_id TEXT UNIQUE NOT NULL,
    timestamp TEXT NOT NULL,

    session_id TEXT,
    run_id TEXT,
    goal_id TEXT,
    step_id TEXT,

    trace_id TEXT,
    span_id TEXT,
    parent_span_id TEXT,

    event_type TEXT NOT NULL,
    intent TEXT,
    tool TEXT,

    status TEXT NOT NULL,
    duration_ms INTEGER,
    error_type TEXT,

    input_digest TEXT,
    output_digest TEXT,

    metadata_json TEXT
);

索引：

SQL
CREATE INDEX IF NOT EXISTS idx_event_ts ON event_log(timestamp);
CREATE INDEX IF NOT EXISTS idx_event_run ON event_log(run_id);
CREATE INDEX IF NOT EXISTS idx_event_goal ON event_log(goal_id);
CREATE INDEX IF NOT EXISTS idx_event_tool ON event_log(tool);
CREATE INDEX IF NOT EXISTS idx_event_type ON event_log(event_type);

你原 schema 里缺的关键字段是：

event_type
run_id
goal_id
step_id
trace_id/span_id
input_digest/output_digest

否则后面你只能知道“patch_file 成功了”，但不知道它属于哪个 goal、哪个 step、哪个 self-heal 阶段、前后状态是什么。

热路径性能策略

事件写入应分级：

critical event：不能丢
normal event：尽量写
debug event：队列满时可采样/丢弃

critical 包括：

patch_file
delete/write
self_heal
rollback
goal_completed/failed
skill_install
permission_denied

如果队列满了，不要阻塞所有工具调用。策略：

critical：阻塞最多 50-100ms，仍失败则写 emergency JSONL
normal：降级采样
debug：直接丢弃
结论

CRUX 最优方案：

SQLite 作为主持久层。
内存 ring buffer 作为实时层。
异步批量 writer 避免阻塞热路径。
JSONL 作为可选崩溃旁路。

不要直接同步写 SQLite。
也不要只用内存 ring buffer，否则 self-heal 和成长引擎无法做长期统计。

Q4：AST 能不能检测 __import__('os').system('rm -rf /')

能检测这个直白版本。

AST 里它大概长这样：

Python
运行
Call(
    func=Attribute(
        value=Call(
            func=Name(id="__import__"),
            args=[Constant(value="os")]
        ),
        attr="system"
    ),
    args=[Constant(value="rm -rf /")]
)

所以 AST visitor 可以抓：

Call(func=Name("__import__"))
Attribute(attr="system")
Constant("os")
Constant("rm -rf /")

最小检测规则：

运行

}

    "os", "sys", "subprocess", "socket", "requests",
    "urllib",
