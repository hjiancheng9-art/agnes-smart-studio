规格 2：Fixability Estimator 三阶段接口协议
1. 目标

Fixability Estimator 用于在 self-heal / execute_plan / patch_file 重试前判断：

这个错误是否可能由 CRUX 自动修复？
应该重试、降级诊断、要求用户操作，还是 quarantine？

它不直接修复，只输出决策。

2. 三阶段结构
阶段	名称	时延目标	作用	是否默认执行
L0	Seed Prior	<5ms	基于 error_type/signature 的先验判断	是
L1	Context Probe	100ms-1s	读取上下文，判断是否有可修复变量	是，若 L0 要求
L2	Bounded Experiment	5s-30s	小范围低风险试验	仅高价值任务
3. 核心枚举
FixabilityAction = Literal[
    "allow_retry",
    "limited_retry",
    "requires_new_evidence",
    "requires_context_probe",
    "downgrade_to_diagnosis",
    "requires_user_action",
    "quarantine",
    "manual_review_required"
]
RepairClass = Literal[
    "parameter_adjustment",
    "dependency_install",
    "environment_fix",
    "code_patch",
    "resource_reduction",
    "permission_fix",
    "network_retry",
    "configuration_fix",
    "not_fixable_by_code",
    "unknown"
]

L2 允许场景
场景	示例
可回滚参数修改	batch_size 32→16
dry-run	ComfyUI workflow validate
低成本测试	单个相关测试
临时环境探测	import check、version check
L2 禁止场景
禁止	原因
删除文件	高风险
修改生产配置	高风险
安装未知依赖	需用户确认
大规模重构	超出 bounded experiment
生成昂贵媒体	消耗预算
9. 决策流
input
→ L0 seed prior
→ if action=quarantine/manual_review_required: stop
→ if requires_context_probe: L1 context probe
→ recompute fixability_score
→ if score >= 0.70: limited_retry
→ if 0.40 <= score < 0.70: requires_new_evidence / diagnosis
→ if score < 0.40: downgrade_to_diagnosis
→ if repeated failure: quarantine backoff
→ optional L2 bounded experiment
10. 推荐阈值
fixability_score	动作
0.80 - 1.00	allow_retry
0.65 - 0.80	limited_retry
0.45 - 0.65	requires_new_evidence
0.25 - 0.45	downgrade_to_diagnosis
0.10 - 0.25	requires_user_action
0.00 - 0.10	quarantine / manual_review_required

模块	作用

规格 3：CWIM 降级为 ASM 子方法论的挂载点映射表
1. 总体定位

CWIM 不再作为 CRUX 全局方法论，而作为 ASM 的 ComfyUI 领域子方法论。

ASM.core
  └── ASM.media
        └── CWIM

CWIM 只在以下任务中挂载：

domain = comfyui
或 tool ∈ {
  submit_workflow,
  build_custom,
  preview_workflow,
  get_result,
  lora_train,
  lora_prepare,
  workflow_validate,
  comfyui_self_heal
}

普通问答、代码修复、文件操作、浏览器控制不默认加载 CWIM。

2. 挂载规则
条件	是否挂载 CWIM	同时挂载
用户要求生成/修改 ComfyUI workflow	是	ASM.core + ASM.media + ASM.file
用户要求 LoRA 训练/管理	是	ASM.core + ASM.media + ASM.skill
用户要求 ComfyUI 错误修复	是	ASM.core + ASM.code + ASM.file + CWIM
用户要求普通 Python 修复	否	ASM.core + ASM.code
用户要求浏览器控制	否	ASM.core + ASM.browser
用户要求通用问答	否	ASM.light
用户要求图像创意提示词但不涉及 ComfyUI	可选	ASM.media
用户要求 ComfyUI 节点兼容性判断	是	ASM.tool + CWIM
3. CWIM 原则映射表
CWIM 原则	原定位	ASM 挂载点	泛化方式	是否全局启用
LLM 不直接生成 ComfyUI JSON	ComfyUI workflow 安全	CWIM.workflow_generation	高风险最终产物先生成 IR/Plan	否
参数不是数字而是语义	参数设计	ASM.core.parameter_semantics	所有工具参数来自语义约束	是
所有推荐必须可解释	推荐与路由	ASM.core.explainability	工具选择/推荐/降级必须给 reason	是
LoRA 是项目不是文件	LoRA 生命周期	CWIM.lora_lifecycle	长生命周期资产按 project state 管理	否
Workflow 是图不是 JSON	Workflow 表征	CWIM.graph_model	复杂结构使用 IR/graph 而非裸 JSON	否
先诊断再生成	生成前置	ASM.core.observe_diagnose_act	任何高风险任务先观察/诊断	是
节点兼容性优先	ComfyUI 兼容性	CWIM.node_compatibility	工具/模型/技能兼容性优先	部分
错误恢复必须闭环	工作流修复	ASM.code.self_heal_loop	复现→修复→验证→fake-fix scan	是
用户意图优先于模板	Workflow 选择	ASM.core.goal_first	工具服从目标，不让目标服从模板	是
产物必须可验证	生成验收	ASM.evaluator	完成由证据裁决	是

6. CWIM 不得影响的任务
任务	原因
普通问答	会过度工程
Python 代码修复	应使用 ASM.code
文件批量修改	应使用 ASM.file
浏览器自动化	应使用 ASM.browser
技能 manifest 校验	应使用 ASM.skill
TRM 路由优化	应使用 ASM.tool
多 Agent 协作	应使用 ASM.agent
非 ComfyUI 文案生成	可用 ASM.media，但不需要 CWIM
7. CWIM 注入粒度

不要整篇 CWIM 常驻注入。按任务注入最小片段。

场景	注入内容
workflow 生成	workflow_ir + graph_validate + parameter_semantics
workflow 修复	graph_validate + node_compat + workflow_repair
LoRA 任务	lora_project + model_binding
ComfyUI 诊断	node_compat + model_binding + workflow_repair
ComfyUI 预览	preview_policy + export_policy
8. CWIM 与 ASM 的优先级
冲突	优先级
CWIM 要自动生成 workflow，但 ASM 风险策略要求确认	ASM 优先
CWIM 建议某节点，但 ASM.tool 判断插件缺失	ASM.tool 优先
CWIM 参数推荐与用户边界冲突	用户边界优先
CWIM workflow 修复会越界改文件	ASM.file 优先
CWIM 认为可自愈，但 FixabilityEstimator 低分	FixabilityEstimator 优先
CWIM 生成成功，但 ASM.evaluator 不通过	ASM.evaluator 优先
