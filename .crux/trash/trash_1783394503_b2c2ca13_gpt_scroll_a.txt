聊一个新痛点：CRUX的TUI在用户用鼠标选取消息文本打算复制的时候，屏幕不能上下滚动。这意味着如果消息长于一屏，用户只能选到当前显示的部分，选不完整。你有好的UI设计方案吗？
已思考 6s

这个痛点本质上不是“滚动坏了”，而是 TUI 的鼠标选择和应用内滚动争夺同一个输入通道。

终端里一旦 TUI 开启鼠标事件，滚轮、拖选、点击通常都会被应用接管；用户想用系统原生方式复制长文本时，就会卡在当前屏。解决方案不要硬修鼠标拖选，而应该设计一个 专用复制模式。

我建议 CRUX TUI 做成三层复制体验。

方案总纲：Copy Mode / Message Actions / Export
普通模式：
  看消息、输入、执行任务

复制模式：
  暂停自动滚动，进入可滚动选择状态

导出模式：
  直接把完整消息/对话导出到剪贴板或文件

核心原则：

不要让用户用鼠标从终端屏幕里“抠文本”，而是让 CRUX 知道用户要复制哪条消息、哪一段、哪一次响应。

方案 1：消息级复制按钮 / 快捷键

每条 assistant / user 消息都应该有隐藏操作栏。

例如鼠标悬停或键盘聚焦时显示：

┌─ Assistant · 14:32 · 1284 tokens ─────────────────────┐
│  这里是长消息内容……                                   │
│                                                        │
│  [c]复制全文  [s]选择片段  [o]打开详情  [e]导出        │
└────────────────────────────────────────────────────────┘

键盘操作：

↑/↓        聚焦上一条/下一条消息
c          复制当前消息全文
Shift+C    复制当前消息为 Markdown
s          进入片段选择模式
o          打开当前消息详情页
e          导出当前消息

鼠标操作：

点击消息左侧边栏       聚焦消息
右键消息              弹出操作菜单
双击消息标题          打开详情页

这能直接解决 80% 的复制需求：用户大多数时候不是要复制“屏幕上的选区”，而是要复制“一整条回复”。

方案 2：Copy Mode，支持跨屏选择

新增一个全屏复制模式。

进入方式：

Ctrl+Shift+C     进入复制模式
s                对当前消息进入选择模式
右键消息 → 选择片段

进入后 TUI 顶部显示：

COPY MODE · ↑↓滚动 · Space开始/结束选择 · Enter复制 · Esc退出

在 Copy Mode 里：

鼠标滚轮：滚动消息内容
鼠标拖拽：移动选择范围
Shift+↑/↓：扩展选择
PageUp/PageDown：跨屏滚动
Enter：复制选区
Esc：退出

关键是：这个选择不是终端原生选择，而是 CRUX 内部的文本选择。

也就是说，CRUX 维护：

Python
运行
SelectionState:
    message_id
    start_offset
    end_offset
    mode: "char" | "line" | "block"

用户滚动到下一屏后，选择范围依然存在。

UI 示例：

┌─ COPY MODE ────────────────────────────────────────────┐
│ Message 42 / Assistant / 218 lines                     │
│ ↑↓ scroll  Space mark  Enter copy  Esc exit             │
└────────────────────────────────────────────────────────┘

001  这个痛点本质上不是“滚动坏了”，而是……
002  终端里一旦 TUI 开启鼠标事件……
003  █████████████████ 被选中的文本 █████████████████
004  █████████████████ 可以跨屏继续 █████████████████
...

这比依赖终端选区稳定得多。

方案 3：消息详情页

对超长消息，不应该只在主聊天流里复制。

给每条消息一个详情视图：

按 o 打开当前消息详情

详情页只显示当前消息，提供独立滚动和复制。

┌─ Message Detail ─ Assistant · 14:32 ───────────────────┐
│ [c]复制全文 [m]复制Markdown [j]复制JSON [s]选择片段     │
├────────────────────────────────────────────────────────┤
│ 这里显示完整消息，独立滚动，不受主聊天流影响             │
│ ...                                                    │
└────────────────────────────────────────────────────────┘

优点：

主聊天流保持干净
长消息复制不影响当前对话
选择范围更容易管理
可以显示 token、时间、工具调用、EventLog 链接

这是我最推荐的长期方案。

方案 4：复制到剪贴板，不依赖鼠标选区

CRUX 应该提供内置复制命令。

/copy last
/copy last markdown
/copy 42
/copy 42:10-80
/copy code
/copy selection

示例：

/copy last

复制上一条 assistant 完整回复。

/copy 42 markdown

复制第 42 条消息的 Markdown 原文。

/copy 42:30-65

复制第 42 条消息第 30 到 65 行。

/copy code 2

复制当前消息里的第 2 个代码块。

这个功能对编程用户非常关键。很多时候用户只想复制代码块，不想复制整段解释。

方案 5：代码块一键复制

代码块应该有独立操作栏：

```python  [c]复制  [w]写入文件  [r]运行  [e]展开
def foo():
    pass

快捷键：

```text
Tab        在代码块之间跳转
c          复制当前代码块
Shift+C    复制所有代码块

这能解决 CRUX 编程场景的大量复制痛点。

方案 6：鼠标拖选时临时让出控制权

这是可选增强，不是主方案。

可以设计一个“原生选择模式”：

Alt+鼠标拖选

或者：

F9: 暂停 TUI 鼠标捕获

进入后：

CRUX 暂停接管鼠标事件
用户使用终端原生选择/滚动
再次按 F9 恢复

UI 显示：

NATIVE SELECT MODE · TUI mouse disabled · Press F9 to return

这个方案有兼容性问题，不同终端表现不同，但作为高级选项很好。

我建议的最终设计
MVP 先做三件事
1. 消息级复制
   c 复制当前消息全文
   /copy last
   /copy <message_id>

2. 消息详情页
   o 打开当前消息
   独立滚动
   c 复制全文
   s 选择片段

3. 代码块复制
   Tab 聚焦代码块
   c 复制当前代码块

这三项不依赖复杂鼠标事件，最快解决真实痛点。

第二阶段再做 Copy Mode
Ctrl+Shift+C 进入 Copy Mode
Space 设置选区起点/终点
Enter 复制
Esc 退出
支持跨屏滚动选择
第三阶段做 Native Select Mode
F9 暂停 TUI 鼠标捕获
允许用户用终端原生方式选择
F9 恢复
推荐交互布局

主界面底部状态栏增加：

↑↓选择消息 · o详情 · c复制 · s选片段 · /copy命令 · F9原生选择

当聚焦某条消息时：

● Assistant #42 · 218 lines · c复制全文 · o详情 · s选片段

当消息超过一屏时，顶部提示：

这条消息很长：按 o 打开详情页，或按 c 复制全
