Q1：假修复检测怎么设计？

你的判断是对的：

不是每种工具都需要假修复检测。只对“会改变系统状态或宣称完成目标”的工具做。

web_search、trm_route、hover、diagnostics 这类只读工具，不需要“假修复检测”。它们需要的是：

结果可信度检测
结果为空检测
结果过期检测
结果与任务不相关检测

而不是 pre/post state。

工具分层

建议把工具分成 4 类：

A. Read-only tools
   search / status / diagnostics / hover / graph_query

B. Generative tools
   generate_code / generate_prompt / generate_image / generate_video

C. State-changing tools
   patch_file / create_file / delete_file / install_package / browser_submit

D. Goal-declaring tools
   execute_plan / self_heal / goal_evaluate / release_create / rollout

假修复检测主要覆盖：

C. State-changing tools
D. Goal-declaring tools
对不同工具的 pre/post 定义
1. patch_file

pre/post state 应该包括：

JSON
{
  "files_touched": [
    {
      "path": "src/foo.py",
      "pre_hash": "sha256:...",
      "post_hash": "sha256:...",
      "diff_hash": "sha256:..."
    }
  ],
  "syntax_pre": "...",
  "syntax_post": "...",
  "tests_pre": "...",
  "tests_post": "...",
  "diagnostics_pre": "...",
  "diagnostics_post": "..."
}

最低检测：

是否改了允许范围外的文件
是否删除/跳过测试
是否只改 snapshot / golden file
是否引入新的 syntax error
是否新增 broad except / pass / TODO / disable lint
是否改动与目标无关
2. self_heal

self_heal 的假修复检测要更严格。

必须记录：

原始失败是否复现
补丁后失败是否消失
相关测试是否新增或更新
旧测试是否仍通过
错误签名是否真的变化
是否只是吞掉异常
是否只是改变日志/提示而非根因

推荐完成条件：

JSON
{
  "reproduced_original_failure": true,
  "applied_patch": true,
  "target_failure_resolved": true,
  "regression_passed": true,
  "new_evidence_added": true,
  "false_fix_score": 0.12
}

如果没有 reproduced_original_failure，self_heal 不能宣称 completed，只能是：

status = mitigated | attempted | needs_review
3. execute_plan

execute_plan 的 pre/post 不应该盯单个工具，而应该盯 goal-level evidence。

JSON
{
  "goal_id": "...",
  "pre": {
    "repo_hash": "...",
    "failing_tests": ["..."],
    "open_diagnostics": 12,
    "known_errors": ["..."]
  },
  "post": {
    "repo_hash": "...",
    "failing_tests": [],
    "open_diagnostics": 2,
    "known_errors": []
  },
  "acceptance": {
    "finish_line_met": true,
    "boundaries_respected": true,
    "budget_respected": true
  }
}

execute_plan 的重点是：

有没有完成 finish_line
有没有越界
有没有消耗超预算
有没有把错误转移到别处
有没有留下未处理 fallback
4. generate_code

代码生成本身不一定改状态，所以不要强行假修复检测。
但如果它输出代码并被 patch_file 应用，则检测在 patch_file 层做。

如果只是生成文本：

schema_valid
syntax_valid
compilation_possible
references_resolved
5. web_search / trm_route

这类不要做 pre/post checksum。

它们应该记录：

JSON
{
  "query": "...",
  "selected_tool": "web_search",
  "candidate_tools": ["web_search", "wiki_search", "file_search"],
  "route_confidence": 0.82,
  "result_count": 8,
  "empty_result": false,
  "fallback_used": false
}

需要的是 route quality，而不是 false-fix。

建议新增一个字段：tool_effect_class

每个工具 manifest / contract 加：

JSON
{
  "effect_class": "readonly | generative | state_changing | goal_declaring",
  "requires_false_fix_scan": false
}

规则：

readonly:
  不做假修复检测

generative:
  做 schema/syntax/quality check

state_changing:
  做 pre/post state diff

goal_declaring:
  做 evidence-level false-fix scan
最小落地版本

你现在可以先实现这 3 个：

1. patch_file pre/post file hash + diff hash
2. self_heal reproduction/evidence checklist
3. execute_plan goal evidence summary

暂时不要覆盖所有工具。

最终结论：

假修复检测不是工具通用能力，而是状态改变和目标完成类工具的验收层。

你给的这个版本基本够 MVP：

    "level": "local"

字段解释
id

必须有。name 可以展示给人看，但系统内部要用稳定 id。

JSON
"id": "crux.browser-control"

否则后面会出现：

browser-control
browser_control
BrowserControl
com.crux.browser-control

导致依赖、权限、统计全乱。

runtime

必须有。

JSON
"runtime": "prompt | declarative | python | subprocess | docker"

这决定怎么加载技能。

推荐 MVP 支持：

prompt
declarative
python_subprocess

不要让任意 Python 默认主进程执行。

capabilities

必须有。

JSON
"capabilities": ["browser_control", "page_inspect"]

permissions 是“允许访问什么”，capabilities 是“能完成什么”。
TRM 路由需要的是 capabilities。

min_crux_version

建议有。

JSON
"min_crux_version": "5.0.0"

否则 CRUX v5.2 改了 skill API，旧技能会静默坏掉。

trust

MVP 可以很简单：

JSON
"trust": {
  "level": "local"
}

枚举：

builtin
local
marketplace
verified
untrusted

这个字段会影响：

是否允许自动运行
是否需要 subprocess
是否需要用户确认
是否允许网络/文件写入
是否进入 TRM 主路由

JSON

  "id": "crux.browser-control",

  "category": "tool",
  "runtime": "declarative",

  "min_crux_version": "5.0.0",
  "trust": {
    "level": "local"
  }
}

这已经足够 MVP，不会吓退贡献者。

要不要加签名？

我的建议：

现在 manifest v1 不强制签名。
但字段要预留。

可以加 optional：

JSON
"signature": {
  "type": "none"
}

未来升级为：

JSON
"signature": {
  "type": "minisign",
  "key_id": "...",
  "value": "..."
}

或者：

JSON
"signature": {
  "type": "sigstore",
  "bundle": "..."
}

不建议用 JWT 作为技能包签名主方案。JWT 更适合声明身份和授权，不是最自然的软件包完整性签名格式。
minisign/cosign/sigstore 这类更贴近包分发场景。

验证在加载时做还是运行时做？

答案是：

加载时验证完整性。
运行时验证权限。

具体：

加载时

检查：

manifest schema 是否合法
id/version 是否合法
entrypoint 是否存在
requires 是否满足
min_crux_version 是否满足
signature 是否有效
hash 是否匹配
权限是否超过 trust level

加载失败则技能不进入 registry。

运行时

检查：

这次调用是否请求了声明外权限
是否访问越界路径
是否联网
是否调用 CDP 危险动作
是否超时
是否消耗预算
是否需要用户确认

也就是说：

signature 证明“这个包没被篡改”
manifest 说明“它声称要什么权限”
runtime guard 决定“它这一刻能不能做”

签名不能替代运行时权限控制。

MVP 不要做太重

你现在只有 51 个技能，不需要立刻引入完整供应链系统。

推荐路线：

v1:
manifest schema + permissions + capabilities + runtime + trust

v1.1:
entrypoint hash / lockfile

v1.2:
optional minisign signature

v1.3:
marketplace install requires signature

v2:
trusted publishing / attestation
最终定版建议
假修复检测
只覆盖高风险和目标完成类工具：
patch_file
execute_plan
self_heal
release / rollout / rollback
install / delete / write

readonly 工具不做 pre/post checksum。
它们做 result quality / route quality 记录。

JSON
{
  "effect_class": "state_changing",
  "requires_false_fix_scan": true
}

Q1：假修复检测的冷启动

你的两个方案里，A 和 B 不应该二选一。最佳方案是：

Seed policy + 本地学习 quarantine

也就是：

预置 seed 规则，避免已知死坑；
同时保留“24h 内同签名 3 次，第 4 次 quarantine”的在线学习。
不建议只用 B

只放行前三次的问题是：

新部署必踩坑
高成本 self-heal 会重复消耗
危险修复可能污染代码库
ComfyUI / pip / 环境类错误很容易无限试错

例如这些错误不应该靠真实失败学习：

pip install permission denied
CUDA out of memory
model file missing
ComfyUI custom node missing
网络超时 / 404
Python import path 错误
Windows 文件锁

它们很多不是“代码修复”能解决的。

但也不建议 seed quarantine 太硬

不要做成：

遇到某类错误 → 永久不重试

因为同一个错误类型在不同上下文下可能可修。

应该做成 seed policy，不是纯 quarantine list。

建议枚举：

allow_retry
limited_retry
requires_new_evidence
requires_user_action
downgrade_to_diagnosis
quarantine

例如：

  "tool": "self_heal",
  "policy": "downgrade_to_diagnosis",
  "reason": "资源不足通常不能通过代码补丁修复",
  "max_retries": 0
}
JSON
{
  "signature_pattern": "ModuleNotFoundError",
  "tool": "self_heal",
  "policy": "requires_user_action",

  "max_retries": 1
}
JSON
{
  "signature_pattern": "SyntaxError",
  "tool": "patch_file",

Q2：Manifest 迁移路径

你的迁移路径是合理的，但阶段 1 需要更保守一点。

你设想：

自动生成默认 manifest:
name = 文件名
runtime = prompt
permissions = []

这个对纯 prompt 技能没问题，但对现有 51 个技能里如果有脚本、浏览器、ComfyUI、文件操作类技能，就可能错误降级。

我建议阶段 1 改成：

自动推断 manifest，而不是统一 runtime=prompt。
阶段 1：兼容加载 + 自动生成 legacy manifest

生成：

JSON
{
  "schema_version": "1.0",
  "name": "browser-control",
  "version": "0.0.0-legacy",
  "description": "Auto-generated legacy manifest.",
  "runtime": "legacy_inferred",
  "permissions": [],
  "requires": [],
  "entrypoint": "skills/browser-control.skill.json",
  "integrity": "",
  "min_crux_version": "5.0.0",
  "author": "unknown",
  "legacy": true,
  "trust": "local_legacy"
}

然后根据文件类型推断：

.skill.json / .yaml      → declarative
.prompt / .md / .txt     → prompt
.py                      → python_subprocess
目录型技能               → inspect entrypoint
未知类型                 → disabled_pending_manifest

不要把 .py 技能推成 prompt。

阶段 2：自动补全建议

CRUX 可以生成一份建议 manifest，但不强制写入源文件：

.crux/generated_manifests/<skill>.manifest.json

并提示：

该技能疑似需要 permissions:
- filesystem:read
- cdp
- network:outbound

请确认后写入正式 manifest。

这样不会破坏现有技能，也不会默默授予权限。

阶段 3：新技能强制 manifest

这个没问题。

建议规则：

新技能：必须有 manifest
旧技能：可 legacy 加载，但降权
市场技能：必须 manifest + integrity
Python 技能：必须 runtime=python_subprocess 或 docker
python_inline：默认 deprecated

我特别建议你把 python_inline 标记为：

experimental / unsafe / only_builtin

不要让第三方技能使用 python_inline。否则 manifest 做了权限系统，最后又被主进程执行绕过。

阶段 4：6 个月宽限期合理

可以这样分阶段：

0-2 个月：
legacy 技能正常加载，但 EventLog 记录 warning

2-4 个月：
legacy 技能可用，但不进入 TRM 自动路由，只能手动调用

4-6 个月：
legacy 技能需要用户确认后运行

6 个月后：
无 manifest 技能默认禁用

这样不会突然破坏兼容，也能推动迁移。

两个最终定版建议
假修复冷启动

采用：

seed policy + 在线 quarantine

不要只靠第 4 次隔离。

最小 seed categories：

environment_error
dependency_missing
resource_exhausted
permission_denied
network_unavailable
model_missing
external_service_error
syntax_error
test_failure
unknown

其中前 6 类默认不进入 aggressive self-heal，只进入 diagnosis / user-action / limited-retry。

Manifest 迁移

你的三阶段路径成立，但阶段 1 改成：

自动推断 legacy manifest
而不是统一 runtime=prompt

并给 legacy 技能降权：

legacy = true
trust = local_legacy
不进入自动路由高优先级
高风险动作需要确认

最终一句话收束：

第一次隔离：24h

反方 1：Seed policy 静态分类不够深

我同意。

你举的 CUDA OOM 例子非常准确：

场景 A：batch_size=32 → 16 可修
场景 B：模型本体超过显存 → 不可修

所以 seed policy 不应该输出：

CUDA OOM => downgrade_to_diagnosis

而应该输出：

CUDA OOM => low_fixability_prior + require_context_probe

也就是说，seed policy 是 先验，不是最终判决。

不建议“一上来用几分钟估算”

你说的 fixability estimator 方向正确，但“几分钟本地估算”不适合作为默认前置门槛。原因：

1. 太慢，会拖垮普通 self-heal
2. 很多错误不值得深估算
3. estimator 本身也可能误判
4. 前置估算过重，会让 CRUX 像 CI 而不是 Agent runtime

更合理的是三层 estimator：

L0 seed prior        < 5ms
L1 context probe     < 100ms - 1s
L2 bounded experiment 5s - 30s

只有高价值任务才进入 L2。

legacy manifest 应该有 confirmed 字段

建议生成：

推断应该从“全文关键词”改成“结构化证据”

重要修正：Python 不应推断为 inline

你这次落地里有一句：

Python → inline

这里我必须强反对。

legacy Python 默认应该是：

python_subprocess

而不是：

python_inline

python_inline 只应允许：

builtin
manually trusted
CRUX core-owned
explicitly confirmed

否则 manifest 治理会被主进程执行绕穿。

建议：

.py 文件：
  builtin + confirmed → python_inline 可选
  local legacy → python_subprocess
  marketplace / unknown → python_subprocess 或 disabled_pending_review
legacy 技能的最终迁移策略
auto-generate legacy manifest
→ confirmed=false
→ trust=local_legacy
→ effective_permissions=[]
→ inferred_permissions 仅作为建议

JSON

  "inferred_permissions": ["browser:screenshot", "network:outbound"],
  "confirmed": false
}

加载时只承认 permissions，不承认 inferred_permissions。

反方 3：Quarantine 缺少衰退 / backoff

完全同意。

当前：

24h 内 3 次，第 4 次隔离

只能解决“短时间重复失败”，不能解决“每天重复浪费 3 次”的问题。

应该改成：

quarantine strike count + exponential backoff + decay + reset condition
推荐状态结构
JSON
{
  "signature": "sha256:...",
  "status": "quarantined",
  "strike_count": 2,

  "last_seen": "2026-07-08T10:00:00+09:00",
  "quarantined_until": "2026-07-15T10:00:00+09:00",
  "last_policy": "downgrade_to_diagnosis",
  "manual_override": false,
  "decay_after_days": 30,
  "context_fingerprint": "sha256:..."
}
Backoff 梯度

你给的梯度合理：

第 1 次 quarantine：24h
第 2 次 quarantine：7d
第 3 次 quarantine：30d
第 4 次及以后：permanent / manual review

我建议稍微改成可配置：

Python
运行
BACKOFF = [
    timedelta(hours=24),
    timedelta(days=7),
    timedelta(days=30),
    timedelta(days=180)
]

第 4 次不要真的叫永久，叫：

manual_review_required

原因：永久封禁容易误伤版本升级后的可修复问题。

解除 quarantine 的条件

不能只有手动解除。还应该支持 evidence reset。

允许解除或降级 quarantine 的条件：

1. 用户手动 clear_quarantine
2. CRUX 版本升级
3. 技能版本升级
4. 相关文件/配置发生重大变化
5. 环境资源变化，例如 VRAM / CUDA / Python 版本变化
6. 出现新的修复证据
7. 同类错误在其他上下文成功修复多次

例如 CUDA OOM：

昨天 8GB 显存 OOM 被隔离
今天换成 16GB 显存
→ context_fingerprint 改变
→ quarantine 可以降级为 limited_retry
context_fingerprint 很关键

你现在的：

error_signature = sha256(tool + error_type + context)

很好，但要把 context 分成两层：

error_signature:
  表示“这是哪类错误”

context_fingerprint:
  表示“这个错误发生在什么环境/参数下”

比如：

error_signature:
  self_heal + CUDA_OOM + ComfyUI

context_fingerprint:
  model=sdxl
  resolution=1024x1024
  batch=32
  vram=8GB
  workflow_hash=...

这样可以避免：

同类错误在不同上下文被错误共享 quarantine

也能避免：

上下文没变，每天重复浪费 3 次
衰退机制

有 backoff 还不够，还要有 decay。

否则历史错误永久污染未来。

建议：

如果 30 天没有再次命中：
  strike_count -= 1

如果 90 天没有再次命中：
  strike_count = 0 或 archived

如果同签名出现成功修复：
  strike_count -= 1
  fixability_prior 上调

伪代码：

Python
运行
def decay_quarantine(record, now):
    days = (now - record.last_seen).days

    if days >= 90:
        record.strike_count = 0
        record.status = "archived"
    elif days >= 30:
        record.strike_count = max(0, record.strike_count - 1)
