非常好，EventLog 和 ADR 已经把 CRUX 从“能跑”推进到“可审计”。最后两个问题，我建议这样定版。

Q1：假修复检测怎么设计？
你的判断是对的：

不是每种工具都需要假修复检测。只对“会改变系统状态或宣称完成目标”的工具做。

web_search、trm_route、hover、diagnostics 这类只读工具，不需要“假修复检测”。它们需要的是：
结果可信度检测结果为空检测结果过期检测结果与任务不相关检测
而不是 pre/post state。
工具分层
建议把工具分成 4 类：
A. Read-only tools   search / status / diagnostics / hover / graph_queryB. Generative tools   generate_code / generate_prompt / generate_image / generate_videoC. State-changing tools   patch_file / create_file / delete_file / install_package / browser_submitD. Goal-declaring tools   execute_plan / self_heal / goal_evaluate / release_create / rollout
假修复检测主要覆盖：
C. State-changing toolsD. Goal-declaring tools

对不同工具的 pre/post 定义
1. patch_file
pre/post state 应该包括：
JSON{  "files_touched": [    {      "path": "src/foo.py",      "pre_hash": "sha256:...",      "post_hash": "sha256:...",      "diff_hash": "sha256:..."    }  ],  "syntax_pre": "...",  "syntax_post": "...",  "tests_pre": "...",  "tests_post": "...",  "diagnostics_pre": "...",  "diagnostics_post": "..."}
最低检测：
是否改了允许范围外的文件是否删除/跳过测试是否只改 snapshot / golden file是否引入新的 syntax error是否新增 broad except / pass / TODO / disable lint是否改动与目标无关

2. self_heal
self_heal 的假修复检测要更严格。
必须记录：
原始失败是否复现补丁后失败是否消失相关测试是否新增或更新旧测试是否仍通过错误签名是否真的变化是否只是吞掉异常是否只是改变日志/提示而非根因
推荐完成条件：
JSON{  "reproduced_original_failure": true,  "applied_patch": true,  "target_failure_resolved": true,  "regression_passed": true,  "new_evidence_added": true,  "false_fix_score": 0.12}
如果没有 reproduced_original_failure，self_heal 不能宣称 completed，只能是：
status = mitigated | attempted | needs_review

3. execute_plan
execute_plan 的 pre/post 不应该盯单个工具，而应该盯 goal-level evidence。
JSON{  "goal_id": "...",  "pre": {    "repo_hash": "...",    "failing_tests": ["..."],    "open_diagnostics": 12,    "known_errors": ["..."]  },  "post": {    "repo_hash": "...",    "failing_tests": [],    "open_diagnostics": 2,    "known_errors": []  },  "acceptance": {    "finish_line_met": true,    "boundaries_respected": true,    "budget_respected": true  }}
execute_plan 的重点是：
有没有完成 finish_line有没有越界有没有消耗超预算有没有把错误转移到别处有没有留下未处理 fallback

4. generate_code
代码生成本身不一定改状态，所以不要强行假修复检测。
但如果它输出代码并被 patch_file 应用，则检测在 patch_file 层做。
如果只是生成文本：
schema_validsyntax_validcompilation_possiblereferences_resolved

5. web_search / trm_route
这类不要做 pre/post checksum。
它们应该记录：
JSON{  "query": "...",  "selected_tool": "web_search",  "candidate_tools": ["web_search", "wiki_search", "file_search"],  "route_confidence": 0.82,  "result_count": 8,  "empty_result": false,  "fallback_used": false}
需要的是 route quality，而不是 false-fix。

建议新增一个字段：tool_effect_class
每个工具 manifest / contract 加：
JSON{  "effect_class": "readonly | generative | state_changing | goal_declaring",  "requires_false_fix_scan": false}
规则：
readonly:  不做假修复检测generative:  做 schema/syntax/quality checkstate_changing:  做 pre/post state diffgoal_declaring:  做 evidence-level false-fix scan

最小落地版本
你现在可以先实现这 3 个：
1. patch_file pre/post file hash + diff hash2. self_heal reproduction/evidence checklist3. execute_plan goal evidence summary
暂时不要覆盖所有工具。
最终结论：

假修复检测不是工具通用能力，而是状态改变和目标完成类工具的验收层。


Q2：技能包 manifest 最小可行版本
你给的这个版本基本够 MVP：
JSON{  "name": "browser-control",  "version": "1.0.0",  "description": "...",  "permissions": ["cdp", "filesystem:read"],  "requires": [],  "entrypoint": "skills/browser-control.skill.json"}
但我建议最小可行版再加 5 个字段，否则后面会返工：
JSON{  "id": "crux.browser-control",  "name": "browser-control",  "version": "1.0.0",  "description": "...",  "category": "tool",  "entrypoint": "skills/browser-control.skill.json",  "runtime": "declarative",  "permissions": ["cdp", "filesystem:read"],  "requires": [],  "capabilities": ["browser_control", "page_inspect"],  "min_crux_version": "5.0.0",  "trust": {    "level": "local"  }}
字段解释
id
必须有。name 可以展示给人看，但系统内部要用稳定 id。
JSON"id": "crux.browser-control"
否则后面会出现：
browser-controlbrowser_controlBrowserControlcom.crux.browser-control
导致依赖、权限、统计全乱。

runtime
必须有。
JSON"runtime": "prompt | declarative | python | subprocess | docker"
这决定怎么加载技能。
推荐 MVP 支持：
promptdeclarativepython_subprocess
不要让任意 Python 默认主进程执行。

capabilities
必须有。
JSON"capabilities": ["browser_control", "page_inspect"]
permissions 是“允许访问什么”，capabilities 是“能完成什么”。
TRM 路由需要的是 capabilities。

min_crux_version
建议有。
JSON"min_crux_version": "5.0.0"
否则 CRUX v5.2 改了 skill API，旧技能会静默坏掉。

trust
MVP 可以很简单：
JSON"trust": {  "level": "local"}
枚举：
builtinlocalmarketplaceverifieduntrusted
这个字段会影响：
是否允许自动运行是否需要 subprocess是否需要用户确认是否允许网络/文件写入是否进入 TRM 主路由

最小 manifest v1 建议
JSON{  "schema_version": "1.0",  "id": "crux.browser-control",  "name": "browser-control",  "version": "1.0.0",  "description": "Control and inspect browser pages through CDP.",  "category": "tool",  "runtime": "declarative",  "entrypoint": "skills/browser-control.skill.json",  "capabilities": ["browser_control", "page_inspect"],  "permissions": ["cdp", "filesystem:read"],  "requires": [],  "min_crux_version": "5.0.0",  "trust": {    "level": "local"  }}
这已经足够 MVP，不会吓退贡献者。

要不要加签名？
我的建议：
现在 manifest v1 不强制签名。但字段要预留。
可以加 optional：
JSON"signature": {  "type": "none"}
未来升级为：
JSON"signature": {  "type": "minisign",  "key_id": "...",  "value": "..."}
或者：
JSON"signature": {  "type": "sigstore",  "bundle": "..."}
不建议用 JWT 作为技能包签名主方案。JWT 更适合声明身份和授权，不是最自然的软件包完整性签名格式。
minisign/cosign/sigstore 这类更贴近包分发场景。

验证在加载时做还是运行时做？
答案是：
加载时验证完整性。运行时验证权限。
具体：
加载时
检查：
manifest schema 是否合法id/version 是否合法entrypoint 是否存在requires 是否满足min_crux_version 是否满足signature 是否有效hash 是否匹配权限是否超过 trust level
加载失败则技能不进入 registry。

运行时
检查：
这次调用是否请求了声明外权限是否访问越界路径是否联网是否调用 CDP 危险动作是否超时是否消耗预算是否需要用户确认
也就是说：
signature 证明“这个包没被篡改”manifest 说明“它声称要什么权限”runtime guard 决定“它这一刻能不能做”
签名不能替代运行时权限控制。

MVP 不要做太重
你现在只有 51 个技能，不需要立刻引入完整供应链系统。
推荐路线：
v1:manifest schema + permissions + capabilities + runtime + trustv1.1:entrypoint hash / lockfilev1.2:optional minisign signaturev1.3:marketplace install requires signaturev2:trusted publishing / attestation

最终定版建议
假修复检测
只覆盖高风险和目标完成类工具：patch_fileexecute_planself_healrelease / rollout / rollbackinstall / delete / writereadonly 工具不做 pre/post checksum。它们做 result quality / route quality 记录。
核心字段：
JSON{  "effect_class": "state_changing",  "requires_false_fix_scan": true}
Skill manifest
你给的版本够 MVP，但建议升级为：
JSON{  "schema_version": "1.0",  "id": "crux.browser-control",  "name": "browser-control",  "version": "1.0.0",  "description": "...",  "category": "tool",  "runtime": "declarative",  "entrypoint": "skills/browser-control.skill.json",  "capabilities": ["browser_control"],  "permissions": ["cdp", "filesystem:read"],  "requires": [],  "min_crux_version": "5.0.0",  "trust": {    "level": "local"  },  "signature": {    "type": "none"  }}
一句话收束：

CRUX 现在不需要“大而全”的治理系统，但每个 schema 都要为未来治理预留位置。
