下面只回答你指定的三块：Q3 EventLog 存储、Q4 AST 边界、TRM 冷启动 bandit 策略。

Q3：EventLog 存储——SQLite 可以用，但不要热路径同步写
你的倾向 SQLite 是对的，但实现方式要避免：
每次工具调用 → 同步 insert SQLite → commit
这样会把工具热路径绑死在 I/O 上，尤其 agent_swarm 并发时会出现锁等待。
我的建议是：
内存队列 / ring buffer→ 异步批量写 SQLite→ 可选 JSONL 旁路备份
推荐结构
Hot Path:tool_call 结束→ event_queue.put_nowait(event)→ 立即返回Background Writer:每 50-200ms 或累计 32-128 条→ SQLite batch insert→ commit
也就是：SQLite 是主存储，但写入由异步 writer 承担。

最小落地架构
EventEmitter  ├─ in-memory ring buffer，用于驾驶舱实时显示  ├─ async SQLite writer，用于持久化和 metrics 聚合  └─ optional JSONL mirror，用于崩溃恢复/人工排查
ring buffer 保最近：
最近 1000-10000 条事件
SQLite 保完整历史或按天清理。

SQLite 设置建议
SQLPRAGMA journal_mode = WAL;PRAGMA synchronous = NORMAL;PRAGMA busy_timeout = 250;PRAGMA temp_store = MEMORY;
WAL + NORMAL 对 CRUX 这类本地 Agent 平台比较合适：写入性能好，崩溃风险可接受。

表结构建议
你给的 schema 可以作为最小版，但我建议多加几个字段，否则后面 trace、goal、self-heal 很快不够用：
SQLCREATE TABLE IF NOT EXISTS event_log (    id INTEGER PRIMARY KEY AUTOINCREMENT,    event_id TEXT UNIQUE NOT NULL,    timestamp TEXT NOT NULL,    session_id TEXT,    run_id TEXT,    goal_id TEXT,    step_id TEXT,    trace_id TEXT,    span_id TEXT,    parent_span_id TEXT,    event_type TEXT NOT NULL,    intent TEXT,    tool TEXT,    status TEXT NOT NULL,    duration_ms INTEGER,    error_type TEXT,    input_digest TEXT,    output_digest TEXT,    metadata_json TEXT);
索引：
SQLCREATE INDEX IF NOT EXISTS idx_event_ts ON event_log(timestamp);CREATE INDEX IF NOT EXISTS idx_event_run ON event_log(run_id);CREATE INDEX IF NOT EXISTS idx_event_goal ON event_log(goal_id);CREATE INDEX IF NOT EXISTS idx_event_tool ON event_log(tool);CREATE INDEX IF NOT EXISTS idx_event_type ON event_log(event_type);
你原 schema 里缺的关键字段是：
event_typerun_idgoal_idstep_idtrace_id/span_idinput_digest/output_digest
否则后面你只能知道“patch_file 成功了”，但不知道它属于哪个 goal、哪个 step、哪个 self-heal 阶段、前后状态是什么。

热路径性能策略
事件写入应分级：
critical event：不能丢normal event：尽量写debug event：队列满时可采样/丢弃
critical 包括：
patch_filedelete/writeself_healrollbackgoal_completed/failedskill_installpermission_denied
如果队列满了，不要阻塞所有工具调用。策略：
critical：阻塞最多 50-100ms，仍失败则写 emergency JSONLnormal：降级采样debug：直接丢弃

结论
CRUX 最优方案：
SQLite 作为主持久层。内存 ring buffer 作为实时层。异步批量 writer 避免阻塞热路径。JSONL 作为可选崩溃旁路。
不要直接同步写 SQLite。
也不要只用内存 ring buffer，否则 self-heal 和成长引擎无法做长期统计。

Q4：AST 能不能检测 __import__('os').system('rm -rf /')
能检测这个直白版本。
AST 里它大概长这样：
Python运行Call(    func=Attribute(        value=Call(            func=Name(id="__import__"),            args=[Constant(value="os")]        ),        attr="system"    ),    args=[Constant(value="rm -rf /")])
所以 AST visitor 可以抓：
Call(func=Name("__import__"))Attribute(attr="system")Constant("os")Constant("rm -rf /")
最小检测规则：
Python运行import astDANGEROUS_NAMES = {    "__import__", "eval", "exec", "compile", "open",    "getattr", "setattr", "delattr", "globals", "locals", "vars"}DANGEROUS_MODULES = {    "os", "sys", "subprocess", "socket", "requests",    "urllib", "importlib", "ctypes", "shutil", "pathlib"}DANGEROUS_ATTRS = {    "system", "popen", "remove", "unlink", "rmdir",    "rmtree", "rename", "replace", "spawn", "fork",    "execv", "execve", "Popen", "run", "call"}class SkillSecurityVisitor(ast.NodeVisitor):    def __init__(self):        self.findings = []    def visit_Import(self, node):        for alias in node.names:            root = alias.name.split(".")[0]            if root in DANGEROUS_MODULES:                self.findings.append(("dangerous_import", alias.name, node.lineno))        self.generic_visit(node)    def visit_ImportFrom(self, node):        root = (node.module or "").split(".")[0]        if root in DANGEROUS_MODULES:            self.findings.append(("dangerous_import_from", node.module, node.lineno))        self.generic_visit(node)    def visit_Call(self, node):        if isinstance(node.func, ast.Name):            if node.func.id in DANGEROUS_NAMES:                self.findings.append(("dangerous_call", node.func.id, node.lineno))        if isinstance(node.func, ast.Attribute):            if node.func.attr in DANGEROUS_ATTRS:                self.findings.append(("dangerous_attr_call", node.func.attr, node.lineno))        self.generic_visit(node)
这个能抓到：
Python运行__import__("os").system("rm -rf /")
但问题是：AST 只能抓明显模式，不能作为安全边界。

AST 抓不稳的绕过方式
例如：
Python运行m = "__im" + "port__"getattr(__builtins__, m)("os").system("rm -rf /")
或者：
Python运行name = "".join(["o", "s"])fn = getattr(__builtins__, "__import__")(name)getattr(fn, "system")("rm -rf /")
或者：
Python运行import importlibimportlib.import_module("os").system("rm -rf /")
或者更隐蔽：
Python运行().__class__.__base__.__subclasses__()
通过对象图找文件、进程、loader、subprocess 相关类。
所以边界是：
AST 可以做 suspicious scoring。AST 不能证明代码安全。

对你三层设计的修正
你现在的设计是：
L1 AST + 能力声明，通过则运行L2 suspicious subprocessL3 Docker
我建议改成：
L1 prompt-only / declarative skill：不执行任意 Python，最快路径L2 trusted Python skill：即使 AST 通过，也在 subprocess 运行L3 suspicious / marketplace Python skill：subprocess + restricted cwd + no secrets + no network + timeout + capability proxyL4 high-risk skill：Docker / Windows Sandbox / WSL sandbox
关键变化：

不要让任意 Python 技能在 CRUX 主进程内运行。哪怕 AST 通过。

因为一旦主进程内执行，技能可以：
读 CRUX 内存改全局状态猴子补丁 monkey patch拿环境变量污染 sys.modules挂死事件循环破坏日志绕过权限代理
所以最低要求：
Python skill = subprocessDeclarative skill = in-processPrompt-only skill = in-process

运行时能力模型
不要给技能真实文件系统 API。给 capability proxy：
Python运行def run(ctx, input):    content = ctx.fs.read_project_file("README.md")    return {"summary": content[:500]}
ctx.fs.read_project_file 内部做：
路径规范化目录越界检测权限检查大小限制EventLog 记录敏感文件拦截
即使技能在 subprocess 中，也不要让它自由 open()。

结论
对你的具体问题：
AST 能检测 __import__('os').system(...) 这种直白写法。但不能可靠检测所有变体。所以 AST 只能用于 suspicious 标记，不能作为安全边界。
CRUX 的安全底线应该是：
任意 Python 技能不进主进程。所有 Python 技能至少 subprocess。高风险技能才 Docker。
这样不会牺牲太多体验，也不会把主进程暴露给技能包。

新话题：TRM 成长引擎冷启动
你的直觉 ε-greedy 简单、可解释，这没错。
但对 Agent 工具路由，我不建议把 ε-greedy 作为主策略。
原因：
ε-greedy 会随机探索。随机探索在普通推荐系统里可以接受。但在 Agent 工具路由里，错误工具可能产生副作用。
例如：
本该 review，却探索到 execute本该 patch preview，却探索到 patch_file本该本地搜索，却探索到联网发送本该只读，却探索到 shell
所以 TRM 不是普通 bandit，而是：

带安全约束的上下文 bandit。


三种策略对比
1. ε-greedy
机制：
1 - ε：选当前最优工具ε：随机选一个候选工具
优点：
实现最简单可解释容易调参
缺点：
探索太盲不利用不确定性对高风险工具不安全低频 intent 学得慢
适合 CRUX 的位置：
只适合 Top-K 安全候选内部的小比例探索。不适合作为全局路由主策略。
例如：
只在 risk <= low 且候选工具同属 intent 时，以 3%-5% 概率在 top-3 内探索。

2. UCB
机制：
score = mean_reward + c * uncertainty_bonus
优点：
鼓励探索样本少的工具比 ε-greedy 不那么随机可解释性较好
缺点：
对非平稳环境不够灵活容易过度探索新工具reward 定义不好时会被污染上下文适配能力弱
适合 CRUX 的位置：
工具健康度探索provider fallback 探索同一能力不同供应商排序
比如：
同样是 search：crux_search vs kimi_search vs codebuddy_search
UCB 很适合比较这些“能力相似、风险相近”的工具。

3. Thompson Sampling
机制：
为每个工具维护成功率分布。每次从分布采样一个可能成功率。选采样值最高的工具。
优点：
天然平衡探索/利用比 ε-greedy 更少盲目随机适合冷启动先验可以表达不确定性可以按 intent / namespace 建模
缺点：
实现比 ε-greedy 稍复杂解释性略差reward 设计要谨慎
我认为它最适合 CRUX 的 TRM 成长引擎。
但不是裸 Thompson Sampling，而是：

Constrained Contextual Thompson Sampling with Priors
带约束、带上下文、带先验的 Thompson Sampling。


推荐方案：规则路由 + 先验分数 + Thompson Sampling 微调
CRUX 工具路由不要完全交给 bandit。应该分层：
1. Policy gate   过滤掉风险不允许的工具2. Intent / capability retrieval   关键词 + embedding 召回 top-K3. Static prior score   根据 tool manifest / schema / route card 打初始分4. Thompson Sampling   在安全候选中做成长排序5. Deterministic override   对危险/确定场景强制固定工具
也就是说，bandit 只做：
候选工具之间的排序微调
不要让 bandit 决定：
是否允许危险工具是否跨 intent是否绕过确认

冷启动先验怎么设
新工具不能随机排。给它一个先验。
先验来自：
intent matchcapability matchschema fit关键词分数embedding 分数namespace 可信度skill trust tier相似工具历史表现静态 eval 分数
例如每个工具维护：
JSON{  "tool": "patch_file",  "intent": "execute",  "success_alpha": 18,  "success_beta": 4,  "latency_mu": 700,  "latency_sigma": 120,  "risk_penalty": 0.15,  "cost_penalty": 0.05}
对于新工具：
如果和高质量旧工具相似：alpha/beta 继承部分先验如果是 marketplace 新工具：保守先验如果本地可信工具：中性偏乐观先验如果高风险工具：强保守先验
示例：
trusted built-in tool:Beta(8, 2)new local trusted skill:Beta(5, 3)new marketplace skill:Beta(2, 4)suspicious skill:Beta(1, 6)
这比 ε-greedy 冷启动安全得多。

Reward 不要只看 success
Agent 工具路由里的 reward 应该是复合的：
reward =  success- wrong_tool_penalty- fallback_penalty- latency_penalty- cost_penalty- user_interruption_penalty- rollback_penalty- false_success_penalty
建议归一化到 0-1：
Python运行reward = 0.0if call.success:    reward += 0.60if evaluator_pass:    reward += 0.25if no_fallback:    reward += 0.05if latency_ms <= latency_budget:    reward += 0.05if no_user_correction:    reward += 0.05if rollback:    reward -= 0.40if false_fix_suspected:    reward -= 0.50if policy_violation:    reward = 0.0
注意：
tool_call success 不是最终成功。比如 patch_file 成功写入文件，但 goal_evaluate failed，这个 reward 不能高。

上下文维度
不要只维护：
tool_success_rate
要维护：
success_rate by intentsuccess_rate by task_typesuccess_rate by project_typesuccess_rate by risk_levelsuccess_rate by model/providersuccess_rate by error_type
最小 key：
(intent, tool)
更好：
(intent, capability, tool)
高级：
(intent, capability, project_language, tool)
例如：
patch_file 在 Python 项目表现好但在 TypeScript monorepo 可能表现一般kimi_search 在中文知识检索表现好但代码库本地搜索不如 ripgrep

冷启动新 intent
新 intent 没历史时，不应该随机试工具。策略：
1. 继承父 intent 先验2. 使用 capability match3. 使用 offline eval seed4. 只在低风险候选中探索
例如新 intent 是：
refactor
可以继承：
review + execute
但权限上更靠近 execute，所以 risk policy 更严格。

新工具加入的上线策略
新工具不要直接进入主路由。
推荐 lifecycle：
shadow→ canary→ active→ preferred→ quarantined
shadow
只参与打分，不实际调用。
记录：如果启用它，它会排第几？它和实际选择工具是否一致？它是否经常误入候选？
canary
小流量，只在低风险场景启用。
risk <= lowconfidence margin 小用户无副作用任务
active
正常参与排序。
preferred
历史表现显著好于旧工具。
quarantined
失败率、超时、误路由、策略违规过高。

我对三种算法的最终选择
主策略：Constrained Thompson Sampling辅助策略：UCB for provider/tool variants兜底策略：ε-greedy within safe top-3
具体：
工具能力差异大：用 Thompson Sampling供应商能力相似：用 UCB低风险且需要探索：top-3 ε-greedy，ε=0.03-0.05高风险/写操作：禁止随机探索，只允许确定性选择或用户确认

TRM 冷启动落地版
最小实现：
Python运行def rank_tools(intent, query, candidates, context):    safe_candidates = policy_gate(candidates, context)    for tool in safe_candidates:        static = static_route_score(intent, query, tool)        prior = load_or_init_prior(intent, tool)        sampled_success = beta_sample(prior.alpha, prior.beta)        tool.final_score = (            0.50 * static            + 0.30 * sampled_success            + 0.10 * tool.health_score            + 0.05 * tool.latency_score            + 0.05 * tool.trust_score            - tool.risk_penalty        )    return sorted(safe_candidates, key=lambda t: t.final_score, reverse=True)
更新：
Python运行def update_tool_posterior(intent, tool, reward):    stat = load_stat(intent, tool)    if reward >= 0.7:        stat.alpha += 1    elif reward <= 0.3:        stat.beta += 1    else:        stat.alpha += 0.3        stat.beta += 0.3    save_stat(stat)

最终建议汇总
EventLog
SQLite 主存储可以。但必须异步批量写。热路径只写内存队列。ring buffer 用于实时 UI。SQLite 用于历史、metrics、self-heal 证据。
AST
能检测直白 __import__('os').system(...)。不能证明安全。任意 Python 技能不要进主进程。AST 只做 suspicious scoring。
TRM 冷启动
不要用裸 ε-greedy。推荐：规则过滤 + 静态先验 + Constrained Thompson Sampling。UCB 用于同能力 provider 选择。ε-greedy 只在低风险 top-3 内小比例探索。新工具先 shadow，再 canary，再 active。
压成一句话：

CRUX 的成长引擎应该是“安全约束下的概率排序器”，不是“随机试错器”。
