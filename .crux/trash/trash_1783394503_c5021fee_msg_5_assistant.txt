Q1：假修复检测的冷启动

你的两个方案里，A 和 B 不应该二选一。最佳方案是：

Seed policy + 本地学习 quarantine

也就是：

预置 seed 规则，避免已知死坑；
同时保留“24h 内同签名 3 次，第 4 次 quarantine”的在线学习。
不建议只用 B

只放行前三次的问题是：

新部署必踩坑
高成本 self-heal 会重复消耗
危险修复可能污染代码库
ComfyUI / pip / 环境类错误很容易无限试错

例如这些错误不应该靠真实失败学习：

pip install permission denied
CUDA out of memory
model file missing
ComfyUI custom node missing
网络超时 / 404
Python import path 错误
Windows 文件锁

它们很多不是“代码修复”能解决的。

但也不建议 seed quarantine 太硬

不要做成：

遇到某类错误 → 永久不重试

因为同一个错误类型在不同上下文下可能可修。

应该做成 seed policy，不是纯 quarantine list。

建议枚举：

allow_retry
limited_retry
requires_new_evidence
requires_user_action
downgrade_to_diagnosis

Q2：Manifest 迁移路径

你的迁移路径是合理的，但阶段 1 需要更保守一点。

你设想：

自动生成默认 manifest:
name = 文件名
runtime = prompt
permissions = []

这个对纯 prompt 技能没问题，但对现有 51 个技能里如果有脚本、浏览器、ComfyUI、文件操作类技能，就可能错误降级。

我建议阶段 1 改成：

自动推断 manifest，而不是统一 runtime=prompt。
阶段 1：兼容加载 + 自动生成 legacy manifest

生成：

JSON
{
  "schema_version": "1.0",
  "name": "browser-control",
  "version": "0.0.0-legacy",
  "description": "Auto-generated legacy manifest.",
  "runtime": "legacy_inferred",
  "permissions": [],
  "requires": [],
  "entrypoint": "skills/browser-control.skill.json",
  "integrity": "",
  "min_crux_version": "5.0.0",
  "author": "unknown",
  "legacy": true,
  "trust": "local_legacy"
}

然后根据文件类型推断：

.skill.json / .yaml      → declarative
.prompt / .md / .txt     → prompt
.py                      → python_subprocess
目录型技能               → inspect entrypoint
未知类型                 → disabled_pending_manifest

不要把 .py 技能推成 prompt。

阶段 2：自动补全建议

CRUX 可以生成一份建议 manifest，但不强制写入源文件：

.crux/generated_manifests/<skill>.manifest.json

并提示：

该技能疑似需要 permissions:
- filesystem:read
- cdp
- network:outbound

请确认后写入正式 manifest。

这样不会破坏现有技能，也不会默默授予权限。

阶段 3：新技能强制 manifest

这个没问题。

建议规则：

新技能：必须有 manifest
旧技能：可 legacy 加载，但降权
市场技能：必须 manifest + integrity
Python 技能：必须 runtime=python_subprocess 或 docker
python_inline：默认 deprecated
