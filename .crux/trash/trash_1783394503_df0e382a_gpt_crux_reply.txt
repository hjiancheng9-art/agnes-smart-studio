1. Agent 平台架构参考
已知最佳实践
1.1 用“运行时”统治 Agent，而不是让 Agent 统治运行时
生产级 Agent 的真正核心不是 prompt，而是：
RunStateEventLogToolCallCheckpointBudgetPolicyEvaluatorArtifactRollbackPlan
LangGraph 的设计非常接近这个方向：它是低层有状态工作流/Agent 基础设施，强调失败后恢复、长任务持久化、人类监督、短期/长期记忆、状态转移可视化与运行指标。Docs by LangChain
CRUX 已经有 Goal Mode、execute_plan、patch_file、self_heal、Quest、ADR、Retro，所以你的下一步不是“引入 LangGraph”，而是把 CRUX 自己的运行时标准化成 LangGraph-like control plane。
建议内部抽象：
Python运行class AgentRun:    run_id: str    goal_id: str    state: Literal["planning", "executing", "waiting_user", "verifying", "completed", "failed"]    contract: GoalContract    event_log: list[RunEvent]    checkpoints: list[Checkpoint]    tool_budget: Budget    risk_policy: RiskPolicy    artifacts: list[ArtifactRef]    evaluator_results: list[EvaluationResult]
不要把 AgentRun 当聊天记录。它应该是 事务日志 + 可恢复状态机 + 证据链。

1.2 单 Agent + 工具优先，多 Agent 作为升级路径
Anthropic 对 agentic workflow 的分类很有参考价值：
Prompt Chaining          固定步骤Routing                  分类后进入专用路径Parallelization          并行视角 / 投票 / 分区处理Orchestrator-Workers     中央调度器动态拆任务Evaluator-Optimizer      生成器 + 评估器循环Autonomous Agent         开放式、多步、工具反馈循环
其中 autonomous agent 适合开放问题，但必须依赖环境 ground truth、检查点、最大迭代数等控制条件。Anthropic
对 CRUX 来说，建议把 agent_swarm / multi_agent 从“能力入口”降级为“执行策略之一”。
推荐路由顺序：
1. deterministic workflow2. single agent + tools3. orchestrator + specialist workers4. parallel reviewers / testers5. swarm only for embarrassingly parallel tasks
不要默认 swarm。大规模并行子智能体最容易制造三类问题：
重复分析上下文发散假共识
CRUX 应该让 swarm 只用于：
代码库大范围扫描多文件静态审计多方案生成测试矩阵执行素材/提示词批量生成跨文档检索
不适合 swarm 的场景：
核心架构决策危险写操作数据库迁移发布/回滚用户意图含糊的任务

2. 工具编排和路由
已知最佳实践
2.1 工具描述必须是“机器契约 + 人类语义 + 路由特征”
OpenAI function calling 把 function tool 定义为 JSON Schema，让模型选择工具并填参数；Structured Outputs 则保证模型输出符合 JSON Schema。OpenAI开发者+1
MCP 的工具规范也强调：每个工具有唯一名称、元数据和 schema；工具由模型发现并调用，但实现层应该提供可见工具、调用指示和用户确认。Model Context Protocol
CRUX 的 Tool Contract 建议扩展为：
JSON{  "name": "patch_file",  "namespace": "crux.fs",  "intent": ["execute", "review"],  "summary": "Apply structured multi-file edits with backup, syntax validation, and rollback.",  "input_schema": {},  "output_schema": {},  "side_effects": ["filesystem.write"],  "risk_level": "medium",  "requires_confirmation": false,  "idempotency": "transactional_with_backup",  "timeout_ms": 30000,  "retry_policy": {    "max_attempts": 2,    "retry_on": ["syntax_check_failed", "transient_io_error"],    "never_retry_on": ["permission_denied", "destructive_scope_unclear"]  },  "fallbacks": ["create_patch_preview", "manual_diff"],  "examples": [    {      "when": "Need to edit several files and validate syntax",      "call": {}    }  ],  "anti_examples": [    "Do not use for deleting user data.",    "Do not use when the requested scope is ambiguous."  ],  "evals": ["tool_patch_file_basic", "tool_patch_file_rollback"]}
重点不是 schema 越长越好。Anthropic 的工具工程实践强调：工具要有清晰命名空间、返回有意义上下文、控制 token 成本，并通过评估来改进工具说明。Anthropic

2.2 大规模工具目录需要三段路由
CRUX 现在有多源工具：crux 内置、codex、kimi、qoder、codebuddy，加上 51 个技能包和 744 个市场技能。不能把所有工具暴露给主模型。
推荐三段式：
Stage 1: Intent Routersearch / review / execute / think / generate / statusStage 2: Capability Retriever按 intent、namespace、embedding、历史成功率、权限、延迟选 Top-K 工具Stage 3: Call Planner在 Top-K 工具中生成 tool plan，并附带 confidence / fallback / risk
伪结构：
Python运行candidate_tools = trm.retrieve(    intent=intent,    query=user_goal,    constraints={        "risk_max": policy.allowed_risk,        "side_effects": allowed_side_effects,        "latency_budget": budget.latency,        "provider_health": "ok"    },    top_k=12)plan = tool_planner.plan(    goal_contract,    tools=candidate_tools,    require_justification=True)
研究方向上，Gorilla/APIBench 和 ToolLLM/ToolBench 都证明“API/工具调用能力”需要检索、训练或专门评估；ToolLLM 覆盖 16,000+ 真实 API，BFCL 则专门评估函数选择、参数填写、并行/串行调用、拒答和多步状态场景。arXiv+2arXiv+2

主流方案对比
方案 A：全工具塞上下文优点：简单缺点：token 爆炸、工具混淆、误调用、延迟高结论：只适合 <30 工具方案 B：LLM 先选 namespace，再加载工具优点：简单可控缺点：namespace 设计差会误路由结论：适合 CRUX 当前阶段方案 C：embedding 检索 Top-K 工具优点：可扩到几百/几千工具缺点：语义相似不等于可执行匹配结论：需要叠加权限、成功率、schema 兼容性方案 D：学习型 router / bandit优点：能利用历史成功率、成本、延迟缺点：冷启动和奖励污染结论：适合 TRM 成长引擎方案 E：agent-to-tool / tool-to-agent retrieval优点：适合多 Agent + MCP 生态缺点：元数据质量要求高结论：适合 CRUX v5.5+
Tool-to-Agent Retrieval 的新研究方向是把工具和其父 Agent 放在同一向量空间中，以细粒度工具能力反推 Agent 选择；这比只检索粗粒度 Agent 描述更适合大规模 MCP/工具生态。arXiv

常见陷阱
陷阱 1：工具描述写给人看，不是写给模型用
很多工具描述只说“做什么”，不说：
何时用何时不用输入边界副作用失败模式返回值如何解释和相似工具的区别
已有研究分析 103 个 MCP server 的 856 个工具，发现 97.1% 的工具描述至少有一种“smell”，其中 56% 没有清楚说明目的；增强描述可提升任务成功率，但也会增加步骤数和成本。arXiv
CRUX 应该做 tool_lint：
purpose missingside_effects missingambiguous nameoverlapping toolno anti-exampleno output contractno failure taxonomyno risk levelno eval coverage

陷阱 2：fallback 只按 provider 切换，而不是按语义降级
错误的 fallback：
codebuddy 失败 -> qoder
正确的 fallback：
patch_file 失败-> create_patch_preview-> ask_user_to_confirm_scope-> manual_diff-> rollback
TRM 的 fallback 应分三类：
provider_fallback      同能力换供应商semantic_fallback      换更保守能力interaction_fallback   要用户确认 / 改为预览

陷阱 3：重试没有“新信息”
无脑 retry 会制造 token 黑洞。每次 retry 必须多一个证据：
新的错误日志新的诊断结果新的静态分析新的测试输出新的上下文检索新的用户确认
没有新证据，不允许 retry。

对 CRUX 的具体建议
把 TRM 升级成：
Tool Registry Mesh= Registry+ Contract+ Semantic Index+ Runtime Health+ Cost/Latency Model+ Success Memory+ Risk Policy+ Eval Harness+ Fallback Graph
新增几个内部工具：
trm_lint_tool(tool_id)trm_compare_tools(tool_a, tool_b)trm_route_explain(goal, selected_tools)trm_replay_failed_call(call_id)trm_promote_tool(tool_id)trm_quarantine_tool(tool_id, reason)
尤其要有 trm_route_explain，让 CRUX 能回答：
为什么选 patch_file 而不是 execute_shell？为什么降级到 preview？为什么这次不调用 generate_video？为什么这个技能被隔离？

3. 自我改进系统
已知最佳实践
3.1 Self-healing 的有效性来自“环境反馈”，不是来自自信反思
Anthropic 明确指出，Agent 在执行时必须从环境获得 ground truth，例如工具结果、代码执行结果，然后用这些反馈判断进展；还要有检查点和最大迭代数。Anthropic
CRUX 的 self_heal 不应是：
LLM 看代码 -> LLM 说问题 -> LLM 改代码
而应是：
Failing Evidence-> Reproduction-> Root Cause Hypothesis-> Minimal Patch-> Test Amplification-> Regression Check-> Behavior Diff-> Commit/ADR/Retro

3.2 调试型自修复比“日志型自修复”更可靠
InspectCoder 这类研究显示：让 LLM 通过断点、状态检查、运行时实验进行动态分析，比只看静态语义或表层日志更有效；其在 BigCodeBench-R 和 LiveCodeBench-R 上相对最强 baseline 的修复准确率提升 5.10%–60.37%，修复效率提升 1.67x–2.24x。arXiv
DebugRepair 也指出，很多 APR 方法只用 stack trace 这种结果级失败症状，缺少中间运行状态，容易误判根因；它通过测试语义净化、模拟插桩、调试驱动对话修复来改进。arXiv
CRUX 应该让 self_heal 自动选择：
static_onlytest_feedbackinstrumentationdebugger_sessionbisect_history
不要每次都直接 patch。

如何检测“假修复”
假修复常见类型：
1. 删除测试2. 放宽断言3. 捕获异常但吞掉错误4. 特判测试输入5. 改返回值但破坏真实语义6. 只修当前失败，不修根因7. 依赖偶发时序8. 降低类型/安全约束9. 改配置掩盖错误10. 让测试跳过
SWE-bench 已经证明“跑测试”是有效基准，但“测试通过不等于真实解决”；相关研究专门指出，测试套件并不穷尽，生成 patch 可能通过测试却不符合开发者预期。SWE-bench+1
CRUX 的假修复检测器建议：
Patch Suspicion Score =  test_deleted_weight+ assertion_weakened_weight+ broad_exception_weight+ behavior_surface_changed_weight+ complexity_delta_weight+ coverage_drop_weight+ mutation_survival_weight+ unrelated_file_touch_weight+ flaky_pass_weight
最低要求：
原失败测试必须失败于旧代码、通过于新代码旧通过测试仍通过新增至少一个针对根因的测试不能删除/跳过相关测试不能只改 snapshot/golden 除非用户确认patch diff 必须和 issue scope 对齐
更强要求：
mutation testingproperty-based testingdifferential testing重复运行 N 次过滤 flaky静态规则扫描行为回放

错误记忆库的衰减/淘汰策略
CRUX 的错误记忆不应该永久保留所有错误，否则会过拟合旧环境。
推荐错误记忆结构：
JSON{  "signature": "UnicodeEncodeError: surrogates not allowed",  "scope": "streaming_transport.encode_json",  "root_cause": "unpaired surrogate in model output or buffer",  "fix_pattern": "sanitize surrogate pairs before json encoding",  "anti_patterns": ["retry provider only", "switch model without sanitization"],  "first_seen": "2026-07-05",  "last_seen": "2026-07-05",  "hit_count": 3,  "success_count": 2,  "confidence": 0.78,  "decay_half_life_days": 45,  "validated_by": ["unit_test", "runtime_replay"],  "superseded_by": null}
衰减规则：
近期成功引用 -> 增强长期未命中 -> 衰减新版本后未复现 -> 降权相似错误但修复失败 -> 分裂 signature同一修复造成回归 -> quarantine被 ADR 替代 -> archived
CRUX 可以建立三层记忆：
Hot Error Cache       近 7-14 天高频问题Validated Playbook    已验证修复手册Cold Archive          历史案例，仅检索不自动应用

对 CRUX 的具体建议
给 self_heal 加一个硬门槛：
No Evidence, No Patch.No Reproduction, No Self-Heal.No Regression Check, No Completion.
新增内部阶段：
self_heal_diagnoseself_heal_reproduceself_heal_patch_candidatesself_heal_validateself_heal_false_fix_scanself_heal_memory_update
完成条件不应是“测试绿了”，而是：
JSON{  "pass": true,  "evidence": {    "reproduced_original_failure": true,    "fixed_failure": true,    "regression_passed": true,    "new_test_added": true,    "false_fix_score": 0.12,    "rollback_available": true  }}

GPT Store 类生态暴露出更大的问题：自定义 bot 的安全、合规、质量很难靠发布前审查解决。研究对 14,904 个 Custom GPTs 做安全分析，发现 95%+ 缺少足够安全保护，常见问题包括 roleplay attack、system prompt leakage、phishing 等。arXiv 另一项对 782 个 Custom GPTs 的黑盒合规评估发现 58.7% 至少出现一次疑似违规响应，并指出规模和黑盒性让系统性审核困难。arXiv
CRUX 的教训：
不要让技能市场变成 prompt 市场不要只看点赞/安装量不要把“能跑”当“可上架”不要让技能默认拥有文件/网络/执行权限不要让技能作者自称安全

Goal Mode 设计最佳实践
你现在的 Goal Mode 已经有：
finish_lineboundariesmax_stepsactive / paused / completed / failed / cancelledgoal_evaluatebudget
建议升级为：
JSON{  "goal_id": "goal_123",  "intent": "execute",  "finish_line": "All tests pass and requested UI bug is fixed.",  "boundaries": {    "allowed_files": ["src/**", "tests/**"],    "forbidden_actions": ["delete_user_data", "network_upload"],    "requires_confirmation": ["install_dependency", "deploy"]  },  "acceptance_tests": [    "pytest tests/test_scroll.py",    "python -m compileall src"  ],  "budgets": {    "max_steps": 12,    "max_tool_calls": 40,    "max_duration_seconds": 900,    "max_cost": "medium"  },  "autonomy_level": "L3",  "evidence_required": [    "diff_summary",    "test_result",    "rollback_checkpoint",    "evaluator_pass"  ]}
Goal Mode 的关键不是“让 Agent 更自主”，而是：
让自主行为有边界让每一步有证据让失败可恢复让完成可裁决

可解释性：Agent 如何说明自己的决策
CRUX 不应该展示完整思维链，而应展示 决策摘要 + 证据 + 替代方案：
我选择 patch_file 而不是 execute_shell，因为：1. 当前任务是结构化多文件修改；2. patch_file 有自动备份和语法校验；3. execute_shell 风险更高，且当前不需要任意命令执行。我没有直接部署，因为：1. Goal boundary 未授权 production 操作；2. 当前只完成本地验证；3. 发布属于高风险动作，需要确认。
推荐 UI 结构：
Decision- selected_action- reason- evidence- risk- fallback- user_control
不要给用户看 200 行日志。给用户看：
我做了什么为什么这么做依据是什么风险是什么你能如何介入

最后给 CRUX 的架构判断
你现在最危险的不是能力不足，而是 能力过剩后缺少统一治理面。
CRUX v5.0 的能力矩阵已经接近：
Agent runtimeIDE assistantworkflow enginecreative pipelinetool marketplaceknowledge baseCI/CD operatorbrowser automation controllerComfyUI orchestrator
所以 v5.1 的进化方向应该是：
少加新功能，多统一语义。少造新 Agent，多强化运行时。少相信自修复，多要求证据。少展示日志，多展示决策。少追求全自动，多做可控自主。
一句话定型：

CRUX 下一阶段不是 Agent Swarm，而是 Agent Governance Runtime。
让每个 Agent、工具、技能、补丁、生成任务都被契约化、证据化、可回滚、可评估。
