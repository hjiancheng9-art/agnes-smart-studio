下面是 CRUX Studio TUI 子系统审计与 4 阶段修复计划。核心原则：先测试止血，不碰真实终端；再拆主界面；再治理渲染/异步/输入；最后做质量门禁。

1. TUI 债务清单
A. 架构债务

ui/tui_v2.py 2197 行，明显是超级文件，可能混合了：

App 生命周期

布局构建

输入处理

消息渲染

动画刷新

状态管理

后端通信

快捷键/命令分发

widgets_v2.py 926 行，组件职责可能过宽，缺少单组件契约测试。

tui_app.py、tui_v2.py 可能存在入口职责重叠。

message_pane.py、widgets_v2.py、theme_v2.py 可能都参与渲染，边界不清。

animation_gov.py 虽小，但涉及刷新节流，是 TUI 崩溃/卡顿高风险点。

clipboard_image.py 涉及系统剪贴板、图片、平台差异，不能直接在单元测试里依赖真实 OS。

panels/ 和 screens/ 如果直接 import 主 App，容易导致循环依赖和测试难写。

B. 测试债务

38 个 TUI 文件，仅 3 个测试文件覆盖，文件级覆盖约 7.9%，严重不足。

现有 77 tests 主要可能覆盖协议/心跳/consumer，不覆盖真实 UI 组件行为。

缺少 tui_v2.py 的生命周期测试。

缺少 widgets_v2.py 的组件事件测试。

缺少 theme_v2.py 的主题 token、颜色合法性、降级测试。

缺少 message_pane.py 的消息追加、裁剪、Markdown/Rich 渲染、长消息保护测试。

缺少 animation_gov.py 的 throttle/debounce 测试。

缺少 Windows 终端兼容测试：ANSI、emoji、宽字符、颜色降级、终端尺寸。

缺少异步测试：Textual worker、外部 async 任务、取消、关闭、重连。

缺少快照测试：不用真实终端，也可以验证 renderable/tree/text snapshot。

缺少回归测试：闪退、输入焦点丢失、滚动卡死、高频刷新打爆。

C. Textual/Rich 典型风险

import 阶段创建 App 或启动事件循环。

测试中直接跑真实终端。

UI 回调里做阻塞 IO。

高频 refresh() / update() 无节流。

set_interval() 没有取消。

worker 没有正确退出。

asyncio.run() 与 Textual 自己的事件循环冲突。

Rich renderable 构造异常被吞掉或导致整屏崩。

宽字符、emoji、中文、ANSI 控制符导致布局计算异常。

Windows Terminal / legacy console 能力不一致。

D. 运行时稳定性债务

缺少统一 TUI 状态模型：connected / disconnected / busy / streaming / error / degraded。

缺少统一异常边界：widget 异常不应导致整个 TUI 崩溃。

缺少统一日志：TUI 崩溃时需要记录最后事件、最后输入、最后刷新、最后 worker 状态。

缺少 backpressure：后端流式消息过快时，message pane 不能每 token 刷新一次。

缺少 terminal capability detection：终端不支持颜色/动画/Unicode 时应降级。

缺少 headless/fake terminal 测试模式。

建议拆分方向：

ui/tui_v2/
  __init__.py
  app.py              # 主 App 壳
