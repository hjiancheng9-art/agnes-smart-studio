P2：ProductionEngine 增加 Job Runtime
视频生成比图片生成更容易失败、超时、卡队列。现在必须把 generateVideo() 从普通函数升级为 job。
最小结构：
JavaScript{
  jobId,
  type: "video_generation",
  status: "created | running | succeeded | failed | retrying | cancelled",
  provider: "comfyui",
  prompt,
  params,
  startedAt,
  finishedAt,
  outputAssets,
  error
}
需要支持：
1. 超时
2. retry
3. cancel
4. progress
5. trace span
6. job status 查询
下一步 server API 可以变成：
POST /api/pipeline/video
GET  /api/jobs/:jobId
GET  /api/jobs/:jobId/result

P3：QualityGate V2：从结构评分升级到真实视频检查
现在第一条视频出来后，质量门要开始检查真实文件。
先不要急着做“审美 AI 评分”，先做硬指标：
FileGate：文件存在、大小合理、可读取
DurationGate：时长是否达标
FrameGate：帧数是否达标
CoverGate：封面图是否存在
LineageGate：资产血缘是否完整
ParamGate：参数是否记录完整
QualityReport 示例：
JavaScript{
  passed: true,
  score: 0.82,
  checks: {
    fileExists: true,
    playable: true,
    durationOk: false,
    frameCountOk: true,
    coverExists: true,
    lineageComplete: true
  },
  issues: [
    {
      severity: "medium",
      message: "Video duration is only 1.04s; target is 4s"
    }
  ]
}
这次生成的视频很适合作为第一条 regression case。

P5：从 txt2vid 走向 image2video
txt2vid 跑通以后，真正可控的视频生产应该优先做：
首帧图 → image2video
原因很简单：
txt2vid 随机性强
image2video 可控性强
角色、场景、构图更容易保持
下一条 Golden Path 建议：
text prompt
→ SDXL 生成首帧
→ LTX image2video 生成 4 秒视频
→ video asset
→ quality report
这会比纯 txt2vid 更接近真正工作台能力。

现在不要做什么
不要继续大规模重构
架构已经过关。现在重构收益下降。
不要马上做长视频

不要急着接剪辑导出

不要急着做复杂 UI
