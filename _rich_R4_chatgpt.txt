正方：多行输入合理

论点1：AI 编程任务天然长输入。
用户经常粘贴错误栈、代码片段、需求说明、JSON、日志、Prompt。如果只能单行，体验会非常差。

论点2：CRUX 不是传统 shell。
传统终端命令通常是一行；但 AI 输入更像“任务说明”。任务说明天然可能是多段文字。

论点3：多行输入能减少外部编辑器依赖。
用户不需要先在 VS Code 写好 Prompt 再粘贴，可以直接在 TUI 内组织复杂任务。

反方：多行输入也有问题

论点1：占用底部空间。
TUI 高度宝贵，多行输入框一展开，消息区就被挤压。你之前遇到“输入框不用显示 3 行，1 行就行”，就是这个问题。

论点2：终端用户习惯单行。
开发者在终端里默认预期是一行命令，输入框太像聊天应用会破坏终端感。

论点3：多行编辑复杂度高。
光标移动、历史浏览、补全、换行、粘贴、滚动、快捷键冲突都会变复杂。

我的方案

默认单行，自动增长，最大 3-5 行。

具体：

默认高度：1 行
输入变长：自动增长到 2-3 行
复杂粘贴：最多 5 行
超过 5 行：输入区内部滚动

这样兼顾两边：

平时像终端命令行，轻；

需要长 Prompt 时像编辑器，够用；

不默认占 3 行，避免底部臃肿。

2. Enter 发送 vs Enter 换行

这是输入体验的核心矛盾。

正方：Enter 发送合理

论点1：符合 AI 聊天习惯。
ChatGPT、Claude、Cursor Chat、Copilot Chat，大多数都是 Enter 发送，Shift+Enter 换行。

论点2：高频动作应该最短路径。
发送是最高频动作。如果每次都要 Ctrl+Enter，会降低连续对话效率。

论点3：单行默认时 Enter 发送自然。
如果当前输入是一行，Enter 发送符合用户预期。

反方：Enter 换行才符合终端肌肉记忆

论点1：终端几十年的习惯是 Enter 确认当前行，不等于发送整段聊天。
尤其在多行编辑状态下，用户很容易以为 Enter 是换行，结果误发。

论点2：开发者经常输入结构化内容。
代码、JSON、Markdown、错误栈都需要换行。Enter 发送会让用户紧张。

论点3：误发成本高。
AI 助手一旦发送，可能触发工具调用、文件修改、模型计费、上下文污染。误触不是小问题。

我的方案：双模式 Enter

我不建议死板规定“Enter 永远发送”或“Enter 永远换行”。

建议采用 Smart Enter：

普通单行模式：
Enter = 发送
Shift/Alt+Enter = 换行

多行编辑模式：
Enter = 换行
Ctrl+Enter = 发送
Esc = 退出多行模式

关键是：什么时候进入多行编辑模式？

建议触发条件：

1. 用户按 Shift/Alt+Enter 主动换行
2. 用户粘贴了多行文本
3. 输入内容包含代码块 ``` 
4. 输入内容超过 2 行宽度
5. 用户按 Ctrl+E 进入编辑模式

界面要明确提示当前模式：

NORMAL  Enter send · Shift+Enter newline

进入多行后变成：

MULTI   Ctrl+Enter send · Enter newline · Esc single-line

我的最终判断：

默认 Enter 发送，但一旦进入多行编辑状态，Enter 改为换行，Ctrl+Enter 发送。

这是最符合 CRUX 的折中方案。

3. /命令体系：合理范围多大？

当前设计：

/clear
/model
/provider
/help
/export
正方：命令体系必要

论点1：结构化控制比自然语言可靠。
“切换模型”“清屏”“导出对话”“查看帮助”这些是明确操作，用命令比自然语言更稳定。

论点2：开发者熟悉 slash command。
Discord、Slack、Claude Code、Cursor、GitHub Copilot Chat 都有类似命令，学习成本不高。

论点3：命令可发现、可补全。
输入 / 弹出菜单，比让用户记住自然语言表达更好。

反方：命令太多会变成第二套 shell

论点1：AI 助手不应该逼用户背命令。
如果用户需要记 30 条命令，体验会从“智能助手”退化为“复杂 CLI”。

论点2：命令和自然语言会重叠。
比如“帮我切到 deepseek-v4-flash”和 /model deepseek-v4-flash 都能做，体系可能混乱。

论点3：命令越多维护成本越高。
每条命令都要补全、帮助文档、错误处理、权限确认、测试。

命令体系应该保持多少条？

我的建议：

核心常驻命令：5-7 条
高级命令：隐藏在 /help 或 /settings 里
Debug 命令：开发模式才显示

不要超过 10 条常驻命令。
超过 10 条，用户就开始记不住了。

什么情况下用命令，什么情况下用自然语言？
应该用命令的场景
状态控制
会话控制
模型/Provider 切换
导出/清空/重置
查看帮助
打开设置

因为这些操作要求确定性。

例如：

/model deepseek-v4-flash
/provider local
/clear
/export
/help
应该用自然语言的场景
分析代码
修改文件
生成方案
解释错误
规划架构
执行复杂任务

例如：

帮我检查这个模块为什么启动失败
把这个 TUI 输入区改成单行自适应
给我设计一个 ComfyUI workflow validator
混合场景

有些可以两者都支持：

/model qwen
切换到 qwen 模型

但底层应该路由到同一个 action。

我认为最重要的 5 个命令
1. /help

作用：查看命令、快捷键、当前模式。

必须有。
TUI 复杂后，没有 /help 新人会迷路。

2. /model

作用：查看/切换模型。

AI 编程助手里模型切换是高频刚需。

示例：

/model
/model deepseek-v4-flash
3. /clear

作用：清屏或清当前会话显示。

注意：要区分“清屏”和“清上下文”。

建议：

/clear screen
/clear context

不要让 /clear 默认清掉上下文，避免误操作。

4. /status

作用：打开当前系统状态详情。

比常驻 Dashboard 更合理。

显示：

model
provider
cwd
git
context
tools
TRM
ComfyUI
5. /export

作用：导出当前会话、日志、执行轨迹。

CRUX 做架构审计、代码修复、Prompt 设计都很需要导出。

可选但不应常驻的命令
/provider
/settings
/theme
/history
/retry

其中 /provider 可以并入 /model，比如 /model 面板里顺带显示 provider。

4. @文件补全：价值大不大？
正方：@file / @folder 是关键差异化

论点1：AI 编程助手必须绑定项目上下文。
纯聊天最大的问题是上下文不准。@file 可以明确告诉模型要看哪个文件，减少误读。

论点2：比粘贴路径更自然。
用户输入 @core/download/manager.py，系统可以补全、校验、读取、加入上下文。粘贴路径只是字符串，语义弱。

论点3：能形成可追踪引用。
@file 不只是文本，它可以变成结构化引用：文件路径、版本、hash、片段范围、是否已读。后续审计和回放更可靠。

反方：实现复杂，用户也可以粘贴路径

论点1：补全实现确实复杂。
要处理路径扫描、忽略规则、Git ignore、大项目性能、模糊搜索、中文路径、Windows 路径、权限错误。

论点2：路径粘贴已经能解决一部分问题。
开发者知道文件路径，直接粘贴也能用，不一定需要复杂补全。

论点3：补全菜单可能干扰输入。
用户打 @ 只是想写邮箱、装饰符、注解时，弹出文件菜单会烦。

我的判断

@文件补全 必须做，但要克制。

它对 CRUX 的价值非常高，因为 CRUX 不是普通聊天助手，而是 AI 编程/多工具执行系统。
@file 是把自然语言变成结构化任务的重要入口。

但实现上不要一开始做太重。

推荐实现策略
P0：基础文件引用
@filename
@path/to/file.py
@folder/

支持：

模糊搜索
最近文件优先
当前项目优先
.gitignore 过滤
P1：范围引用
@core/download/manager.py:120-180

这个很重要。
用户经常只想让模型看某一段，不想塞整个文件。

P2：语义引用
@last-error
@current-file
@changed-files
@git-diff

这些比普通路径更强。

尤其推荐：

@diff
@staged
@recent
@errors
如何避免干扰？

@ 触发补全，但要有规则：

@ 后面跟路径字符才弹出
邮箱格式不触发
代码块内部不触发
用户按 Esc 可关闭

例如：

hello@test.com

不应该弹文件补全。

5. 输入历史：需要吗？保留多久？
正方：需要输入历史

论点1：开发者会重复执行类似请求。
比如反复让模型“继续修复”“重新运行测试”“解释这个错误”“导出报告”。历史能提高效率。

论点2：命令尤其需要历史。
/model、/export、/status、@file 这些操作重复性高。

论点3：AI 任务也有可复用性。
长 Prompt、审计模板、测试指令、代码修复模板都值得回调。

反方：AI 对话历史和终端命令历史不是一回事

论点1：AI 输入通常上下文相关。
上一条 Prompt 在当前状态下有意义，换个上下文可能没意义。盲目翻历史可能误用。

论点2：长 Prompt 历史难浏览。
终端命令历史通常短；AI Prompt 可能几十行，用上下箭头翻会很灾难。

论点3：隐私和污染问题。
输入历史可能包含路径、API Key、错误日志、敏感内容。永久保存有风险。

我的方案：分层历史

不要把所有输入混在一个历史里。

建议分三类：

1. 命令历史
2. Prompt 历史
3. 文件引用历史
具体设计
上下箭头

只浏览当前输入模式下的短历史。

空输入 + ↑：上一条短输入/命令
输入中 + ↑：移动光标或历史搜索，取决于光标位置
Ctrl+R

打开历史搜索。

Ctrl+R → search prompt/command history

支持筛选：

cmd:
/model
@file
最近使用
保存多久？

我的建议：

命令历史：保存 1000 条
Prompt 历史：保存最近 200 条
长 Prompt：只保存摘要 + 可选全文
文件引用历史：保存最近 100 个

更重要的是：提供开关。

history.enabled = true
history.save_prompts = true
history.max_prompts = 200
history.redact_secrets = true
哪些不应该保存？

默认不保存或脱敏：

包含 API key 的输入
包含 password/token/secret 的输入
超长日志全文
图片粘贴内容
二进制内容
最终判断

输入历史需要保留，但不能照搬 shell history。

结论：

命令历史强保留，Prompt 历史弱保留，长输入走搜索，不靠上下箭头硬翻。

6. 输入区设计 3 条铁律
铁律一：默认轻，复杂时自动升级

输入区默认必须像单行命令行，只有在用户需要时才变成多行编辑器。

正方

论点1：保护消息区空间。
单行默认不会挤压主要内容。

论点2：符合终端使用感。
用户一进来不会觉得这是一个臃肿聊天框。

论点3：渐进复杂度。
新手可以直接输入，高手可以粘贴多行、引用文件、用命令。

反方

论点1：自动升级可能不可预测。
用户可能不理解为什么输入框突然变高、Enter 行为改变。

论点2：长 Prompt 用户需要明确编辑模式。
自动增长不如明确进入“编辑模式”安全。

论点3：实现复杂。
要处理高度、滚动、快捷键、历史、补全联动。

结论

铁律成立，但要有明显模式提示：

NORMAL Enter=send
