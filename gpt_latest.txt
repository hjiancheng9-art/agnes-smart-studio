下面是我按 2026-07-09 当前资料整理的“最新干货版”。重点不是基础教程，而是你们做 ComfyUI 智能体 / workflow compiler 时应该内置到系统里的判断规则。

1. ComfyUI workflow JSON 最新格式变化
1.1 两种 JSON 必须彻底区分

现在仍然要严格区分：

A. Workflow Save Format
   用于前端保存、拖入、编辑。
   包含 version / state / nodes / links / groups / config / extra / models 等。

B. API Prompt Format
   用于 POST /prompt 执行。
   根对象是 node_id -> {class_type, inputs, _meta}。

官方文档明确说，浏览器保存格式和 API 格式不同；API 格式应该通过 File → Export Workflow (API) 导出，省掉 UI 布局信息，适合程序提交。
ComfyUI

1.2 最新 Save Format 是 Workflow JSON v1.0

官方 Workflow JSON schema 当前最新是 Version 1.0。顶层关键字段：

JSON
{
  "version": 1,
  "config": {},
  "state": {},
  "groups": [],
  "nodes": [],
  "links": [],
  "reroutes": [],
  "extra": {},
  "models": []
}

必需字段是：

version
state
nodes

官方 schema 里 version 固定为 1，并且 state、nodes 是 required。
ComfyUI
+1

1.3 最新 Save Format 的 node outputs 结构

Save Format 里的每个 node 是数组元素，不是 API Format 那种 "3": {...}。

outputs 当前 schema 是：

JSON
{
  "name": "IMAGE",
  "type": "IMAGE",
  "links": [12, 13],
  "slot_index": 0
}

字段含义：

name       输出口显示名
type       类型，可为 string / string[] / number
links      连到哪些 link id，可为 number[] 或 null
slot_index 输出槽位索引，可为 integer 或 string

官方 v1.0 schema 明确规定 outputs 是数组，每项至少需要 name 和 type，可带 links 和 slot_index。
ComfyUI

1.4 links / reroutes / models 是新编译器要认真处理的字段

Save Format 的 links 现在是对象数组，不是老 LiteGraph 风格的纯数组更安全假设：

JSON
{
  "id": 1,
  "origin_id": 4,
  "origin_slot": 0,
  "target_id": 3,
  "target_slot": 1,
  "type": "MODEL"
}

官方 schema 还包括 reroutes、extra.info、extra.linkExtensions、models，其中 models 每项包含 name、url、directory，可选 hash、hash_type。这对你们的“可复现 manifest / 自动下载模型”很有价值。
ComfyUI
+1

1.5 API Prompt Format 没有 outputs 字段

执行用 API Format 仍然是：

JSON
{
  "4": {
    "class_type": "CheckpointLoaderSimple",
    "inputs": {
      "ckpt_name": "model.safetensors"
    },
    "_meta": {
      "title": "Load Checkpoint"
    }
  },
  "6": {
    "class_type": "CLIPTextEncode",
    "inputs": {
      "clip": ["4", 1],
      "text": "prompt"
    }
  }
}

API Format 的连接关系写在 inputs 里：

JSON
"model": ["4", 0]

不要给 /prompt 直接提交 Save Format。官方说明 API Format 使用数字 node id 作为 key，保留 widget 值，但不包含位置、颜色、分组等前端布局信息。
ComfyUI

1.6 真正的 breaking point：Custom Node V3 Schema

2025 之后你们要重点关注 Custom Node V3 schema，因为它影响节点能力扫描和 object_info 解析。

V3 变化：

V1:
  INPUT_TYPES
  RETURN_TYPES
  FUNCTION
  CATEGORY
  NODE_CLASS_MAPPINGS

V3:
  define_schema()
  inputs=[...]
  outputs=[...]
  execute()
  comfy_entrypoint()
  ComfyExtension

官方说明 V3 的主要变化是：输入输出用对象定义，不再是字典；执行方法固定为 execute；通过 comfy_entrypoint() 返回 ComfyExtension 暴露节点，而不是旧的 NODE_CLASS_MAPPINGS；未来节点特性只会加到 V3 schema。
ComfyUI

对你们编译器的影响：

不要只静态扫描 NODE_CLASS_MAPPINGS。
必须优先用 /object_info 获取真实节点定义。
自定义节点扫描器要兼容 V1 + V3。
节点合同 NodeContract 应从 object_info 动态生成。

2.6 视频节点生态

视频现在分两条路线：

A. 原生支持优先：
   Wan2.1 / Wan2.2 / LTX-2.3 原生模板

B. 自定义节点补充：
   VideoHelperSuite / AnimateDiff-Evolved / LTXVideo / WanVideoWrapper

ComfyUI 官方 Wan2.1 文档说明 Wan2.1 是阿里 2025 年 2 月开源的视频模型，包含 14B 和 1.3B 两个版本，覆盖 T2V/I2V，轻量版约 8GB VRAM 可运行。

ComfyUI

3. ComfyUI API 最新用法
3.1 /prompt 仍是核心入口

官方 server routes 说明：前端提交 workflow 时会 POST 到 /prompt，它会验证 prompt 并加入执行队列；成功返回 prompt_id 和 number，失败返回 error 和 node_errors。
ComfyUI

推荐 payload：

JSON
{
  "prompt": {
    "3": {
      "class_type": "KSampler",
      "inputs": {}
    }
  },
  "client_id": "uuid",
  "prompt_id": "uuid"
}

ComfyUI 官方 websocket 示例也是这么提交：{"prompt": prompt, "client_id": client_id, "prompt_id": prompt_id}。
GitHub

3.2 WebSocket 必须接

不要只轮询 /history。

/ws 官方用于实时双向通信，可以接收执行进度、节点状态、错误、队列更新。消息类型包括：

status
execution_start
execution_cached
executing
progress
executed

官方 websocket 示例通过监听 executing，当 data.node is None 且 prompt_id 匹配时判断执行完成，然后再走 /history/{prompt_id} 获取结果。

4. 2024-2025 高级工作流范式
4.1 图像生成主线：SDXL → Flux

推荐优先级：

低显存 / 稳定出图：
  SD1.5 + HiresFix / Tiled VAE / Ultimate upscale

4.4 角色一致性 / FaceSwap 范式

不要把“角色一致性”单一化。建议拆成 4 类：

A. Prompt + Seed + Model Lock
   最轻量，适合非脸部角色一致性。

B. IPAdapter / FaceID
   适合参考图主体、风格、身份迁移。

C. InstantID / PuLID
   适合人脸身份保持生成，不是简单贴脸。

D. ReActor / FaceSwap
   适合后处理换脸，但要明确风险和限制。

InstantID 论文定位为 zero-shot identity-preserving generation，只需单张人脸图，可保持身份并适配 SD1.5/SDXL 等模型；PuLID 论文则强调 tuning-free ID customization，同时保持背景、光照、构图、风格等编辑性。
arXiv
+1

ReActor 是快速换脸节点，但它本质是后处理 face swap，不等同于“原生角色一致性生成”；其 README 也强调使用者需要自行承担责任，并带有成人内容防护。

模型精度：fp16 / bf16 / fp8 / int8 / nf4 / gguf q4

batch size
是否 refiner
是否 ControlNet
是否 IPAdapter
是否 video
是否 tiled VAE / offload
ComfyUI 版本和 attention backend

5.2 SDXL

实用建议：

6GB:
  不推荐小白默认 SDXL。

8GB:
  可尝试 SDXL basic，限制 1024 以下或接近 1024，batch=1，tiled VAE。

12GB:
  SDXL basic 稳定，轻量 upscale 可用。

16GB:
  SDXL + refiner 可用。

24GB:
  SDXL + refiner + upscale + detailer 全链路。

SDXL 自身包含更大的 UNet、第二文本编码器和 refiner，显存压力天然高于 SD1.5；ComfyUI 可能通过卸载等机制跑得更省，但 refiner/upscale/detailer 叠加后仍应按 12-24GB 档规划。
