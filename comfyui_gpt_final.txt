真实 ComfyUI 在线验证

全量语法检查

集成测试通过

但还有 5 个需要在 RC1 前补的高危点，不一定是致命 bug，但属于“上线后最容易爆炸”的地方。

A. 冷启动干净环境测试

你现在是在本机已配置环境里通过测试。RC1 前必须做：

Bash
git clone fresh
python -m venv .venv
pip install -r requirements.txt
python launch.py
python -m pytest

目标是确认：

没有依赖本机隐藏环境变量

没有依赖 IDE 工作目录

没有硬编码路径

没有遗漏 requirements

launch.py 在干净环境可复现启动

这是最容易遗漏的发布致命点。

B. 真实失败场景回归测试

现在你测了“成功链路”，但 RC1 要测“坏链路”：

ComfyUI 没开

ComfyUI 中途断开

checkpoint 缺失

workflow JSON 损坏

workflow 节点不存在

显存 OOM

请求超时

API 并发请求

生成过程中用户取消

输出目录无权限

你的 V4 价值核心是 diagnose→repair→validate→execute，所以失败链路比成功链路更重要。

C. API 鉴权/本地边界

如果 server.py 默认开 0.0.0.0:5000 或无鉴权，后面可能被本机局域网误调用。RC0 可以接受，但 RC1 至少要明确：

Python
运行

API_TOKEN = optional

并在 README 里写清楚。

D. 运行日志与 run_id 追踪

现在如果用户说“刚才那张图失败了”，你要能定位：

run_id
prompt
workflow_id
comfy_prompt_id
status
error
repair_attempts
output_files
duration_ms

没有 run_id 追踪，RC0 可以跑；有用户后会很难排错。

E. 发布包边界

113 个 .py 文件已经可控，但 RC1 前最好确认：

哪些是 public API

哪些是 internal

哪些 wrapper 会在 v4.1 删除

README 中写清楚推荐入口：launch.py / main.py / server.py / ComfyUIAgent

否则用户和你自己后续都会绕回旧入口。

结论：没有看到立即阻止 RC0 的致命遗漏。
但 RC1 的重点必须是“干净环境 + 失败场景 + 追踪日志 + 安全边界”。

3. 这个项目现在最大的技术债是什么？

最大的技术债不是旧文件了。旧文件已经包装器化，问题降级了。

现在最大的技术债是：

缺少“契约化运行层”

也就是 Showrunner、Orchestrator、Agent、ComfyUI 四层之间虽然已经跑通，但它们之间的输入输出契约还没有完全固化。

现在架构是：

Showrunner
  ↓
Orchestrator
  ↓
ComfyUIAgent V4
  ↓
FlowCompiler / Diagnose / Repair / Execute
  ↓
ComfyUI API

但每层之间应该有明确 schema：

Python
运行
CreativeGoal
ScenePlan
ShotPlan
WorkflowRequest
CompiledWorkflow
ExecutionRequest
ExecutionResult
RepairReport
RunTrace

如果这些契约不固化，后面会出现三个问题：

1. Showrunner 会慢慢侵入 Agent

比如 Showrunner 开始关心 checkpoint、sampler、node name。
这会让“导演层”和“摄影机执行层”再次耦合。

2. Agent 会反向污染创意层

比如 Agent 把 ComfyUI 的错误、节点概念、workflow 细节暴露给上层。
这会让用户体验变得工程化，而不是创作化。

3. 测试会越来越难写

没有契约，测试只能测“最终有没有图”。
有契约，才能测：

Goal → ScenePlan 是否合理
ScenePlan → ShotPlan 是否完整
ShotPlan → WorkflowRequest 是否合法
WorkflowRequest → Workflow JSON 是否连通
ExecutionResult 是否可追踪
RepairReport 是否可解释

所以我建议你下一个大重构不是再拆文件，而是建一个：

comfyui_agent/contracts/

里面放：

Python
运行
creative.py
workflow.py
execution.py
repair.py
trace.py
api.py

核心例子：

Python
运行
from pydantic import BaseModel, Field
from typing import Any, Literal

class WorkflowRequest(BaseModel):
    prompt: str
    negative_prompt: str = ""
    width: int = 1024
    height: int = 1024
    steps: int = 30
    cfg: float = 7.0
    seed: int | None = None
    workflow_type: Literal["txt2img", "img2img", "video", "upscale"] = "txt2img"
    style_tags: list[str] = Field(default_factory=list)

class CompiledWorkflow(BaseModel):
    workflow: dict[str, Any]
    node_count: int
    entry_nodes: list[str]
    output_nodes: list[str]
    warnings: list[str] = Field(default_factory=list)

class ExecutionResult(BaseModel):
    ok: bool
    run_id: str
    comfy_prompt_id: str | None = None
    output_files: list[str] = Field(default_factory=list)
    duration_ms: int | None = None
    error: str | None = None
    repair_attempts: int = 0

然后所有模块只传这些对象，不再乱传 dict。
