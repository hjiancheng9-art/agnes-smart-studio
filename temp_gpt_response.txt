下面按“先救测试收集 → 再切架构边界 → 再补核心回归 → 再治理工具系统 → 最后固化质量门禁”的顺序审计。

总体判断

CRUX Studio 现在不是“小修小补”阶段，而是大型智能体平台进入架构债务清算期。当前最大风险不是代码量，而是：

测试系统已经失去快速反馈能力。

核心模块数量多，但边界可能不够清晰。

CDP / MCP / LSP / TDD / Tool Registry / Agent Swarm 都是高状态、高异步、高副作用模块，若缺少分层测试，很容易出现“能跑但不可维护”。

技能包和内部工具数量巨大，缺少统一契约测试会导致工具路由、权限、环境、版本兼容持续退化。

tests/test_zcode_launcher.py ImportError 阻塞收集，说明测试入口有副作用或依赖污染，必须先修。

1. 债务清单
A. 测试债务

tests/test_zcode_launcher.py 的 ImportError 阻塞全部测试收集，说明测试文件在 import 阶段执行了不安全依赖。

pytest 全量超过 180 秒，说明测试未分层，缺少 unit / integration / e2e / slow / browser / network 标记。

测试/源码比 19.2%，对 24 万行项目偏低，尤其不适合智能体系统。

很可能存在“测试导入即启动服务 / 浏览器 / 网络连接 / 文件扫描”的问题。

CDP、MCP、LSP、GitHub、TDD、pipeline 等副作用模块如果没有 mock/fake 层，会导致测试慢、不稳定、依赖本机环境。

缺少契约测试：工具接口、技能包接口、MCP 协议、TRM 路由结果都需要固定输入输出。

缺少回归测试：历史 bug，例如输入区消息炸开、滚动失效、模型断连，应沉淀为 regression tests。

缺少性能预算测试：pytest 超时、工具路由耗时、上下文构建耗时、知识图谱检索耗时都应有上限。

B. 架构债务

core/ 模块过多，已经形成“超级核心层”。core 既包含 agent、browser、executor、registry、knowledge、wiki、TDD、pipeline、self-heal，职责过宽。

多个执行相关模块可能存在重叠：code_executor、sandbox、pipeline_executor、plan_executor、tdd_engine、task_governor。

多个智能体编排模块可能存在边界重叠：agent_swarm、multi_agent、goal_mode、quest_engine、retrospective、review_engine、self_heal。

工具系统规模大：80+ 内部工具、56 技能包、750 市场技能，若缺少统一 ToolSpec / ToolResult / ToolError / Capability 模型，会快速失控。

tool_registry_mesh 和 trm_growth 可能是核心复杂度中心，必须明确“路由、发现、授权、执行、降级、统计”边界。

CDP 浏览器控制天然脆弱，若选择器、事件监听、重连、输入缓冲没有隔离，会导致 UI 状态污染。

MCP 与本地工具如果共享执行路径但错误模型不同，会造成重试、降级、权限边界混乱。

context_manager、knowledge_graph、wiki_engine 可能重复承担“记忆/知识/检索/上下文裁剪”职责，需要统一分层。

approval_gate、sandbox、file_locker、code_executor 涉及安全边界，必须不能被 agent 或工具绕过。

C. 代码质量债务

582 个 Python 文件，若没有强制 import lint、循环依赖检测、类型检查，很容易出现隐式耦合。

JSON 738 文件、66k 行，说明配置/蓝图/技能元数据量很大，必须用 schema 校验。

TypeScript/JavaScript 文件虽少，但如果承载 UI、CDP bridge 或 dashboard，缺少测试也会成为高风险入口。

可能存在动态 import、字符串路由、反射式工具加载，静态分析覆盖不足。

异步模块可能混用 sync/async，容易出现事件循环冲突、阻塞、未 await、任务泄漏。

需要统一错误类型，否则工具失败、浏览器断连、LLM 失败、权限拒绝、超时都会被混成普通异常。

D. 文档债务

Markdown 137 文件，接近 2 万行，文档量不小，但可能存在版本漂移。

缺少“当前真实架构图”：Agent → Planner → Tool Registry → Executor → Browser/MCP/LSP/GitHub 的边界需要固定。

缺少“测试运行指南”：哪些测试可本地快速跑，哪些需要浏览器、网络、GitHub token、MCP server。

缺少“工具开发规范”：新工具如何声明能力、权限、输入输出 schema、错误、超时、幂等性。

缺少“故障处理手册”：CDP 断连、模型断连、输入区污染、滚动锁死、MCP 不可用、LSP 崩溃等应有 runbook。

阶段1: 解除测试阻塞，建立最小可信反馈环

目标：先让测试能稳定收集、快速运行、可分层执行。不要一开始重构架构。

 定位阻塞导入错误：python -m pytest tests/test_zcode_launcher.py -vv --tb=short

 单独验证 pytest 收集：python -m pytest --collect-only -q

 修复 tests/test_zcode_launcher.py 的 import 阶段副作用：把真实 launcher 启动、环境探测、路径扫描移动到测试函数内部。

 如果 launcher 依赖本机环境，改为条件跳过：pytest.importorskip("module_name")

 给测试加标记文件 pytest.ini：unit、integration、e2e、slow、browser、network、mcp、lsp。

 先跑快速单元测试：python -m pytest -m "unit and not slow" --tb=short

 对全量测试生成耗时报告：python -m pytest --durations=50 --tb=short

 把超过 2 秒的测试先标记为 slow。

 把需要真实浏览器的测试标记为 browser。

 把需要网络、GitHub、MCP server、LSP server 的测试分别标记为 network、mcp、lsp。

 建立 smoke 测试入口：python -m pytest tests -m "unit and not slow and not browser and not network and not mcp and not lsp" --maxfail=1

 建立全量测试入口：python -m pytest tests --tb=short --durations=50

 给 CI 或本地脚本增加超时保护：python -m pytest -m "unit and not slow" --timeout=120

 检查是否存在测试 import 即启动服务：python -X importtime -m pytest --collect-only tests/test_zcode_launcher.py

 输出第一份测试基线报告：收集数量、通过数量、失败数量、跳过数量、耗时 Top 50。

建议新增 pytest.ini：

INI
[pytest]
testpaths = tests
addopts = -ra --strict-markers
markers =
    unit: fast isolated unit tests
    integration: integration tests with local components
    e2e: end-to-end tests
    slow: tests slower than 2 seconds
    browser: tests requiring browser or CDP
    network: tests requiring network access
    mcp: tests requiring MCP server
    lsp: tests requiring LSP server

阶段完成标准：

python -m pytest --collect-only -q 不再失败。

python -m pytest -m "unit and not slow" --maxfail=1 可在 60 秒内完成。

全量测试即使超过 180 秒，也能通过 marker 分层运行。

阶段2: 切清核心边界，定义统一契约

目标：不急着删模块，先给 80+ 工具、Agent、执行器、浏览器、MCP 建统一接口。否则后续补测试会被动态行为拖垮。

 新建核心契约目录：core/contracts/

 定义 ToolSpec：工具名、版本、输入 schema、输出 schema、权限、超时、幂等性、依赖能力。

 定义 ToolResult：ok、data、error、logs、artifacts、metrics。

 定义 ToolError：VALIDATION_ERROR、PERMISSION_DENIED、TIMEOUT、DEPENDENCY_MISSING、RUNTIME_ERROR、REMOTE_ERROR。

 定义 AgentTask：目标、上下文、约束、允许工具、禁止工具、预算、审批策略。

 定义 ExecutionContext：workspace、env、权限、trace_id、deadline、cancel_token。

 定义 BrowserSession 抽象接口，隔离 CDP / Playwright 实现。

 定义 MCPClient 抽象接口，隔离真实 MCP server 和 fake MCP。

 定义 LspSession 抽象接口，隔离真实 LSP 和 fake LSP。

 把 tool_registry_mesh 的输出统一成 ToolSpec。

 把所有工具执行结果统一成 ToolResult，禁止直接返回裸 dict 或裸异常。

 给 approval_gate 增加统一入口：任何高危工具必须通过审批。

 给 sandbox 增加不可绕过的执行入口：任何 shell/python/git 文件写操作必须经过执行上下文。

 给 context_manager 增加 token 预算和消息来源标记，防止历史消息被错误注入输入区。

 给 cdp_browser 增加状态机：DISCONNECTED、CONNECTING、READY、DEGRADED、RECOVERING、FAILED。

 给 task_governor 增加全局任务预算：最大并发、最大重试、最大运行时间、最大工具调用数。

核心工具、执行器、浏览器、MCP、LSP 都能被 fake 实现替换。

单元测试不再依赖真实浏览器、真实网络、真实 MCP、真实 LSP。

approval_gate 和 sandbox 成为强制边界，而不是可选工具。
