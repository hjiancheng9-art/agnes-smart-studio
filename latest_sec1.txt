1. ComfyUI workflow JSON 最新格式变化
1.1 两种 JSON 必须彻底区分

现在仍然要严格区分：

A. Workflow Save Format
   用于前端保存、拖入、编辑。
   包含 version / state / nodes / links / groups / config / extra / models 等。

B. API Prompt Format
   用于 POST /prompt 执行。
   根对象是 node_id -> {class_type, inputs, _meta}。

官方文档明确说，浏览器保存格式和 API 格式不同；API 格式应该通过 File → Export Workflow (API) 导出，省掉 UI 布局信息，适合程序提交。
ComfyUI

1.2 最新 Save Format 是 Workflow JSON v1.0

官方 Workflow JSON schema 当前最新是 Version 1.0。顶层关键字段：

JSON
{
  "version": 1,
  "config": {},
  "state": {},
  "groups": [],
  "nodes": [],
  "links": [],
  "reroutes": [],
  "extra": {},
  "models": []
}

必需字段是：

version
state
nodes

官方 schema 里 version 固定为 1，并且 state、nodes 是 required。

1.3 最新 Save Format 的 node outputs 结构

Save Format 里的每个 node 是数组元素，不是 API Format 那种 "3": {...}。

outputs 当前 schema 是：

JSON
{
  "name": "IMAGE",
  "type": "IMAGE",
  "links": [12, 13],
  "slot_index": 0
}

字段含义：

name       输出口显示名
type       类型，可为 string / string[] / number
links      连到哪些 link id，可为 number[] 或 null
slot_index 输出槽位索引，可为 integer 或 string

官方 v1.0 schema 明确规定 outputs 是数组，每项至少需要 name 和 type，可带 links 和 slot_index。
ComfyUI

1.4 links / reroutes / models 是新编译器要认真处理的字段

Save Format 的 links 现在是对象数组，不是老 LiteGraph 风格的纯数组更安全假设：

JSON
{
  "id": 1,
  "origin_id": 4,
  "origin_slot": 0,
  "target_id": 3,
  "target_slot": 1,
  "type": "MODEL"
}

官方 schema 还包括 reroutes、extra.info、extra.linkExtensions、models，其中 models 每项包含 name、url、directory，可选 hash、hash_type。这对你们的“可复现 manifest / 自动下载模型”很有价值。

1.5 API Prompt Format 没有 outputs 字段

执行用 API Format 仍然是：

JSON
{
  "4": {
    "class_type": "CheckpointLoaderSimple",
    "inputs": {
      "ckpt_name": "model.safetensors"
    },
    "_meta": {
      "title": "Load Checkpoint"
    }
  },
  "6": {
    "class_type": "CLIPTextEncode",
    "inputs": {
      "clip": ["4", 1],
      "text": "prompt"
    }
  }
}

API Format 的连接关系写在 inputs 里：

JSON
"model": ["4", 0]

不要给 /prompt 直接提交 Save Format。官方说明 API Format 使用数字 node id 作为 key，保留 widget 值，但不包含位置、颜色、分组等前端布局信息。
ComfyUI

1.6 真正的 breaking point：Custom Node V3 Schema

2025 之后你们要重点关注 Custom Node V3 schema，因为它影响节点能力扫描和 object_info 解析。

V3 变化：

V1:
  INPUT_TYPES
  RETURN_TYPES
  FUNCTION
  CATEGORY
  NODE_CLASS_MAPPINGS

V3:
  define_schema()
  inputs=[...]
  outputs=[...]
  execute()
  comfy_entrypoint()
  ComfyExtension

官方说明 V3 的主要变化是：输入输出用对象定义，不再是字典；执行方法固定为 execute；通过 comfy_entrypoint() 返回 ComfyExtension 暴露节点，而不是旧的 NODE_CLASS_MAPPINGS；未来节点特性只会加到 V3 schema。
ComfyUI

对你们编译器的影响：

不要只静态扫描 NODE_CLASS_MAPPINGS。
必须优先用 /object_info 获取真实节点定义。
自定义节点扫描器要兼容 V1 + V3。
节点合同 NodeContract 应从 object_info 动态生成。
