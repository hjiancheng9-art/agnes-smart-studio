二、逐项回答关键问题

1. 6 层架构和 24 阶段仍是两张皮
问题本质
agent-kernel.js 的 6 层是架构健康模型：
input_layerinput_governancepreprocessingcore_layerprompt_knowledge_routingoutput_layer
autoflow.js 的 24 阶段是业务生产模型：
script → storyboard → keyframes → ... → delivery → archive → retro
它们不是同一类东西。
6 层回答的是：
系统能力是否存在？
24 阶段回答的是：
这个项目生产到哪一步？
所以 checkAgentHealth 60% 实际上只能说明：
系统大概有多少能力模块看起来可用。
它不能说明：
当前 stage 是否能跑当前 input 是否真实当前 artifact 是否完整当前 CRUX task 是否可恢复当前 QA 是否通过
裁判结论
6 层不该再做执行准入门槛。
它应该降级为：
诊断层 / HUD 健康摘要 / 能力覆盖报告
真正的阶段准入应该由：
stage-contract + rule-engine + quality-gate + artifact-registry + crux task state
共同判断。

2. buildControlledLoop 太胖
production-governance.js 28KB，本身不是问题。问题是 buildControlledLoop 承担了太多职责：
同步 workflow run构建 showrunner 决策检查 health 状态检测输入/资产/节点完成生成 controlled loop
这会导致：
它像控制器，又像规则引擎，又像状态机，又像诊断器。
应该拆成
production-governance.js         只保留公共 facadegovernance-policy.js             deliveryGoal / workflowNodesFor / depsshowrunner-decision.js           buildShowrunnerDecisioncontrolled-loop-builder.js       buildControlledLoopworkflow-sync.js                 syncWorkflowRungovernance-audit.js              governanceAuditinput-readiness.js               hasRealInputs / hasVisualInputs / invalidInputs
buildControlledLoop 最终应该变薄：
JavaScriptfunction buildControlledLoop(ctx) {  const sync = workflowSync.sync(ctx);  const readiness = inputReadiness.evaluate(ctx);  const decision = showrunnerDecision.build(ctx, readiness);  const audit = governanceAudit.record(ctx, decision);  return {    sync,    readiness,    decision,    audit  };}

新 9 层不是简单替代老 6 层，而是把老 6 层从“执行判断”降级为“健康观测”。

老 6 层 = 诊断面板新 9 层 = 执行内核
