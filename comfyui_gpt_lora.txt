2. “工作流合成大师”需要懂 LoRA 吗？

需要，但只懂到 inference / workflow compile 层。

它应该懂：

用户需求里有没有 LoRA 意图
例如：
“用某角色 LoRA”、“保持这个人物一致”、“赛博朋克风格 LoRA”、“加细节 LoRA”。

当前 ComfyUI 里有哪些 LoRA
从 /object_info/LoraLoader 或模型目录拿 lora_name 候选。

LoRA 类型和用途
例如：

character LoRA：角色一致性

style LoRA：画风

concept LoRA：特定概念

detail / quality LoRA：细节增强

clothing / pose / object LoRA：局部控制

怎么插入工作流
典型结构是：

CheckpointLoaderSimple
  ├─ MODEL ─→ LoraLoader ─→ KSampler
  └─ CLIP  ─→ LoraLoader ─→ CLIPTextEncode

多个 LoRA 就链式插入：

CheckpointLoaderSimple
  → LoraLoader(character)
  → LoraLoader(style)
  → LoraLoader(detail)
  → KSampler / CLIPTextEncode

怎么校验

LoRA 文件是否存在

LoraLoader 节点是否存在

lora_name 是否在可选项里

base model 是否兼容

是否叠太多

strength 是否越界

是否和用户需求冲突

它不应该懂：

真正训练 LoRA

数据集清洗

caption 生成

optimizer / scheduler 细节

kohya 参数爆炸式配置

GPU 训练进程管理

checkpoint 保存、断点恢复

这些属于 Training Service，不是 Workflow Compiler。

from dataclasses import dataclass
from typing import Literal

LoraKind = Literal["character", "style", "concept", "detail", "clothing", "pose", "object", "unknown"]

@dataclass
class LoraAsset:
    name: str
    kind: LoraKind
    path: str | None = None
    base_family: str | None = None   # sd15 / sdxl / flux / wan / unknown
    trigger_words: list[str] | None = None

@dataclass
class LoraRequest:
    purpose: str                  # "character consistency", "style", etc.
    query: str
    required: bool = False
    preferred_kind: LoraKind = "unknown"

@dataclass
class LoraSelection:
    asset: LoraAsset
    strength_model: float = 0.7
    strength_clip: float = 0.7
    reason: str = ""

@dataclass
class LoraPlan:
    selections: list[LoraSelection]
    missing: list[LoraRequest]
    warnings: list[str]

class FlowCompiler:
    def compile(self, nl: str, llm_chat_func=None) -> dict:
        intent = self.intent_parser.parse(nl)
        workflow = self.base_workflow_builder.build(intent)

        lora_requests = self.lora_planner.extract_requests(nl, intent)
        lora_plan = self.lora_planner.plan(lora_requests)

        workflow = self.lora_compiler.inject(workflow, lora_plan)
        self.validator.validate(workflow)

        return {
            "success": True,
            "workflow": workflow,
            "lora_plan": lora_plan,
        }
最终建议

不要让 LoRA 模块继续伪装成训练系统。

正确定位应该是：

LoRA subsystem = ComfyFlow Compiler 的模型适配层，负责识别需求、选择 LoRA、插入 LoRALoader、校验缺失与冲突，而不是训练 LoRA。

具体处理：

lora_knowledge.py
→ 拆成 catalog.py + policy.py + planner.py
→ 接入 FlowCompiler

lora_manager.py
→ 从 V4 core 移除
→ 改名 training_spec_builder.py，放 extras/labs

lora_trainer.py
→ 改名 lora_training_advisor.py
→ 保留为知识助手，不参与主链路

新增 lora_compiler_hook.py
→ 真正负责 workflow 注入

一句话总结：

LoRA 要留，但不能以“Trainer”身份留；它应该变成 Compiler 的“LoRA 编排插件”。
