V2 QA 层总评
我给当前 QA 层健康评分：
40 / 100
原因：

V2 已经有 QA 意识，但还没有 QA 闸门。
现在有 QA_SUMMARY.md、QC Inspector 技能、artifact evidence、creative-bible 校验、rule-engine quality 规则、vitest 配置和部分测试文件，但它们还没有形成一个能真正阻断低质量产物进入下游的质量保证系统。

更直接地说：
现在的 QA = 文档 + 检查函数 + 证据链 + 少量测试真正需要的 QA = 阶段质量门 + 自动/人工 QC 分层 + 阻断机制 + 审计 + 回归测试 + CRUX 产出验收
还有一个信息矛盾要先指出：你前面说“仅 1 个测试文件”，后面又说“共 27 个测试文件”。我按更保守的判断处理：即使有 27 个测试文件，只要没有 E2E、没有 CI、没有覆盖 24 阶段和 CRUX mock，QA 仍然不算成熟。

一、正方：QA 现状哪里是好的？
1. QA_SUMMARY.md 说明已经有质量基线意识
QA_SUMMARY.md 不是随便写的，它已经定义了：
产品核心基线背景资产冻结原因release readiness 分数必跑 QA 命令工具格式准备状态
这说明 V2 已经开始区分：
product core baselinerelease readinessmanual validationtool format readiness
这是好事。
尤其这一点很重要：
QA sample score must not overwrite product core baseline.
这说明系统已经意识到：不同 QA 场景不能互相污染评分。

2. QC Inspector skill 方向正确
QC Inspector 已经覆盖：
visualIssuestextIssuescontinuityIssuesassetIssuesdeliveryIssuesseveritypass/failrecommendedFixes
这个结构非常接近一个真正的 QC 报告。
它还明确了：
Do not pass QC when evidence is missing.Do not hide failures behind placeholder success.Severity must be explicit.
这些都是核心质量原则。
如果后续把这个 skill 接入规则引擎和 stage gate，它会很有价值。

3. artifact-evidence.js 是 QA 的重要基础
QA 不能只看最终结果，还要看：
这个产物从哪里来？由哪个阶段生成？由哪个执行器生成？有没有 checksum？有没有 evidence？有没有重复 evidence？能不能进入 return package？
artifact-evidence.js 里的这些函数：
checksumFileregisterArtifactEvidencebuildArtifactEvidenceLedgerdedupeEvidencesummarizeEvidenceevidenceToReturnPackageItem
说明 V2 已经具备质量追溯的基础。
这是很重要的，因为没有 evidence 的 QA 很容易变成主观评价。

4. creative-bible 和 rule-engine 已经有 quality 维度
creative-bible 目前有：
角色存在性风格漂移色板引用外貌一致性镜头数匹配
rule-engine 有：
style_consistencyshot_count_consistency
这说明 V2 已经开始把“创作一致性”纳入质量体系。
对视频生成工具来说，这比普通代码测试更关键。

5. 已经有测试基础设施
有 vitest.config.js，也有测试文件，这比完全没有测试好。
至少说明项目已经可以开始走：
单元测试回归测试状态机测试artifact 测试规则测试CRUX mock 测试
不是从零开始。

二、反方：隐藏的 QA 问题

1. QA 没有独立阶段，也没有阶段质量门
这是最大问题之一。
24 阶段里有：
21-review
但 review 不等于系统性 QA。
真正的 QA 应该不是只在第 21 阶段发生，而应该贯穿：
每个阶段完成前每个 artifact commit 前每个 CRUX result 接收后每个 delivery package 生成前每次 archive 前
现在的问题是：
阶段完成 → 直接进入下一阶段质量检查 → 可能只是 warning产物不合格 → 仍可能被下游消费
这会导致典型灾难：
3-keyframes 风格已经漂移但只是 warn9-animation 继续使用这些关键帧15-edit 再继续剪辑22-delivery 最后才发现全片风格不一致
这时返工成本极高。
反方结论：
QA 不应该是一个末尾 review 阶段，而应该是每个阶段的 quality gate。

2. 5 维校验和 quality 规则只有 warn，等于“看到了但不管”
creative-bible 和 rule-engine 的质量规则目前偏 advisory/warn。
这适合创作探索，但不适合交付。
例如：
角色缺失风格漂移镜头数不匹配色板没引用外貌不一致
这些在不同阶段应该有不同严重度。
比如：
问题草稿阶段keyframes 阶段delivery 阶段轻微风格漂移warnerrorblock角色缺失errorblockblock镜头数不匹配warnerrorblockevidence 缺失blockblockblockplaceholder outputblockblockblock
现在如果全是 warn，就会变成：
系统知道有问题，但仍然允许继续生产。
这不是 QA，这是提示。

3. 无 QA 仪表盘，质量状态不可见
现在没有统一视图可以回答：
这个项目质量分是多少？哪些阶段有质量问题？哪些问题是 blocking？哪些只是 warning？哪个 artifact 没有 evidence？哪个阶段最常失败？哪些项目风格漂移最多？CRUX 哪个平台输出质量最不稳定？
没有 QA dashboard，就会出现：
问题分散在 creative-bible、rule-engine、artifact-evidence、QC report、QA_SUMMARY 里用户和开发者都看不到整体质量
V2 是制作工作台，HUD 应该能显示：
Project QA ScoreStage QA StatusArtifact Evidence StatusCRUX Output QualityDelivery ReadinessBlocking IssuesRecommended Fixes

4. 测试覆盖率仍然不足
无论是 1 个测试文件还是 27 个测试文件，只要没有覆盖这些核心路径，就仍然危险：
project-store 原子写stage-contract 状态转换autoflow 24 阶段推进artifact evidence 注册rule-engine 阻断CRUX task mockdelivery package gaterollback / rerun / invalidation
目前最大风险是：
测试存在，但没有覆盖业务关键路径。
这会导致：
改了 project-store，不知道 archive 是否坏了改了 rule-engine，不知道 autoflow 是否还能跑改了 crux-bridge，不知道 stage completion 是否会假完成

5. 无 E2E 测试，24 阶段没有自动验收
V2 的真正价值是 24 阶段流水线。
如果没有 E2E，就无法证明：
用户创建项目选择 deliveryGoalCRUX mock 生成 scriptCRUX mock 生成 storyboardlocal 生成 artifactrule-engine 检查QC report 生成delivery package 完成archive 成功
这条链能稳定跑通。
CRUX 外部平台可以在 E2E 里 mock，不需要真的打开 ChatGPT / Kling / Runway。
应该有：
FakeCruxExecutorFakeArtifactGeneratorFakeProviderResultFakeEvidence
否则每次修复都只能靠手测。

6. 无回归测试，修 P0 会引入新 P0
你前面已经有很多 P0：
project-store 安全crux-bridge 任务落盘stage-contractrule-engineknowledge-runtimebusiness DAG
这些一旦开始修，如果没有回归测试，很容易出现：
修了原子写，破坏聊天记录修了 stage 状态，autoflow 不再推进修了 evidence，delivery 找不到 artifact修了 CRUX task store，HUD 看不到任务进度
所以 QA 层必须优先补回归保护。

7. CRUX 产出的质量没有系统验收
这是 QA 层最关键的业务问题之一。
CRUX 负责：
scriptstoryboardkeyframescinematicvoiceoverreview
这些都是创意核心。如果 CRUX 输出质量不稳定，整个 V2 会不稳定。
现在必须回答：
ChatGPT 返回的 script 是否符合项目 brief？storyboard 是否覆盖所有剧情节点？keyframe prompts 是否能生成一致画面？voiceover 是否和 script 同语言？review 是否真的按 matching mode 检查？CRUX 是否把建议当成终稿？CRUX 是否返回结构化 schema？
如果没有 CRUX output QA，V2 就会变成：
CRUX 说完成了 → V2 相信了 → 下游继续跑
这很危险。
CRUX 输出必须经过：
schema validationrule validationcreative-bible validationartifact evidence validationQC Inspector validationhuman approval when needed

8. 自动 QA 和人工 QA 边界不清
QC Inspector 文档说：
manual web evidence importmanual validationQC Panel
但系统现在似乎没有明确区分：
哪些检查自动完成？哪些检查需要用户确认？哪些检查 CRUX 自动评估？哪些必须人工审核？哪些可以 override？
这会造成用户困惑。
应该明确：
QA 类型自动 / 人工schema 是否有效自动artifact 是否存在自动checksum 是否存在自动placeholder 是否存在自动evidence 是否缺失自动镜头数是否匹配自动色板是否引用自动视觉质量是否好看CRUX + 人工风格是否符合审美CRUX + 人工最终是否可交付自动 gate + 人工确认

9. QA 数据没有闭环
creative-bible 发现风格漂移，rule-engine 也发现 style_consistency，但它们没有统一聚合。
结果：
creative-bible 有一份 warningrule-engine 有一份 warningQC Inspector 有一份 reportartifact-evidence 有一份 ledgerHUD 没有统一显示retro 无法总结knowledge layer 也无法吸收经验
真正的 QA 应该形成闭环：
QA issue  → stage quality status  → fix recommendation  → user approve/reject  → rerun or override  → archive evidence  → retro  → knowledge candidate

10. stage-contract 缺少质量状态
当前 stage-contract 有 10 种状态，但没有：
quality_pendingquality_passedquality_failedquality_holdneeds_human_qc
这会导致质量状态和执行状态混在一起。
例如：
stage.status = completed
但这不说明：
质量是否通过？是否有 warning？是否需要人工确认？是否可以进入 delivery？
应该拆成两个维度：
executionStatus: pending / running / completed / failedqualityStatus: not_checked / checking / passed / warning / failed / blocked / needs_human
否则“完成”和“合格”会被混淆。

五、CRUX 总控如何在 QA 层发挥最大价值
CRUX 最适合做三类 QA：
1. 智能语义质量判断2. 视觉/多模态检查3. 质量修复建议与返工规划

1. CRUX 负责验证自己的产出
CRUX 输出不能自己说自己合格。
但 CRUX 可以参与第一轮质量检查。
每个 CRUX 阶段完成后，V2 应该立即发起一个 QA task：
CRUX Generate Task      ↓V2 schema validation      ↓V2 evidence validation      ↓CRUX QC Inspector / secondary model review      ↓QA gate decision
例如 script 阶段：
检查：- 是否符合 brief- 是否有完整起承转合- 是否语言匹配- 是否包含角色/场景/动作- 是否可拆成 storyboard- 是否输出 script.v1.json
storyboard 阶段：
检查：- 是否覆盖 script beats- shot count 是否匹配- 每个 shot 是否有画面、动作、镜头、时长- 是否符合 deliveryGoal
keyframes 阶段：
检查：- 每个 shot 是否有 keyframe prompt- 是否引用 creative-bible- 是否有角色/场景 continuity- 是否有 negative constraints
voiceover 阶段：
检查：- 语言是否和 script 一致- 文案是否匹配镜头时长- 语气是否匹配项目风格
review 阶段：
检查：- 是否按 matching mode 审阅- 是否输出 blocking/non-blocking issues- 是否给出修复建议

2. CRUX 做跨模型交叉审查
不要让同一个 CRUX 生成者直接给自己打满分。
可以设计：
Generator Model ≠ Reviewer Model
例如：
ChatGPT 生成 scriptGemini / Kimi / DeepSeek 审查 scriptGemini 做视觉 QCChatGPT 做结构化修复建议
CRUX 总控负责路由：
text quality → ChatGPT / Kimi / DeepSeekvisual quality → Geminidelivery risk → DeepSeeklong report synthesis → Kimilocal checklist → LM Studio Gemma
这正好对应 QC Inspector 的 Preferred Providers。

3. CRUX 做视觉 QC
对于关键帧和视频片段，CRUX 可以调用 Gemini 或可视模型检查：
画面是否符合 prompt？角色是否一致？风格是否漂移？色板是否匹配？是否有明显错误？镜头是否表达 storyboard？字幕是否遮挡主体？
输出结构化 visual QC：
JSON{  "visualMatchScore": 0.82,  "styleConsistency": "warning",  "characterConsistency": "pass",  "blockingIssues": [],  "recommendedFixes": [    "Regenerate shot 04 with stronger palette lock."  ]}

4. CRUX 做 QA 修复规划
QA 不应该只说“错了”，还要告诉用户怎么修。
CRUX 可以生成：
最小返工路径影响下游阶段推荐重跑哪些 shot哪些 artifact 可保留哪些必须废弃
例如：
JSON{  "fixPlan": {    "minimalRerun": true,    "rerunStages": ["3-keyframes"],    "affectedShots": [4, 7],    "keepArtifacts": ["shot_01", "shot_02", "shot_03"],    "invalidateArtifacts": ["shot_04_keyframe", "shot_07_keyframe"],    "reason": "Only two shots violate palette lock."  }}
这能直接连接业务层的回溯机制。

5. CRUX 做 QA 到知识层的沉淀
如果同类问题反复出现，比如：
神性建筑项目里人物经常抢主体赛博雨夜项目容易变成廉价霓虹关键帧常忘记引用色板voiceover 语言经常和 script 不一致
CRUX 可以在 retro 阶段提炼为：
knowledge candidaterule candidatecreative-bible improvementprovider contract update
但不要直接写入 native knowledge，仍然走知识层的 curation 流程。

V2 的 QA 层现在最大问题不是“没有检查”，而是“检查没有变成质量门”。
下一步要把 QC Inspector、creative-bible、rule-engine、artifact-evidence 和测试系统统一到一个 QA Orchestrator 里，让每个阶段都必须经过可审计、可阻断、可修复的质量门。
