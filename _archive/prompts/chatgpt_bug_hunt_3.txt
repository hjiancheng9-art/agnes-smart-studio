无法做局部事务。

不需要断线重连；

无事件时间线。

浪费请求。

8. plugin 系统：plugins/ 目录文件未知 + 缺乏中间件机制
正方
极简 router.js 只有 1.4KB，优点是：


简单；


可控；


无 Express 依赖；


插件挂载容易理解；

反方
你问“plugins/ 目录有哪些文件”，这部分你没有提供实际列表，所以我不能假设具体文件名。
但从架构看，插件系统至少会遇到这些限制：


无中间件，插件无法统一处理鉴权、日志、错误。
每个插件都要自己写：
parse bodyhandle errorreturn JSONcheck methodvalidate inputlog request
结果就是每个插件风格不同。


无法做统一 body parser。
原生 http 如果没中间件，JSON body、文件上传、表单上传会很快变乱。


路由冲突难处理。
两个插件都注册：
/api/generate
谁赢？是否报警？是否覆盖？


fallback 静态文件可能吞 API。
如果路由匹配顺序不严谨，插件路由 miss 后可能被静态 fallback 处理，返回 HTML 而不是 JSON。


无权限边界。
插件能否访问 project-store？能否写 artifact？能否调用 CRUX？需要能力限制。


无依赖声明。
插件 A 依赖插件 B，加载顺序怎么保证？


无生命周期。
插件热加载时无法可靠 init / teardown / healthcheck。


裁判结论
轻量 router 可以保留，但要升级成“插件路由注册器”，不需要上 Express。
建议 router.js 支持：
use(middleware)registerRoute(method, path, handler, meta)registerPluginRoutes(pluginId, routes)detectConflict()json()errorBoundary()notFound()
插件 manifest 至少要有：
JavaScript{  "pluginId": "xxx",  "version": "1.0.0",  "routes": [],  "capabilities": [],  "dependsOn": [],  "permissions": ["project:read", "artifact:write"],  "init": "function",  "teardown": "function",  "healthcheck": "function"}
