V2 管线层总评
我给当前管线层健康评分：
41 / 100
核心判断：

V2 现在有很多“流”，但还没有统一的“管线协议”。
输入流、执行流、制品流、证据流、决策流、失败流、CRUX 流都存在，但各自使用自己的字段、阶段名、状态语义和归档方式。最大风险是：某个阶段看起来成功了，但它的输入、输出、证据、决策、QA 没有真正流到下一层。

一句话：

V2 的管线问题不是“没有流程”，而是“流程之间没有标准接口”。


一、当前管线的最大结构性问题
现在至少存在三套管线语言。
第一套是 autoflow 的 phase/node 语言：
intakelocal_videovisionreconstructionassetsasset_imagesstoryboardstoryboard_imagesvideo_generationaudio_promptsremotionfinal_qcexport_package
第二套是 24 阶段语言：
1-script2-storyboard3-keyframes4-visual-dev5-character...22-delivery23-archive24-retro
第三套是 artifact/archive 的 group/kind 语言：
inputassetstoryboardvideofinal_videoknowledgepromptrequestresponseevidenceqcdiagnostic
这些语言没有统一映射，就会导致：
input-route-pipeline 认为下游是 reconstructionautoflow 认为当前阶段是 1-scriptartifact-archive 归档为 storyboardrule-engine 阻断的是 2-storyboardHUD 显示的是 Asset & StoryboardCRUX task 记录的是 stageId = storyboard
用户和系统都很难判断：这些到底是不是同一个节点。
这是管线层的 P0 问题。

二、每个管线接口的具体问题
1. autoflow 24 阶段之间的接口问题
当前风险
你问得很关键：

阶段 A 输出 data.text 和 data.characters，阶段 B 是否消费了这些字段？

现在最大问题是：阶段输出没有统一 output contract。
如果 reconstruction 输出：
JavaScript{  data: {    text: "...",    characters: [...]  }}
但 storyboard 期望的是：
JavaScript{  logline,  beats,  characterList}
那么阶段 B 可能根本没有消费阶段 A 的关键字段。
典型断层：
上游阶段可能输出下游真正需要风险reconstructiontext / characterslogline / beats / character list字段名不匹配storyboardshotsshotId / duration / assetRefs分镜不能绑定资产asset_imagesimage pathsassetId / realPath / evidenceId图像存在但不可追溯storyboard_imagesstoryboard image filesshotId → imageArtifactId镜头和图断链video_generationvideo outputshotId / duration / providerEvidence视频无法对应分镜audio_prompts音效建议cueId / shotId / timing声音无法同步画面final_qcreportpass/fail / blockersQC 无阻断力export_packagefilesmanifest / checksum / evidence交付不可验证
具体 Bug 形态
阶段 A complete阶段 B running但 B 没有读取 A 的关键产物最终流程能跑完，但产物之间没有 lineage
这就是“假流水线”。
建议
每个阶段必须定义：
JavaScript{  stageId: "2-storyboard",  requiredInputs: ["script.v1"],  outputs: ["storyboard.v1"],  consumesFields: ["script.beats", "script.characters"],  producesFields: ["storyboard.shots", "storyboard.assetRefs"],  blockingIfMissing: true}
不要靠隐式字段传递。

2. 输入管线和执行管线的接口问题
当前风险
input-route-pipeline.js 里定义：
JavaScriptVIDEO_INPUT_TYPES = ['video_file', 'video_url']ASSET_INPUT_TYPES = ['asset_image', 'asset_file', 'asset_url']TEXT_INPUT_TYPES = ['customer_text', 'note', 'text']DOCUMENT_INPUT_TYPES = ['document']
但执行管线的阶段节点是：
intake → vision → reconstruction → storyboard ...
关键问题是：
input-route-pipeline 的输出格式是什么？autoflow 的 stage input 格式是什么？
如果输入管线只是统计：
JavaScript{  videoCount,  assetCount,  textCount,  documentCount}
这只能说明“有什么”，不能说明“怎么被阶段使用”。
例如用户上传视频：
video_file
它应该流向：
input registry→ preprocess video→ vision analysis→ reconstruction→ script/storyboard
但如果中间没有标准对象，就会断：
用户上传视频preview 可见hasRealInputs = true但 vision 阶段没有拿到 normalized video pathreconstruction 阶段也没有拿到 vision output
建议
输入管线输出不能只是 count，必须输出 InputRoutePlan：
JavaScript{  routeId: "route_001",  projectId: "project_001",  inputs: [    {      inputId: "input_video_001",      type: "video_file",      modality: "video",      normalizedPath: "...",      preprocessRequired: true,      routedToStages: ["vision", "reconstruction", "storyboard"],      requiredByStages: ["vision"],      status: "ready"    }  ],  blockers: [],  recommendedEntryStage: "vision"}
然后 workflow-engine 或 autoflow 只能读取这个 route plan，而不是自己猜输入。

3. 制品归档管线接口问题
当前风险
artifact-archive.js 使用：
JavaScriptcategoryForAsset(asset)categoryForInput(input)categoryForRecord(record)copyArtifact(...)
这说明归档主要靠：
record.groupasset.kindinput.type
来决定目录。
这很灵活，但风险是：阶段产物没有统一 Artifact Contract。
例如阶段执行成功后得到：
JavaScriptoutcome.result
但 archive 可能需要：
JavaScript{  group,  kind,  sourcePath,  fileName,  meta}
如果没有转换层，就会出现：
阶段 outcome.result 有内容但没有 sourcePatharchive 无法 copy阶段仍然标记 successartifact 丢失
或者：
CRUX 返回 text resultarchive 只会 copy file文本结果没有落盘证据链也没有 artifactId
制品丢失的典型场景
CRUX stage completedresponse text 存在内存stage.status = completed但 artifact-archive 没有 addArchiveFile项目重启后 response 丢失
建议
新增统一 ArtifactEnvelope。
JavaScript{  artifactId: "artifact_001",  projectId: "project_001",  runId: "run_001",  stageId: "3-keyframes",  kind: "keyframe_plan",  group: "prompt",  mediaType: "application/json",  contentRef: {    type: "inline_json",    data: {}  },  fileRef: {    sourcePath: null,    targetPath: null  },  derivedFromInputs: ["input_001"],  evidenceIds: [],  status: "staged"}
所有阶段产物先变成 ArtifactEnvelope，再交给 artifact-registry / artifact-archive。

4. 证据链管线接口问题
当前风险
artifact-evidence.js 有：
registerArtifactEvidencebuildArtifactEvidenceLedgerdedupeEvidencesummarizeEvidenceevidenceToReturnPackageItem
这是好基础。
但关键问题是：
哪些阶段必须注册 evidence？CRUX 6 个阶段是否都有 evidence？evidence 和 artifact 是否强绑定？
CRUX 阶段尤其危险：
1-script2-storyboard3-keyframes11-cinematic12-voiceover21-review
这些大多来自外部 AI 平台。必须记录：
promptSnapshotIdprovider/platformtaskIdresponse hashbrowser evidenceinput artifactsoutput artifactsmodel/tool identitytimestamp
如果只记录最终文本，没有 evidence，则后续无法回答：
这个分镜是谁生成的？用了什么 prompt？从哪个平台回来？有没有被用户修改？它是不是 mock？
建议
所有 artifact commit 前必须强制 evidence：
JavaScript{  evidenceId: "ev_001",  artifactId: "artifact_001",  taskId: "task_001",  stageId: "2-storyboard",  executor: "crux",  provider: "chatgpt",  promptSnapshotId: "ps_001",  responseHash: "sha256...",  sourceInputIds: ["input_001"],  createdAt: "..."}
规则：
没有 evidence 的 artifact 只能是 draft，不能进入 approved / delivery。

5. 决策管线接口问题
当前风险
director-runtime.direct() 输出：
decisionIdfinalconflicts
问题是：输出之后谁消费？
如果它只是记录：
JavaScript{  decisionId,  final,  conflicts}
但没有进入：
creative-biblestage task payloadprompt-managerworkflow-engineQA gateartifact evidence
那导演决策只是“旁白”，不是管线输入。
典型问题：
director-runtime 决定走写实风格prompt-manager 没拿到 directionSnapshotIdCRUX prompt 仍然走默认模板最终产物没有执行导演决策
建议
导演决策必须转成 DecisionEnvelope，并写入统一流：
JavaScript{  decisionId: "dir_001",  runId: "run_001",  stageId: "global",  type: "creative_direction",  final: {    style: "IMAX cinematic realism",    palette: ["cyan", "gold", "violet"],    negativeConstraints: ["avoid cartoon"]  },  conflicts: [],  appliesToStages: ["3-keyframes", "9-animation", "16-color"],  writeTargets: {    creativeBible: true,    promptContext: true,    qaCriteria: true  },  status: "committed"}
然后必须发生三件事：
1. 更新 creative-bible 或 direction snapshot2. prompt-manager 引用 decisionId3. artifact/evidence 记录该产物使用了哪个 decisionId

6. 例外 / 失败管线接口问题
当前风险
现在失败可能来自：
rule-engine blockquality-gate failCRUX timeoutinput preprocess failartifact copy failevidence missingstage-contract illegal transitionplugin error
如果这些失败都只是 throw error 或写 message，就会混进正常流程。
典型问题：
rule-engine 阻断autoflow 捕获 errorstage.status = warnHUD 只看到 failed用户不知道是哪条规则挡住了CRUX 也不知道能不能介入
或者：
quality-gate 发现质量问题但没有生成 fix task流程停住用户不知道下一步该做什么
建议
失败流必须有独立 PipelineIssue。
JavaScript{  issueId: "pipe_issue_001",  projectId: "project_001",  runId: "run_001",  stageId: "3-keyframes",  source: "quality-gate",  type: "quality_failed",  severity: "blocker",  blocking: true,  message: "Keyframe shot count does not match storyboard.",  affectedArtifacts: ["artifact_001"],  suggestedActions: [    {      action: "rerun_stage",      stageId: "3-keyframes"    },    {      action: "edit_storyboard",      stageId: "2-storyboard"    }  ],  status: "open"}
失败不应该只是错误信息，应该变成可操作对象。

7. CRUX 总控跨管线协调问题
当前风险
CRUX 要想介入，必须知道：
哪条管线卡住？卡在哪里？是输入缺失、规则阻断、质量失败、证据缺失，还是外部平台超时？需要 CRUX 做什么？
如果没有统一管线看板，CRUX 只能从局部信息猜。
这会导致：
CRUX 以为是 prompt 问题实际是 artifact 没归档CRUX 重跑外部平台但真正问题没解决
建议
CRUX 需要一个统一 Pipeline Cockpit API。
GET /api/pipeline/runs/:runId/statusGET /api/pipeline/runs/:runId/blockersGET /api/pipeline/runs/:runId/lineagePOST /api/pipeline/issues/:issueId/intervenePOST /api/pipeline/runs/:runId/resume

四、改进建议
1. 建立统一 Pipeline Contract
新增：
pipeline/  pipeline-contract.js  pipeline-stage-map.js  pipeline-envelope.js  pipeline-lineage.js  pipeline-issue-store.js  pipeline-event-bus.js  pipeline-dashboard-api.js
核心任务：
统一 stageId统一 input/output envelope统一 artifact/evidence/decision/prompt/QA 引用方式统一失败对象统一管线事件

2. 统一阶段命名映射
建立 pipeline-stage-map.js：
JavaScript{  "1-script": {    canonicalStageId: "1-script",    legacyNodes: ["reconstruction"],    phase: "intake",    archiveGroups: ["prompt", "knowledge"],    outputKinds: ["script.v1"],    userLabel: "剧本"  },  "2-storyboard": {    canonicalStageId: "2-storyboard",    legacyNodes: ["storyboard"],    phase: "production",    archiveGroups: ["storyboard"],    outputKinds: ["storyboard.v1"]  },  "3-keyframes": {    canonicalStageId: "3-keyframes",    legacyNodes: ["storyboard_images"],    phase: "production",    archiveGroups: ["storyboard"],    outputKinds: ["keyframe_image", "keyframe_plan.v1"]  },  "9-animation": {    canonicalStageId: "9-animation",    legacyNodes: ["video_generation"],    phase: "delivery",    archiveGroups: ["video"],    outputKinds: ["video_clip"]  }}
所有旧节点必须映射到 canonical stageId。

3. 定义统一 Pipeline Envelope
任何管线对象都用统一 envelope 包装。
JavaScript{  "envelopeId": "env_001",  "kind": "artifact",  "projectId": "project_001",  "runId": "run_001",  "stageId": "3-keyframes",  "taskId": "task_001",  "status": "staged",  "payload": {},  "refs": {    "inputIds": ["input_001"],    "artifactIds": [],    "evidenceIds": ["ev_001"],    "decisionIds": ["dir_001"],    "promptSnapshotIds": ["ps_001"],    "qaReportIds": []  },  "createdAt": "..."}
kind 可以是：
inputstage_outputartifactevidencedecisionpromptqa_reportissueevent
这样所有流都能挂在同一个 run lineage 上。

4. 每个阶段定义 I/O Contract
JavaScript{  "stageId": "3-keyframes",  "requires": {    "inputs": [],    "artifacts": ["storyboard.v1"],    "decisions": ["visual_direction"],    "prompts": ["stage.3-keyframes"],    "knowledge": ["visual_rules"]  },  "produces": {    "artifacts": ["keyframe_plan.v1", "keyframe_image"],    "evidence": true,    "qaReport": true  },  "gates": {    "beforeStart": ["storyboard.exists", "direction.exists"],    "beforeComplete": ["artifact.exists", "evidence.exists", "qa.pass"]  }}
workflow-engine 每跑一个阶段前后都检查这个 contract。

5. 建立 Lineage Graph
每个 artifact 必须能回答：
来自哪个 input？用了哪个 prompt？由哪个 CRUX task 生成？用了哪个 director decision？通过了哪个 QA report？被哪个 downstream stage 消费？
Lineage 示例：
JavaScript{  "artifactId": "artifact_keyframe_001",  "lineage": {    "inputs": ["input_style_ref_001"],    "upstreamArtifacts": ["artifact_storyboard_001"],    "decisions": ["dir_visual_001"],    "promptSnapshots": ["ps_keyframe_001"],    "cruxTasks": ["task_003"],    "evidence": ["ev_003"],    "qaReports": ["qar_003"],    "downstreamStages": ["9-animation"]  }}
这会让 CRUX、HUD、QA、archive 都能追踪。

6. 阶段成功必须绑定 artifact + evidence + QA
禁止：
stage complete but no artifactstage complete but no evidencestage complete but QA failed
完成条件：
executionStatus = completedartifactCommitStatus = committedevidenceStatus = completequalityStatus = passed 或 approved_with_warnings
否则只能是：

7. 建立独立失败管线

pipeline-issue-store.jspipeline-recovery-router.js

JavaScript{  "issueId": "issue_001",  "type": "missing_evidence",  "source": "artifact-registry",  "stageId": "3-keyframes",  "blocking": true,  "status": "open",  "suggestedActions": [    "retry_crux_task",    "register_evidence",    "manual_upload"  ]}

六、最终落地顺序
第一阶段：统一命名和 stage map
1. 建 pipeline-stage-map.js2. 旧 node 全部映射 canonical stageId3. archive / QA / rules / CRUX task 全部使用 canonical stageId
第二阶段：建立阶段 I/O contract
1. 每个阶段定义 requires / produces2. workflow-engine 执行前后校验3. 缺输入或缺产物直接 blocked
第三阶段：统一 artifact + evidence 提交流
1. 阶段 outcome.result 统一转换成 ArtifactEnvelope2. artifact-registry commit3. registerArtifactEvidence4. 没有 evidence 不允许 complete
第四阶段：建立 lineage graph
1. input → decision → prompt → task → artifact → QA → delivery2. HUD 和 CRUX 都可查询
第五阶段：建立失败管线和 CRUX 看板
1. pipeline-issue-store2. pipeline-dashboard-api3. CRUX intervention API4. SSE events

最终一句话

V2 管线层现在最大的问题不是“流程不够多”，而是“流与流之间没有统一接口”。

最重要的 5 个修复：
1. 统一 stageId：把 phase nodes、24 阶段、archive group 映射到一个 canonical stage map2. 每个阶段定义 input/output contract3. 阶段完成必须绑定 artifact + evidence + QA4. 建立 lineage graph，追踪 input → prompt → CRUX → artifact → QA → delivery5. 建立 pipeline issue + CRUX cockpit，让阻塞可见、可修、可自动介入
最终权责建议：
V2 管 pipeline truth。CRUX 管 pipeline intervention。V2 负责记录、校验、归档和状态。CRUX 负责发现阻塞、补证据、重试外部平台、提出修复方案。
