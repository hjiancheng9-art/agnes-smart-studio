下面按“能不能上线、会不会丢状态、会不会被外部误连、会不会难维护”的标准快速审。
V2 剩余 8 层快速健康表
层面健康评分优先级核心问题一句话改进建议部署/运维层25/100P1裸跑 node server.js，无 Docker、无进程管理、无优雅关闭、无日志轮转；崩溃后状态和任务恢复全靠手动。至少加 pm2/启动脚本/优雅关闭/日志轮转；后续再补 Docker。ComfyUI 集成层38/100P2/P1旧架构里 ComfyUI 是子执行器，新架构 CRUX 已经接管 Kling/Jimeng/Runway/Luma，ComfyUI 角色变模糊。把 ComfyUI 降级为可选本地 provider，通过统一 executor-router 接入。安全层20/100P0本地 API、CRUX bridge、浏览器扩展连接都无鉴权；如果被同机恶意网页或扩展调用，可能触发生成、读写项目、导入外部内容。给所有写操作和 CRUX bridge 加本地 token、origin 校验、能力权限。前端 HUD 层50/100P1HUD 现在像旧控制台，而 CRUX 成为总控后，HUD 没有明确转型为 CRUX 可视化驾驶舱。HUD 应成为 CRUX/V2 共同的“管线看板 + 人工确认 + 规则/QA/证据面板”。数据模型层30/100P0/P1defaultProject 在 project-store.js 内部定义，无集中 schema、无迁移、无兼容策略；新增字段容易破坏旧项目。建立 schemas/ + migrations/，每个项目带 schemaVersion。日志/监控层28/100P1日志零散，非结构化；CRUX、workflow、rule、QA、artifact、API 错误无法串联。加统一 JSON logger、runId/taskId/stageId 贯穿日志，并加错误事件表。国际化层45/100P2UI 有 locales，但 Prompt、creative-bible、rule-engine 中仍混硬编码中文；CRUX 操作外部英文界面时语言策略不统一。区分 UI locale、project language、provider prompt language，Prompt 模板外部化并带语言字段。API 设计层32/100P168 个 API 分散，无 /api/v1、无统一错误码、无 schema 校验、无文档；CRUX 接入后风险放大。建立 API registry、统一响应格式、错误码、request/response schema 和版本前缀。

分层结论
1. 部署/运维层 — 25/100，P1
当前最大问题是不可恢复。
裸跑 node server.js 对原型可以，但对 CRUX 总控 + 外部平台长任务不够。
必须先补：
process managergraceful shutdownstartup recoverylog rotationcrash restart
建议顺序：
1. 加 npm scripts: start/dev2. 加 graceful shutdown，保存 inFlight task3. 加 pm2 ecosystem.config.js4. 加 logs/ 分目录5. 后续再 Docker 化

2. ComfyUI 集成层 — 38/100，P2/P1
ComfyUI 不是一定要删，但它的定位必须改。
旧定位：
ComfyUI = 核心本地子执行器
新定位应该是：
ComfyUI = 可选 local image/video workflow provider
统一纳入：
executor-routerprovider-registrycapability-registry
不要让它和 CRUX 外部平台并列抢控制权。
优先级：
如果当前不依赖 ComfyUI 交付，是 P2；如果流程仍会被 ComfyUI preflight 阻断，则升为 P1。

3. 安全层 — 20/100，P0
这是最需要补的一层。
即使是本地工具，也不能默认所有 localhost 请求都可信。浏览器页面、扩展、恶意网页、其他本地进程都有可能请求：
/api/crux/*/api/project/*/api/autoflow/*/api/artifacts/*
最低修复：
CRUX_SHARED_TOKENV2_LOCAL_TOKENOrigin / Referer 校验只允许 127.0.0.1写操作必须带 token危险操作二次确认
统一错误：
JSON{  "ok": false,  "error": {    "code": "auth_required",    "message": "Missing local auth token."  }}

4. 前端 HUD 层 — 50/100，P1
HUD 不应该被 CRUX 替代，而应该升级成 CRUX 的可视化驾驶舱。
HUD 的新职责：
管线总览CRUX 状态阶段状态规则阻断QA issue证据链人工确认输入/制品/Prompt 快照查看
CRUX 是总控执行者，HUD 是人类可见控制台。
推荐关系：
V2 Core = 状态真相CRUX = 执行总控HUD = 可视化驾驶舱 + 人工接管界面

5. 数据模型层 — 30/100，P0/P1
这层和 project-store.js 绑定太深。现在最危险的是：
新增字段无 schema旧项目无法迁移CRUX task / prompt snapshot / rule snapshot / QA report 字段可能漂移
最低修复：
schemas/project.schema.jsschemas/run.schema.jsschemas/task.schema.jsschemas/artifact.schema.jsschemas/input.schema.jsmigrations/
每个项目必须有：
JSON{  "schemaVersion": "2.0.0",  "createdWith": "v2",  "migrationsApplied": []}
如果当前已经有很多旧项目，数据模型层是 P0；如果还在原型期，是 P1。

6. 日志/监控层 — 28/100，P1
现在最大问题是出错后难复盘。
至少日志要统一为：
JSON{  "ts": "2026-07-06T00:00:00+09:00",  "level": "info",  "module": "workflow-engine",  "projectId": "p1",  "runId": "r1",  "stageId": "3-keyframes",  "taskId": "t1",  "event": "stage.started",  "message": "Stage started"}
必须贯穿：
API requestworkflow stageCRUX taskartifact commitrule blockQA failprompt sendevidence register
没有结构化日志，CRUX 介入会很难定位问题。

7. 国际化层 — 45/100，P2
现在问题不是 locales 少，而是语言边界混乱。
应该拆成三种语言：
UI Language：HUD 显示语言Project Language：剧本/字幕/旁白语言Provider Prompt Language：发给 ChatGPT/Gemini/Kling 的提示词语言
例如：
JSON{  "uiLocale": "zh-CN",  "projectLanguage": "zh-CN",  "providerPromptLanguage": "en-US",  "subtitleLanguages": ["zh-CN"]}
Prompt、规则、creative-bible 里的硬编码中文应该逐步模板化。

8. API 设计层 — 32/100，P1
68 个端点已经需要治理了。
最低要统一：
/api/v1/*统一响应格式统一错误码请求 schema 校验响应 schema 校验API registryCRUX 专用 namespace
推荐结构：
/api/v1/projects/*/api/v1/runs/*/api/v1/stages/*/api/v1/crux/*/api/v1/artifacts/*/api/v1/inputs/*/api/v1/pipeline/*/api/v1/qa/*/api/v1/rules/*
统一响应：
JSON{  "ok": true,  "data": {},  "meta": {    "requestId": "req_001"  }}
统一错误：
JSON{  "ok": false,  "error": {    "code": "stage_blocked",    "message": "Stage is blocked by QA gate.",    "details": {}  }}

“要不要修”总排序
P0：必须修，不修会有真实风险
1. 安全层：CRUX bridge / 本地 API / 浏览器扩展无鉴权2. 数据模型层：无 schema / 无 migration / 旧项目兼容风险
尤其安全层必须马上补。
V2 现在已经不是单纯网页工具，而是可以通过 CRUX 操控外部 AI 平台、写项目、注册制品，不能裸开放写接口。

P1：尽快修，不修会影响稳定性和长期开发
1. 部署/运维层：裸跑、无进程管理、无优雅关闭2. 前端 HUD 层：需要升级为 CRUX 可视化驾驶舱3. 日志/监控层：无统一结构化日志4. API 设计层：无版本、无错误码、无 schema5. ComfyUI 集成层：如果仍会阻断新流程，则 P1

P2：可以排期修，但不能永久不管
1. 国际化层：UI / 项目 / Provider Prompt 语言策略2. ComfyUI 集成层：如果只是可选 provider，则 P2

最终一句话
这 8 层里最危险的是：
安全层数据模型层日志/API/运维
优先级建议：
1. 先给本地 API 和 CRUX bridge 加 token 鉴权2. 再建立 schemaVersion + migrations3. 然后加统一 logger 和 API response/error 格式4. 接着用 pm2/优雅关闭解决裸跑问题5. 最后把 HUD 升级成 CRUX 的管线驾驶舱
整体判断：

V2 的创作系统已经很复杂，但基础设施还停留在本地原型阶段。下一步不要再加生产能力，先补安全、schema、日志、API 和运维底座。
