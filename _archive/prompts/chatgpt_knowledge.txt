下面这轮我会直接把 V2 的知识层当成一个独立子系统来审。
我的核心判断是：

V2 现在有“知识文件”，但还没有真正的“知识层”。
它有 knowledge/manifest.json、packs、domain-features、creative-bible、promptKnowledgeAudit、knowledgeGrowthPlan，但这些东西还没有形成一个可检索、可版本化、可参与 24 阶段流程、可被 CRUX 调度的统一知识系统。


一、总体判断
当前知识层像这样：
knowledge/manifest.jsonknowledge/packs/*.mdknowledge/domain-features.jsonproject.bibleproject.promptKnowledgeAuditproject.knowledgeGrowthPlanartifact-archive 的 knowledge 文件归档brain-chat-service 的 knowledge feedbackCRUX 外部知识
它们都有价值，但现在缺少一个中心：
knowledge-registry / knowledge-runtime / knowledge-router
也就是说，V2 现在有很多知识资产，但没有统一回答这几个问题：

这就是知识层最大的隐患。

从你给的片段看，它至少包含：

知识包有加载模式；


知识包有触发提示；

明确拒绝乱码、重复、不可验证 provider claims；


强调每个知识项要映射到 production stage 和 acceptance gate。


这些政策非常对。
尤其这条很重要：
Every knowledge item must map to a production stage and acceptance gate.
这说明 V2 的知识层不是“百科资料库”，而是“生产规则库”。

2. 知识不可检索：没有语义搜索，autoflow 需要找相似项目时怎么找？
正方
目前先用关键词匹配可以理解。
因为 manifest.json 里已经有：
domainsstageRoutestriggerHintssummarybudgetHintdefaultLoad
这些字段本身就可以做轻量路由。
比如用户要做赛博雨夜项目，关键词可以命中：

对于 46KB manifest 和少量 packs 来说，关键词 + stageRoutes 早期能跑。
反方
问题是，V2 是视频制作系统，不是普通关键词问答。
autoflow 需要找的不是“某个词”，而是“相似生产经验”。
例如：
用户做：原创赛博雨夜，垂直霓虹市集，活体光轮，琥珀记忆核心
关键词检索可能找到 cyberpunk，但找不到这些更关键的相似关系：
雨夜 + 霓虹 + 垂直城市 + 载具追逐 + 记忆核心
也找不到过往项目里类似的失败经验：
之前某个项目提示词过度复刻红白机车原片之前某个项目右侧画面空之前某个项目神性建筑丢失之前某个项目角色抢主体
这类知识不是关键词能稳定命中的。
没有语义检索会导致：


autoflow 每次都像第一次做项目；


creative-bible 积累的经验无法被相似项目复用；


失败经验无法召回；


domain-features 只能给通用套路，无法给项目级生产记忆；


CRUX 有大量外部知识，但 V2 内部无法精确选择该加载什么。


裁判结论
语义检索不是锦上添花，而是 V2 知识层从“规则文件”变成“生产记忆”的关键。
但不必一步到位上复杂向量库。可以分三层：
第一层：关键词 / stageRoutes / triggerHints第二层：BM25 / TF-IDF 本地文本检索第三层：embedding 向量检索
最小可行方案：
knowledge-index.jsonknowledge-chunks.jsonlknowledge-search.js
每个 chunk 存：
JavaScript{  "chunkId": "core-showrunner:001",  "packId": "core-showrunner",  "version": "1.0.0",  "domains": ["story_creation", "agent_operations"],  "stageRoutes": ["intake", "autopilot", "qc"],  "gates": ["honest_blocker", "controlled_chain"],  "text": "...",  "summary": "...",  "keywords": ["blocked node", "stage brain loading"],  "sourceHash": "..."}

3. 知识不更新：creative-bible 积累了新经验，knowledge/ 会自动更新吗？
正方
brain-chat-service.js 已经引用：

反方

现在的问题可能是：
creative-bible 有新经验brain-chat-service 检测到反馈但 knowledge/packs 没有被更新manifest 没有版本 bump旧项目没有 snapshotpromptKnowledgeAudit 不知道新知识是否纳入

流程：

关键是：
creative-bible 是项目级经验，knowledge/ 是 V2 原生长期知识。两者不能自动无脑合并。

5. CRUX 总控和知识的关系：V2 内部知识 vs CRUX 外部知识
正方
CRUX 能通过 ChatGPT、Gemini 等外部平台获得大量知识，这是巨大优势。
CRUX 可以帮 V2：


查缺补漏；


扩写专业规则；


生成平台提示词；


判断平台输出质量；


从外部模型中提炼 provider contract；


将失败经验转成规则；


做跨平台比较。


这比 V2 只靠本地知识强很多。
反方
危险点也很明显：

CRUX 外部知识丰富，但不一定可验证、可复现、可版本化。

如果 CRUX 临时问 ChatGPT 得到一条规则，然后直接进入 V2 runtime prompt，会出现：


这条知识没有来源记录；


旧项目无法复现；


下次 ChatGPT 回答变了；


它可能和 V2 creative-bible 冲突；


它可能是不可验证 provider claim；


它可能把外部平台的临时建议当成 V2 永久规则。


这和 manifest policy 里的要求冲突：
Reject unverifiable provider claims.Every knowledge item must map to a production stage and acceptance gate.
裁判结论
CRUX 应该是“知识获取与执行总控”，但不应该绕过 V2 知识层。
推荐关系：
V2 Native Knowledge = 稳定、已策展、可复现、可版本化的内部知识Creative Bible = 当前项目的创作约束和锁定规则CRUX External Knowledge = 临时外部参考、候选知识、执行策略
优先级建议：
Stage Contract  > Project Creative Bible  > V2 Native Knowledge Snapshot  > Project Artifact History  > CRUX External Knowledge
CRUX 外部知识只能这样进入 V2：
CRUX external finding        ↓evidence record        ↓candidate knowledge        ↓curation / validation        ↓pack update        ↓manifest version bump
不能直接变成长期知识。

manifest.json 顶部有：

反方
单个 manifest.version 不够。
真正需要版本化的是：
manifest schema versionpack versionknowledge item versionchunk hashproject knowledge snapshot versionprompt template versioncreative-bible versionprovider contract version
否则会出现：

这是可复现性问题。

旧 project rerun 时默认使用旧 snapshot。
只有用户明确选择“升级知识版本”，才使用新 manifest。

不要让 CRUX 自己去读：
knowledge/manifest.jsonknowledge/packs/*.mdartifacts/knowledge/*project.bible/*
这会造成 CRUX 和 V2 对知识理解不一致。
CRUX 应该通过 V2 的知识 API 获取上下文：
GET /api/knowledge/context?projectId=xxx&runId=xxx&stageId=3-keyframes
返回：
JavaScript{  "knowledgeSnapshotId": "ks_001",  "stageId": "3-keyframes",  "rules": [],  "templates": [],  "creativeBible": {},  "acceptanceGates": [],  "negativeConstraints": [],  "providerContracts": []}

2. CRUX 执行任务时必须携带 knowledgeSnapshotId
CRUX task 应该包含：
JavaScript{  "taskId": "task_001",  "stageId": "9-animation",  "knowledgeSnapshotId": "ks_001",  "knowledgeContext": {    "loadedChunks": [],    "rules": [],    "providerContracts": []  }}
CRUX 回传结果时也要带回：
JavaScript{  "taskId": "task_001",  "knowledgeSnapshotId": "ks_001",  "usedKnowledgeChunks": [],  "externalKnowledgeUsed": []}
这样 artifact-evidence 才能记录：
这个视频片段是在哪个知识版本下生成的。

3. CRUX 外部知识只能进入 candidate knowledge
CRUX 从 ChatGPT / Gemini 得到的新知识，不要直接写进 packs。
应该走：
POST /api/knowledge/candidates
格式：
JavaScript{  "source": "crux_external",  "platform": "chatgpt",  "stageRoutes": ["provider_adapter", "animation"],  "claim": "Kling 更适合某类镜头运动……",  "evidence": {    "taskId": "task_001",    "platform": "chatgpt",    "promptHash": "...",    "responseHash": "..."  },  "status": "pending_review"}

4. CRUX 调度前必须先问 V2：当前阶段该用什么知识？
推荐调用顺序：
V2 autoflow 创建 stage task        ↓V2 knowledge-runtime 选择知识        ↓V2 task-broker 把 knowledgeContext 放入 task        ↓CRUX 执行 task        ↓CRUX 使用 knowledgeContext 生成/操作外部平台        ↓CRUX 回传 result + usedKnowledge        ↓V2 archive + evidence
不要让 CRUX 自己决定长期规则。

5. CRUX 可以做知识审查官
CRUX 非常适合做这几件事：
1. 检查候选知识是否重复2. 检查候选知识是否和 creative-bible 冲突3. 把项目经验改写成 bounded rule4. 给每条知识绑定 stageRoute 和 acceptance gate5. 生成 provider contract 草案6. 从 retro 中提炼失败经验
但最终写入 V2 native knowledge 之前，仍然应该经过 V2 的 knowledge-curation。

八、最终优先修复顺序
1. 新增 knowledge-registry.js，统一 manifest / packs / hash / validation2. 新增 knowledge-runtime.js，让 autoflow 每阶段都调用 selectStageKnowledge3. 新增 knowledge-snapshot.js，每个 run 固定知识版本4. 建立 knowledge candidate / curation 流程，承接 creative-bible 和 CRUX 外部知识5. 把 manifest 拆成轻量 manifest + pack metadata + indexes6. 加本地检索：stageRoute + keyword + BM257. 后续再加 embedding / 知识图谱8. CRUX 只能通过 V2 knowledge API 获取知识上下文，不直接读散乱文件
最终一句话：

V2 知识层现在最缺的不是更多知识，而是“知识运行时”。
先让知识能被选择、被冻结、被验证、被归档、被 CRUX 使用，再谈向量化和知识图谱。
