我先按“架构意图”审，不假设源码细节。整体判断：新烬龙V2 的方向是对的，像一个本地化多模态视频制作工作台的雏形；但现在最大风险不是功能少，而是“状态模型、流程模型、插件模型”三套东西没有完全统一。
一、正方：这个架构哪里设计得好，为什么可行
1. 原生 Node HTTP + 插件系统，适合 MVP 快速演化
不用 Express，直接用 http 模块，优点是：


依赖少，部署简单；


路由行为完全可控；


适合做本地工作台、内网工具、单人创作工具；


插件系统可以让后续能力横向扩展，比如字幕插件、配音插件、生图插件、生视频插件、导出插件。


正方观点：
如果目标是“先跑通一键流”，这个选择是合理的。它不像企业级 SaaS 那样一开始就背负复杂框架负担。

2. agent-kernel 的 6 层架构是正确的抽象
Input / Core / Facilitators / Workspace / Storage / Output 这 6 层，本质上把 Agent 工作台拆成了：


输入接收；


核心决策；


辅助工具；


工作空间；


存储；


输出交付。


这比直接写一个大流程函数更健康。
正方观点：
6 层不是为了对应 24 阶段，而是为了提供“系统能力底座”。24 阶段是业务流程，6 层是运行时能力，两者不一定要 1:1 映射。

3. 24 阶段自动流程符合视频制作管线
从 script → storyboard → keyframes → visual-dev → animation → cinematic → voiceover → sound → edit → delivery，这个顺序基本符合专业视频生产链路。
尤其是把：


character；


environment；


prop；


fx；


lighting；


cinematic；


color；


subtitle；


packaging；


archive；


retro；


都拆出来，是有价值的。
正方观点：
它不是简单的“生成一个视频 prompt”，而是试图建立完整制作管线。这一点比普通 AI 视频工具更像“导演工作台”。

4. artifact-archive / artifact-evidence 是很好的方向
很多 AI 工具最大的问题是：
生成完一堆东西，但不知道哪个版本来自哪里、为什么这么生成、哪一步失败过、哪个制品可复用。
你这里有：


制品归档；


证据链；


项目持久化；


资源导演；


说明架构已经意识到“AI 工作流需要可追溯”。
正方观点：
这是从玩具工具走向专业工具的关键。视频制作不是一次性输出，而是资产、版本、证据、复盘的组合。

5. HUD 驾驶舱界面方向正确
多模态视频制作工作台不适合普通表单页面。HUD 驾驶舱能承载：


当前阶段；


Agent 状态；


资源状态；


进度；


错误；


证据链；


输出制品；


一键流控制。


正方观点：
这个 UI 方向和项目定位匹配，尤其适合“导演台 / 控制台 / 制片管线”的体验。

二、反方：哪里藏着 bug 或设计缺陷，为什么有问题
1. 6 层架构和 24 阶段流程没有契约，容易出现“看似覆盖，实际断层”
你提到的 gap 是核心问题。
例如：


storyboard 阶段到底需要 Input 层什么能力？


keyframes 阶段需要 Workspace 里哪些资产？


voiceover 阶段是否依赖 script、character、emotion beat？


delivery 阶段是否必须检查 subtitle、packaging、archive？


agent-kernel 显示 ok，但某个阶段其实缺少关键产物，是否还能继续？


如果没有明确契约，就会出现：

Agent 内核显示 6 层基本正常，但 24 阶段中某些阶段实际没有可用输入、没有验证规则、没有产物标准。

反方结论：
6 层是“系统健康度”，24 阶段是“业务完成度”。这两者必须有映射矩阵，否则 60% 覆盖率会变成假安全感。

2. plugin 热加载容易制造“半新半旧”的运行状态
热加载最大的问题不是加载失败，而是加载成功一半。
典型风险：


旧插件的路由还在；


新插件的 handler 已注册，但状态对象还是旧的；


插件内部定时器没有清理；


插件缓存没有释放；


两个版本同时处理同一个请求；


某个 autoflow run 启动时用 A 版本插件，中途热加载成 B 版本插件。


反方结论：
插件热加载如果没有生命周期管理，会造成非常隐蔽的状态错乱。尤其是 AI 工作流一跑就是多阶段，不能中途被插件版本污染。

3. project-store 文件系统无锁读写会出并发损坏
JSON 文件读写做 MVP 可以，但有几个硬伤：


两个请求同时写，后写覆盖先写；


写入中断导致 JSON 半截损坏；


autoflow 正在写阶段状态，HUD 同时读，可能读到不完整文件；


archive / evidence / project-store 之间可能状态不一致；


如果未来支持多个项目、多窗口、多任务，问题会扩大。


反方结论：
文件系统不是不能用，但必须有最小事务保护：锁、原子写、版本号、写入队列或事件日志。

4. 覆盖率 ≥60% 是危险的魔法数字
60% 作为启动门槛，看起来像质量控制，实际可能不可靠。
比如：


Input、Core、Output 正常，Storage 异常，仍可能超过 60%；


配音、字幕、交付相关能力缺失，但流程仍启动；


某些阶段是关键阻断项，不能用平均分抵消；


不同项目类型需要不同阈值：短视频、电影分镜、广告、MV、解说视频的要求都不同。


反方结论：
一键流不能只靠总覆盖率判断，应该有“关键门禁”。比如 Storage error 必须 blocked，Core error 必须 blocked，Output missing 不能 delivery。

5. 缺少 SSE / WebSocket，会让 HUD 变成“假驾驶舱”
如果没有实时进度推送，HUD 只能：


轮询；


手动刷新；


看不到流式状态；


看不到阶段开始、阶段完成、阶段失败；


用户不知道卡在哪一步；


Agent 内部状态无法形成现场感。


对于 24 阶段流程，实时反馈很重要。否则用户会觉得“一键流没反应”。
反方结论：
HUD 驾驶舱没有事件流，就只是静态面板。至少应该上 SSE。

6. 24 阶段固定流程过硬，可能不适配不同项目
固定流程有利于标准化，但也可能过度僵硬。
例如：


纯图文分镜项目不需要 voiceover；


无角色环境片不需要 character；


纯旁白视频可能不需要 prop / fx；


已上传角色图时，不应重新生成 character；


用户只想做 keyframes，不一定要跑完整 delivery。


反方结论：
24 阶段应该是默认 pipeline，而不是唯一 pipeline。最好支持 profile / preset / skip / optional / required。

7. artifact-archive 只有 30KB 可能过小
如果 artifact-archive.js 只有 30KB，不是问题本身，但归档系统通常需要处理：


大 prompt；


多版本 JSON；


图片路径；


音频路径；


视频路径；


生成参数；


模型信息；


时间戳；


错误信息；


人工修改记录。


如果它只是简单存储，很快会变成“文件列表”，而不是可靠归档系统。
反方结论：
归档不要只存最终文件，还要存“制品元数据 + 来源 + 版本 + 依赖关系”。

三、正反交锋后的核心判断
这个项目不是“架构不行”，而是已经进入了一个临界点：

现在它从普通工具升级成“AI 制作系统”，最大问题会从“能不能生成”变成“状态是否可信、流程是否可恢复、制品是否可追溯”。

所以真正要补的不是更多阶段，而是：


状态契约；


阶段契约；


插件生命周期；


并发安全；


实时事件；


质量门禁。



四、最重要的 5 个修复建议
1. 建立 Stage Contract：给 24 阶段定义统一输入、输出、依赖、验证规则
每个阶段都应该有标准结构：
JavaScript{  id: "3-keyframes",  name: "Keyframes",  requiredInputs: ["script", "storyboard"],  optionalInputs: ["styleBible", "characterBible"],  outputs: ["keyframePrompts", "keyframeList"],  evidenceRequired: true,  blockingRules: [    "script.exists",    "storyboard.shots.length > 0"  ],  kernelLayersRequired: ["Core", "Workspace", "Storage", "Output"]}
这样可以解决：


6 层和 24 阶段无法对应；


覆盖率虚高；


阶段之间输入输出混乱；


某阶段失败后不知道能不能继续。


这是第一优先级。

2. 把 60% 覆盖率 改成“关键门禁 + 可配置策略”
不要只用：
JavaScriptcoverage >= 0.6
建议改成：
JavaScript{  minimumCoverage: 0.6,  hardBlocks: [    "Core:error",    "Storage:error",    "Output:blocked"  ],  stageRequiredLayers: {    "script": ["Input", "Core", "Storage"],    "storyboard": ["Core", "Workspace", "Storage"],    "delivery": ["Storage", "Output"]  }}
也就是说：


总分可以参考；


关键层不能坏；


每个阶段有自己的最低要求；


不同项目类型可以用不同策略。



3. 给 project-store 加“原子写 + 项目级锁 + 版本号”
最小修复方案：


写入时先写 .tmp 文件；


写完后 rename 替换正式 JSON；


每个 projectId 一个 mutex；


每次保存带 version；


读写时检查版本，避免旧状态覆盖新状态。


示意：
JavaScriptproject.version += 1writeFile(project.tmp)rename(project.tmp, project.json)
进一步可以加：


events.log 追加日志；


snapshots/project-v12.json；


崩溃后从最后一个完整 snapshot 恢复。


这会大幅提升稳定性。

4. 插件热加载改成“两阶段提交 + 旧插件 teardown”
插件系统需要明确生命周期：
JavaScriptplugin.init()plugin.registerRoutes()plugin.healthcheck()plugin.teardown()
热加载时不要直接覆盖全局 registry。建议流程：


加载新插件到临时 registry；


执行 healthcheck；


成功后一次性切换路由表；


对旧插件执行 teardown；


如果失败，继续使用旧插件。


关键是：正在运行的 autoflow 不要中途换插件版本。
每次 autoflow run 应该绑定：
JavaScript{  runId,  pluginSnapshotVersion,  startedAt}
这样才能避免半新半旧状态。

5. 增加 SSE 事件流，让 HUD 真正实时
优先用 SSE，不一定上 WebSocket。SSE 更简单，适合服务端单向推送进度。
事件类型可以设计成：
JavaScriptstage:startstage:progressstage:completestage:errorartifact:createdagent:statusplugin:reloadproject:saved
HUD 就可以实时显示：


当前跑到第几阶段；


每阶段状态；


新生成了什么制品；


哪个插件失败；


哪个 Agent 层变成 warn/error；


是否被 blocked。


这会显著提升工作台体验。

五、我给这个架构的最终评价
正方赢在方向：
它已经不是简单 AI prompt 工具，而是在尝试做“多模态视频制作管线 + Agent 驾驶舱 + 制品归档系统”。这个方向是成立的。
反方赢在落地风险：
当前最危险的是状态一致性。6 层 Agent、24 阶段 Autoflow、插件系统、项目存储、HUD API 如果各自为政，后面 bug 会非常难查。
最终优先级排序：


先做 Stage Contract，统一 6 层和 24 阶段。


再修 project-store，加锁、原子写、版本号。


然后重构插件热加载生命周期。


接着加 SSE 事件流。


最后把 60% 硬编码改成项目类型可配置的质量门禁。


一句话总结：

新烬龙V2 的架构方向可行，但现在必须从“能跑流程”升级到“流程状态可信”。否则 24 阶段越完整，隐藏状态 bug 越多。
