2. P0 / P1 / P2 问题清单
P0：仍然存在的高优先级问题
P0-1：agent_flow.py 仍含 LLM 调用
这是当前最危险的结构问题。
既然已经有 llm_client.py，那么 agent_flow.py 继续直接引用 LLM，就说明 LLM 边界还没有真正收口。
最终依赖应该是：
agent_flow.py↓flow_planner / intent_service / repair_planner↓llm_client.py
不应该是：
agent_flow.py↓OpenAI / provider / chat / completion / prompt raw call
建议目标：
agent_flow.py 中 LLM provider、messages、completion、chat、OpenAI-compatible 调用全部归零。
agent_flow.py 只保留业务编排，不保留底层 LLM 调用。

P0-2：executor.py 2120 行仍是执行层巨石
executor.py 是运行时最容易出事故的地方。
它比 agent_flow.py 更危险，因为它直接关系到：
超时重试取消队列状态输出收集ComfyUI 连接节点执行错误文件路径解析资源错误
如果这些还在一个 2120 行文件里，后期问题会很难定位。
建议拆成：
execution/├── executor.py            # 对外门面├── client.py              # HTTP / WebSocket 客户端├── queue.py               # 队列提交 / 查询 / 取消├── monitor.py             # 进度监听 / 状态轮询├── result_collector.py    # 输出收集├── errors.py              # 执行错误类型├── retry.py               # 重试策略├── timeout.py             # 超时策略└── state.py               # job 状态模型
第一阶段不要全拆，先抽：
errors.pytimeout.pyresult_collector.pyqueue.py
这四个最值得先拆。

P0-3：真实集成测试不足
现在 20+ 测试只能说明“能 import、部分逻辑不炸”，还不能证明系统真的稳定。
尤其需要补：
HTTP healthcheck 测试V2 bridge 请求测试agent_flow → llm_client mock 测试executor → fake ComfyUI server 测试超时测试错误响应 schema 测试workflow diagnose / build endpoint 测试
否则后面继续拆 executor.py 和 agent_flow.py 时，风险会很高。

P1：中优先级问题
P1-1：52 个文件超过 300 行
这个不是单个 bug，但说明模块粒度仍偏粗。
不过不建议机械地按 300 行拆文件。
应该优先拆这些类型的文件：
一个文件包含多个职责一个文件同时有 IO + 业务 + schema + report一个文件中函数超过 20 个一个文件修改时经常牵动很多测试
判断标准不是“超过 300 行就拆”，而是：
是否有多个变化原因。
如果一个文件 500 行但职责单一，可以暂时接受。
如果一个文件 280 行但混了 5 种职责，也应该拆。

P1-2：agent.py 是否退役
不建议现在彻底退役。
agent.py 已经从 2255 行降到 272 行，说明它现在更像兼容门面。
这种情况下，正确策略不是马上删除，而是：
保留 agent.py 作为 legacy facadeCRUX / V2 新代码不再直接依赖它新入口走 main.py + V2 bridge + service registryagent.py 只负责旧 API 兼容
建议给它加清晰定位：
agent.py = compatibility facade不是核心业务层不是新功能入口不是内部服务依赖对象
可以在代码里标注：
Python运行class ComfyUIAgent:    """    Legacy compatibility facade.    New code should use V2 bridge / service registry directly.    This class only preserves the old public API.    """
等所有调用方迁移后，再退役。

P1-3：类型提示覆盖率未知
这会影响后续重构安全性。
建议先不追求全项目 100%，而是锁定核心路径：
executor.py / execution/*agent_flow.py / flow/*llm_client.pyV2 bridgecontracts / typeshandlers
最低要求：
公开函数必须有参数类型和返回类型service 层方法必须有返回类型错误对象必须可序列化V2 request/result 必须强类型
可以用：
Bashmypypyrightruff
逐步开，而不是一次性严格化。

P1-4：brain.py 存在但未被 agent_flow.py 使用
现在有两种选择：
方案 A：brain.py 退役
如果 llm_client.py 已经完全覆盖它，那么 brain.py 应该变成废弃兼容层。
brain.py → deprecated wrapper → llm_client.py
方案 B：brain.py 改名为高级语义层
如果你还想保留 brain 的概念，那么它不应该是连接器，而应该是：
brain.py = reasoning/planning facadellm_client.py = provider gateway
也就是：
agent_flow.py↓brain.py / planner.py↓llm_client.py
不要让 brain.py 和 llm_client.py 都做“LLM连接器”。
我的建议是：
保留 llm_client.py 作为唯一 LLM 网关。brain.py 要么退役，要么改成 planner facade。
