应重点审计的错误类型
建议 executor 至少区分这些错误：

4. brain.py 和 agent.py 的关系
现在的关系很可能是：brain.py 是 LLM 能力，agent.py 是业务总控。
但风险在于：
如果 agent.py 直接拼 prompt、直接解析 LLM 输出、直接决定 workflow 修改，那 brain.py 就只是一个“薄封装”，没有形成真正的 LLM 子系统。
推荐关系
应该是：
agent.py 不直接懂 promptagent.py 不直接解析模型原始文本agent.py 只调用 brain 的结构化能力
例如：
Python运行intent = brain.classify_intent(user_message)plan = brain.make_workflow_plan(intent, context)patches = brain.generate_workflow_patches(plan, workflow_summary)
brain.py 应该输出结构化对象，而不是一大段自然语言：
Python运行@dataclassclass BrainPlan:    intent: str    target_workflow_type: str    required_models: list[str]    required_nodes: list[str]    edit_steps: list[WorkflowEditStep]    risk_level: str
关键建议
不要让 LLM 直接修改完整 workflow JSON。
更安全的方式是：
LLM 生成意图 / patch plan↓WorkflowEditService 执行确定性修改↓Validator 验证↓AutoFix 修复↓Executor 执行
也就是：
LLM 负责“想”确定性代码负责“改”Validator 负责“拦”Executor 负责“跑”

8. CRUX 总控如何调用？
CRUX 总控不要调用 agent.py 的细碎函数，而是调用一个统一的执行协议。
推荐调用链
CRUX Orchestrator↓Executor Registry↓ComfyUISubExecutorAdapter↓ComfyUIAgentFacade↓SmartBuild / Validate / Execute↓ComfyUI Executor
也就是：
CRUX 不懂 ComfyUI 细节ComfyUI 子执行器不控制 CRUX 总流程双方通过 Task Contract 通信
CRUX 任务请求格式
JSON{  "task_id": "task_001",  "executor": "comfyui",  "intent": "text_to_image",  "input": {    "prompt": "warm lunch scene...",    "negative_prompt": "",    "aspect_ratio": "16:9",    "style": "photorealistic"  },  "constraints": {    "timeout_seconds": 900,    "max_retries": 1,    "safe_mode": true  },  "assets": [],  "return_format": ["image", "metadata", "diagnostics"]}
CRUX 调用流程
1. CRUX 判断任务类型2. 选择 comfyui executor3. 调用 healthcheck4. 调用 plan5. plan 通过后调用 execute6. execute 返回结果7. CRUX 统一记录日志、产物、错误、重试
CRUX 不应该做的事
不要直接拼 ComfyUI workflow不要直接改节点 JSON不要直接处理 ComfyUI 队列不要直接读 ComfyUI 输出目录不要直接装节点/模型
这些都应该由 ComfyUI 子执行器内部处理。
