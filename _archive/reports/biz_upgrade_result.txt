2. 最急需做的 3 件事
第一件：给 698 个工作流建立“参数模板 / Manifest”
这是最优先的事情。
现在的问题是：
工作流只是 JSON 文件系统不知道哪些字段应该让用户填系统不知道 prompt 在哪个节点系统不知道 seed、width、height、LoRA、输入图、输出节点在哪里系统不知道这个 workflow 适合什么任务
所以 Registry 虽然能加载文件，但加载出来的是“文件”，不是“能力”。
你需要把每个 workflow 变成：
workflow.json + workflow.manifest.yaml
或者：
workflow.json + workflow.meta.json
Manifest 负责描述：
这个工作流能做什么需要哪些模型需要哪些自定义节点有哪些用户可填参数参数应该写入哪个 node / input输出结果在哪里取适合 CRUX 哪个阶段
完成这个后，698 个工作流才会从“库存”变成“可调度能力”。

第二件：把 executor.submit() 真正接上 ComfyUI
这是业务闭环的硬门槛。
现在如果 executor.submit() 只是接口定义，那系统最多只能做到：
选模板填参数生成 workflow JSON
但还不能真正完成：
提交 ComfyUI排队轮询执行拿结果回传 CRUX
最小可用 executor 必须支持：
POST /promptGET /history/{prompt_id}GET /viewWebSocket 或轮询状态超时失败错误结构化结果文件下载 / 本地索引
建议最小结果结构：
JSON{  "job_id": "comfy_20260706_001",  "prompt_id": "abc123",  "status": "success",  "workflow_id": "flux_txt2img_basic",  "outputs": [    {      "type": "image",      "filename": "ComfyUI_00001_.png",      "path": "/outputs/ComfyUI_00001_.png",      "url": "/api/results/comfy_20260706_001/ComfyUI_00001_.png"    }  ],  "duration_ms": 84321,  "error": null}
没有这个，CRUX 调用也只是“调用接口”，不是真正拿到产物。

第三件：做“工作流推荐器 + 参数引导”
当用户说：
做视频做分镜用参考图生成把图动起来高清放大用 LoRA 出角色图
系统必须能回答：
应该用哪个 workflow为什么用它需要用户补哪些参数缺哪些模型或节点能不能直接跑
推荐器不是 LLM 随便猜，而应该基于 manifest 打分。
推荐逻辑可以这样：
用户请求↓识别任务类型：txt2img / img2img / video / upscale / storyboard / lora↓匹配可用 workflow manifest↓根据输入资产、模型可用性、阶段、质量、速度打分↓返回 Top 3 推荐↓选择最佳 workflow↓生成参数表单或自然语言追问
推荐打分示例：
score =任务匹配度 40%输入资产匹配 20%模型已安装 15%CRUX 阶段匹配 10%成功率 / 历史表现 10%速度偏好 5%

3. 如果先做“参数模板”，技术方案是什么？
核心原则
不要直接改原始 workflow JSON。
应该采用：
原始 workflow.json+参数模板 manifest+运行时 patch
也就是：
workflow.json 是结构manifest 是说明书runner 根据 manifest 把用户参数写入 workflow

推荐文件结构

Registry 加载时，不再只索引 JSON，而是生成一张能力卡：
WorkflowCapability├── workflow_id├── task_type├── supported_inputs├── required_params├── optional_params├── required_models├── required_nodes├── output_types├── crux_stage└── quality_tags

参数模板的关键字段
每个参数至少要有这些字段：
id              参数唯一名label           UI 展示名type            string / integer / float / boolean / image / video / enumrequired        是否必填default         默认值constraints     min / max / options / regexui              前端组件建议binding         写入 workflow 的位置
其中最重要的是 binding：
YAMLbinding:  node_id: "3"  class_type: KSampler  input: seed
这代表：
把用户填写的 seed 写入 workflow["3"]["inputs"]["seed"]

Runner 执行流程
有了 manifest 后，workflow_runner 应该这样跑：
1. 根据 workflow_id 找到 workflow.json2. 加载 manifest3. 合并用户参数 + 默认参数4. 校验参数类型 / 范围 / 必填项5. 根据 binding patch workflow6. WorkflowValidator 校验 JSON7. executor.submit(workflow)8. ResultManager 收集结果9. 返回 CRUX 统一结果
伪代码：
Python运行def run_workflow(workflow_id: str, user_params: dict) -> RunResult:    workflow = registry.load_workflow_json(workflow_id)    manifest = registry.load_manifest(workflow_id)    params = param_resolver.resolve(        manifest=manifest,        user_params=user_params,    )    param_validator.validate(        manifest=manifest,        params=params,    )    patched_workflow = workflow_patcher.apply(        workflow=workflow,        manifest=manifest,        params=params,    )    workflow_validator.validate(patched_workflow)    job = executor.submit(patched_workflow)    return result_manager.wait_and_collect(job)

参数模板自动生成方案

建议用“自动推断 + 人工校正”的方式。

扫描 workflow JSON，识别常见节点：

Manifest 可以加状态：

审核后改为：
YAMLstatus:  generated: true  verified: true  last_tested_at: "2026-07-06"  test_status: passed

或者：

参数模板和自然语言引导如何连接？
自然语言不要直接修改 workflow。
自然语言应该先变成标准参数：
用户：做一个 16:9 的赛博雨夜视频，镜头慢慢推进，4 秒↓NL 参数解析↓{  "task_type": "image_to_video",  "prompt": "赛博雨夜，霓虹街道，镜头慢慢推进",  "aspect_ratio": "16:9",  "duration": 4,  "camera_motion": "slow_push_in"}↓推荐 workflow↓填充 manifest 参数↓patch workflow↓执行
关键是：
NL 只负责提取意图和参数workflow_runner 负责确定性填充

推荐器如何利用参数模板？
有了 manifest 后，推荐器才能工作。
例如用户说：
做视频
推荐器会筛选：
task_type in ["text_to_video", "image_to_video", "video_to_video"]
如果用户上传了图片：
优先 image_to_video
如果用户没有上传图片：
优先 text_to_video 或 先 txt2img 再 img2video 的两阶段 pipeline
如果 CRUX 当前阶段是“分镜转视频”：
优先 storyboard / keyframe_to_video 模板
推荐结果应该返回：
JSON{  "recommended_workflow": "ltxv_img2video_basic",  "reason": "用户请求视频生成，并且提供了输入图片；该模板支持 image_to_video。",  "missing_params": ["prompt", "input_image", "duration"],  "required_models_missing": [],  "confidence": 0.86}

最终结论
你现在真正缺的不是更多工作流，也不是更多 API。
你最缺的是：
把 698 个 workflow 文件变成“可理解、可推荐、可填参、可执行、可回传结果”的业务能力。
最急需做的 3 件事是：
1. 参数模板 Manifest2. 真实 ComfyUI Executor + ResultManager3. 推荐器 + 参数引导
其中第一件最关键。
因为没有参数模板，后面所有东西都会变成猜：
推荐器不知道选哪个前端不知道显示什么表单Runner 不知道 patch 哪个字段CRUX 不知道要传哪些参数LLM 不知道该追问什么
所以正确顺序是：
先 Manifest再 Executor再 推荐器 / UI
一旦 20 个黄金模板跑通，再扩展到 698 个工作流，系统业务成熟度可以从 52 / 100 提升到 72～78 / 100。
