反方，给结论。
2/2

先把上一轮输入区结论补一句：输入区建议采用“默认单行 + Smart Enter + 多行编辑模式 + 5条核心命令 + @文件引用 + 分层历史”。现在继续性能问题。

我的总判断：

CRUX TUI v2 的性能治理必须从第一版就做，不然一旦消息、Markdown、Dashboard、动画、流式输出叠加，prompt_toolkit 会很快出现卡顿、滚动抖动、输入延迟。

性能不是后期优化项，而是 TUI 架构项。

1. 虚拟滚动：是否必须？多大量启用？
正方：虚拟滚动必须做

论点1：消息区是长期增长的。
AI 编程助手不是普通命令行，一次排查可能产生几百条消息、几十个代码块、很多错误栈。如果每次刷新都渲染全部历史，性能一定崩。

论点2：Markdown 和 CJK 宽度计算很贵。
每条消息都要分行、测宽、渲染样式、处理代码块。全量重绘会把 CPU 消耗放大。

论点3：滚动体验依赖虚拟化。
用户上翻历史时，只需要看当前 viewport 附近内容。渲染不可见消息没有意义。

反方：虚拟滚动实现复杂

论点1：消息高度不固定。
Markdown、代码块、CJK、换行、终端宽度变化，都会导致每条消息渲染高度变化。虚拟滚动要维护高度索引，复杂度高。

论点2：小规模消息没必要。
如果只有几十条消息，全量渲染更简单、更稳定。过早虚拟化可能引入 bug。

论点3：搜索、跳转、自动滚到底部会变复杂。
虚拟滚动不仅是只画可见区域，还要支持定位、历史搜索、流式追加、底部跟随。

我的结论

必须做，但分阶段启用。

建议阈值：

0-99 条消息：
允许全量渲染，但使用缓存

100-299 条消息：
启用轻量虚拟滚动，只渲染可见区域 + 上下缓冲

300+ 条消息：
强制虚拟滚动

500+ 条消息：
开启历史压缩策略，旧消息默认折叠

1000+ 条消息：
进入长会话模式，旧消息摘要化/分页化

推荐可见范围：

可见区域：viewport 高度
缓冲区域：上下各 1.5-2 屏
最大渲染：不超过 4-5 屏内容

举例：终端消息区 30 行，则最多渲染：

30 行可见 + 上方 60 行 + 下方 60 行 ≈ 150 行

不是渲染 500 条消息。

2. 增量渲染缓存：LRU？上限？何时失效？
正方：渲染缓存必须做

论点1：消息内容大多数时候不变。
历史消息一旦生成完成，就不会变化。重复解析 Markdown 是浪费。

论点2：代码高亮成本高。
Pygments 对长代码块不便宜，尤其在多次刷新、滚动、Dashboard 更新时重复执行会卡。

论点3：缓存能让 Dashboard/动画刷新不拖累消息区。
右侧刷新时，左侧历史消息不应该重新解析。

反方：缓存也有坑

论点1：终端宽度变化会导致缓存失效。
同一条消息在 120 列和 100 列下换行完全不同，所以缓存必须和宽度绑定。

论点2：主题变化会导致样式失效。
换主题、no-color、high-contrast、ascii-only 都会影响渲染结果。

论点3：缓存会吃内存。
长会话里，如果每条消息都缓存原文、Markdown AST、渲染行、Pygments tokens，内存会涨。

我的缓存策略

建议分三层缓存：

L1：消息原文 MessageStore
L2：结构解析缓存 Markdown AST / block list
L3：终端行缓存 RenderedLines
LRU 是否需要？

需要，但不要只做全局 LRU。建议：

最近消息强保留
可见区域强保留
旧消息 LRU 淘汰
大代码块单独限额
缓存上限建议

按数量和内存双上限：

RenderedLines 缓存：
最近 300 条消息

Markdown block 缓存：
最近 500 条消息

代码高亮缓存：
最近 100 个代码块，或 32-64MB 上限

总渲染缓存：
建议 64-128MB 可配置

轻量版本可以先这样：

render_cache_max_messages = 300
code_cache_max_blocks = 100
render_cache_max_mb = 64
什么情况下主动失效？

必须失效：

1. 终端宽度变化
2. 主题变化
3. no-color / high-contrast / ascii-only 切换
4. 字体宽度策略变化
5. 消息内容仍在流式追加
6. Markdown 解析规则更新
7. CJK 宽度配置变化

部分失效：

Dashboard 刷新：不失效消息缓存
Spinner 动画：不失效消息缓存
状态栏变化：不失效消息缓存
滚动位置变化：只取缓存，不重新解析

关键原则：

消息内容不变，终端宽度不变，主题不变，就不应该重新渲染历史消息。

3. Markdown + Pygments 性能：流式输出怎么渲染？
正方：Pygments 可以继续用

论点1：代码高亮质量好。
CRUX 是编程助手，代码块可读性很重要。Pygments 成熟、语言多、效果稳定。

论点2：最终消息可以高质量渲染。
流式过程中可以降级，输出完成后再用 Pygments 做完整高亮。

论点3：缓存后成本可控。
只要不在每个 token 到来时跑 Pygments，性能问题可以接受。

反方：流式逐行高亮太慢

论点1：Pygments 不适合 token 级实时渲染。
模型一边吐 token，一边解析 Markdown，一边高亮代码，会造成明显卡顿。

论点2：代码块未闭合时解析不稳定。
流式输出中经常出现半个 Markdown 代码块，语言标识还没完整，强行高亮容易错。

论点3：逐字符刷新会导致重绘风暴。
每来一个字符就 invalidate 一次，是 TUI 性能灾难。

我的结论：流式阶段降级，完成后精渲染

不要逐字符，不要逐 token，不要逐行 Pygments。

推荐策略：

流式中：
纯文本/轻量 Markdown 渲染
代码块只显示 monospaced + 边界
不做 Pygments 完整高亮

消息完成后：
一次性 Markdown parse
一次性 Pygments highlight
写入 render cache
流式输出粒度

排序：

逐字符：最差
逐 token：偏差
逐词：可接受但没必要
逐句：较好
逐块：最好

实际推荐：

按时间节流 + 按内容块刷新

具体数值：

流式刷新频率：10-15 fps 上限
最小刷新间隔：66-100ms
积累字符数：>= 20-50 字符再刷新
遇到换行：可以触发刷新
遇到代码块结束：触发精渲染
消息完成：最终精渲染
Markdown 流式渲染分级
StreamingRender：
- 普通文本
- 简单换行
- 简单列表
- 代码块边界
- 不做复杂表格
- 不做 Pygments

FinalRender：
- 完整 Markdown
- Pygments
- 表格/链接/强调
- 缓存结果

这套很关键。

流式阶段追求顺滑，完成阶段追求漂亮。

4. Dashboard 刷新频率：1s 是否合理？
正方：1s 刷新合理

论点1：实时感强。
Agent、工具、ComfyUI、任务进度如果 1 秒更新一次，用户会觉得系统活着。

论点2：实现简单。
固定 1s ticker 好写、好调试，不需要复杂状态机。

论点3：大多数系统能承受。
如果 Dashboard 内容很少，1s 刷新不一定是大问题。

反方：固定 1s 不合理

论点1：空闲时浪费。
没有工具调用、没有生成、没有错误时，1s 刷新纯属无效重绘。

论点2：忙碌时 1s 可能太慢。
工具执行进度、ComfyUI 队列、下载进度，有时 1s 显得不够实时。

论点3：Dashboard 不该打扰消息渲染。
固定刷新会和流式输出、动画、输入光标竞争渲染资源。

我的结论：动态刷新

建议按状态分级：

Idle 空闲：
5-10s 刷新一次，甚至事件驱动

Typing 用户输入中：
2-5s 刷新，避免干扰输入

Generating 模型生成中：
1s 刷新

Tool Running 工具执行中：
500ms-1s 刷新

Download / ComfyUI Queue：
500ms-1s 刷新

Error 状态：
事件触发刷新，不需要持续高频

Hidden Dashboard：
不刷新 UI，只更新后台状态缓存
推荐默认值
空闲：5s
普通生成：1s
工具执行：1s
高频进度任务：500ms
Dashboard 隐藏：不渲染
SSH：最低 2s

关键不是“多久刷新一次”，而是：

只有可见面板才刷新 UI；不可见面板只更新数据，不触发重绘。

5. 动画帧率控制：Spinner / PulseText / AnimatedBorder
正方：动画应该保留

论点1：反馈系统正在工作。
AI 生成和工具调用有等待时间，Spinner 能减少“是不是卡死了”的疑虑。

论点2：增强品牌感。
AnimatedBorder 和 PulseText 在欢迎屏、执行中状态能让 CRUX 更有生命感。

论点3：比静态文本更容易吸引注意。
临时提示、等待状态、当前焦点可以用动画强调。

反方：动画是性能和可读性风险

论点1：动画会触发持续重绘。
终端动画不是免费，尤其 prompt_toolkit 下多组件同时动，会造成输入延迟。

论点2：视觉干扰大。
用户看代码时，边框渐变、文字闪烁、spinner 同时动，会明显分散注意力。

论点3：SSH/tmux 下兼容性差。
braille spinner、Unicode 边框、快速刷新，在远程环境里可能抖动、乱码、延迟。

我的结论：统一动画调度器 + 严格帧率

不要每个 Widget 自己定时刷新。
要有统一的 AnimationScheduler。

推荐帧率
本地 full 动画：
Spinner 8 fps
PulseText 2 fps
AnimatedBorder 4 fps

reduced-motion：
Spinner 4 fps
PulseText 关闭
AnimatedBorder 关闭或 1 fps

SSH/tmux/screen：
Spinner 2-4 fps 或 ASCII spinner
PulseText 关闭
AnimatedBorder 关闭

no-animation：
全部关闭，用静态状态符号
是否 SSH 下全关？

我的建议：

SSH 默认 reduced-motion，不是全关
检测到低带宽/抖动/TERM=dumb 时全关
用户可手动开启 full

SSH 下推荐：

⠋ braille spinner → /-\| ASCII spinner
AnimatedBorder → 静态边框
PulseText → 静态提示
One Motion Rule

必须坚持：

同一时间只有一个主动画源

优先级：

工具执行 Spinner > 生成 Spinner > 焦点提示 > 边框动画

如果模型正在流式输出，就不要再让 Dashboard 边框疯狂动。

6. 内存管理：长时间对话后如何处理？
正方：需要上下文窗口管理

论点1：AI 上下文不是无限的。
长对话会导致模型上下文膨胀，成本增加、响应变慢、质量下降。

论点2：UI 消息存储也会膨胀。
原文、渲染缓存、代码高亮缓存、附件引用、工具日志都会占内存。

论点3：编程任务需要保留有效上下文，而不是所有上下文。
旧的闲聊、失败尝试、重复日志不一定值得留在模型上下文里。

反方：自动裁剪有风险

论点1：旧消息可能仍然重要。
架构讨论、约束条件、用户偏好、关键错误原因可能在很早之前出现。

论点2：摘要可能丢信息。
自动摘要如果做不好，会把细节压没，导致后续模型判断错误。

论点3：用户不喜欢系统偷偷删上下文。
如果 CRUX 自动裁剪，但没有提示，用户会觉得模型“失忆”。

我的结论：UI 历史、模型上下文、渲染缓存三者分开管理

不要混为一谈。

UI 历史：
可以保存很多，负责用户回看

模型上下文：
必须精选，负责模型理解当前任务

渲染缓存：
只为性能服务，可以随时淘汰
超过多少条自动裁剪？

建议按 token 和消息数双阈值。

UI 层
0-500 条消息：
完整保留

500-1000 条消息：
旧消息折叠，缓存淘汰

1000+ 条消息：
分页/归档，只保留最近 300 条活跃渲染

UI 层不一定删除，只是不全部渲染。

模型上下文层

按上下文使用率触发：

ctx < 60%：
不压缩

ctx 60-75%：
提醒即将压缩，优先移除低价值工具日志

ctx 75-85%：
自动生成阶段摘要，压缩旧消息

ctx > 85%：
强制上下文整理，保留任务目标、约束、文件引用、最新错误、关键决策
渲染缓存层
最近 300 条消息保留渲染缓存
超过 300 条 LRU 淘汰
大代码块优先淘汰高亮缓存
裁剪策略：截断还是摘要？
截断的优点
简单
确定
不会制造错误摘要
截断的缺点
容易丢关键约束
长任务会突然失忆
摘要的优点
保留长期任务脉络
适合架构讨论和修复链路
摘要的缺点
可能漏细节
需要额外模型调用
需要摘要质量评估
我的方案：分类型处理
闲聊/重复确认：
可截断

工具日志：
保留最后 N 行 + 错误摘要

代码 diff：
保留文件名、改动摘要、关键 hunks

用户明确约束：
永不自动丢弃，进入 pinned memory

架构决策：
摘要化保留

错误栈：
保留错误类型、文件、行号、最后 30-80 行

最重要的是支持 pinned：

/pin 当前消息
/pin constraint
/pin @file

或者自动识别：

用户说“以后都要”
用户说“必须”
用户说“不要”
项目配置
安全约束
7. 性能原则：3 条铁律
铁律一：不可见的东西不渲染

任何不可见区域、隐藏 Dashboard、折叠 thinking、屏幕外消息，都不能触发 UI 重绘。

正方

论点1：这是 TUI 性能的根本。
终端渲染能力有限，只渲染可见内容是最直接有效的优化。

论点2：能解决长会话卡顿。
消息再多，只要只渲染 viewport，性能就稳定。

论点3：能隔离 Dashboard 和消息区。
Dashboard 刷新不应该导致消息历史重新渲染。

反方

论点1：实现复杂。
需要维护可见区域、消息高度、滚动偏移、缓存、失效逻辑。

论点2：调试难。
虚拟滚动 bug 往往表现为跳动、错位、空白、滚不到底。

论点3：搜索和跳转更难做。
不可见内容不渲染，但搜索仍要能命中。

结论

必须坚持。
具体落地：

100条以上启用虚拟滚动
300条以上强制虚拟滚动
Dashboard 隐藏时不刷新 UI
thinking 折叠时只渲染一行摘要
铁律二：流式阶段只求顺滑，完成后再求漂亮

模型输出过程中不要做重 Markdown、高亮、复杂排版；消息完成后再精渲染并缓存。

正方

论点1：流畅比漂亮更重要。
用户在等待模型输出时，最怕卡顿和延迟，而不是少一点高亮。

论点2：避免 Pygments 重复解析。
代码块没闭合时高亮没有意义，完成后一次高亮更合理。

论点3：降低重绘频率。
按 66-100ms 节流刷新，比 token 级刷新稳定得多。

反方

论点1：流式时视觉效果下降。
用户会看到先是朴素文本，完成后突然变漂亮，有轻微跳变。

论点2：最终重渲染可能导致布局变化。
精渲染后代码块、表格高度变化，滚动位置可能轻微跳。

论点3：实现两套渲染路径。
StreamingRender 和 FinalRender 需要维护一致性。

结论

必须坚持。
具体落地：

流式刷新上限 10-15 fps
Pygments 只在代码块完成后或消息完成后执行
Markdown 流式只做轻量解析
最终渲染结果写入缓存
铁律三：所有刷新必须统一调度、合并、限频

流式输出、Dashboard、动画、状态栏不能各自 invalidate；必须通过统一刷新调度器。

正方

论点1：避免重绘风暴。
多个组件各自刷新，会造成抖动、输入延迟和 CPU 飙升。

论点2：可以按优先级调度。
用户输入优先级最高，其次是流式输出，再是工具状态，最后是动画。

论点3：便于 SSH 降级。
统一调度器可以根据环境降低帧率或关闭动画。

反方

论点1：架构复杂。
简单组件变成事件队列、节流器、调度器，初期开发成本上升。

论点2：状态同步更难。
不同组件刷新频率不同，可能出现状态短暂不一致。

论点3：过度节流会显得迟钝。
如果限频太狠，用户会感觉 Dashboard 和进度条不实时。

结论

必须做。
建议优先级：

P0：用户输入与光标
P1：模型流式输出
P2：工具执行状态
P3：Dashboard
P4：动画
P5：装饰

推荐刷新调度：

输入中：优先保证按键无延迟
流式中：10-15 fps
工具执行：500ms-1s
Dashboard 空闲：5s
动画：2-8 fps，按环境降级
最终性能方案

虚拟滚动强制	300 条消息

最终结论

CRUX TUI v2 的性能设计不能靠“机器够快”。它应该从架构上保证：

1. 消息再多，只渲染当前能看到的
2. 输出再快，也要节流合并刷新
3. Markdown 再漂亮，也不能阻塞流式体验
4. Dashboard 再实时，也不能抢输入和消息

2.【代码高亮】Pygments 是唯一选择吗？
正方：Pygments 是合理默认

论点1：Python 生态成熟。
CRUX TUI 是 Python prompt_toolkit，Pygments 是最自然的语法高亮方案。

论点2：语言覆盖广。

论点3：集成成本低。

反方：Pygments 不是完美答案

论点2：自动识别语言不稳定。
Pygments 的 guess 机制可能误判，尤其是短代码片段、混合日志、错误栈。

论点3：prompt_toolkit 样式整合有成本。
Pygments 输出 token 后还要映射到 prompt_toolkit style，主题同步需要额外维护。

3.【CJK 宽度】east_asian_width 实际效果如何？
正方：east_asian_width 是必要基础

论点1：中文、日文、韩文必须按宽字符处理。
如果不处理 CJK，中文会把边框、表格、列表、状态栏全部挤歪。

论点2：Python 标准库可用。
unicodedata.east_asian_width() 能区分 Fullwidth、Wide、Narrow 等类别，是基本工具。

论点3：对普通中文文本效果不错。
大部分汉字、全角标点、日文假名、韩文字符都能得到接近正确的宽度。

反方：east_asian_width 不够

论点1：Ambiguous 字符依赖终端环境。
一些符号在东亚终端里显示为 2 宽，在西方终端里显示为 1 宽。east_asian_width 无法知道实际终端怎么渲染。

论点2：emoji 很复杂。
emoji、肤色修饰符、ZWJ 组合、国旗、variation selector，不能简单按单字符宽度计算。

例如：

👨‍💻
🏳️‍🌈
🇯🇵
⚙️

这些不是简单 1 个 Unicode code point。

论点3：Nerd Font 图标不可预测。
很多图标在某些字体里是 1 宽，在某些字体里是 2 宽，甚至显示为 tofu 方块。

我的结论：不要只用 east_asian_width，应该使用 wcwidth + 保守符号集

推荐宽度策略：

1. 使用 wcwidth/wcswidth 作为主宽度计算
2. east_asian_width 作为辅助判断
3. 对 emoji 和组合字符做保守处理
4. 对 UI 装饰符号使用白名单
5. 提供 ascii-only 模式

中文：你我他

论点3：部分终端可以显示 Unicode 数学符号。

反方：终端不适合公式渲染

论点1：LaTeX 公式无法优雅排版。
分式、矩阵、上下标、积分、根号，在终端里很难还原。

论点2：渲染依赖外部工具。
真正渲染公式通常需要 MathJax、KaTeX、图片或 Unicode 转换，TUI 不适合内置。

论点3：公式渲染可能严重破坏宽度计算。

数学公式策略

我的建议：

不做真正公式渲染
保留 LaTeX 原文
对 inline math 做轻量突出
对 block math 做代码块式展示

例如：

Inline:
损失函数为 `$L = \sum_i (y_i - \hat{y_i})^2$`

渲染：
损失函数为 [math: L = \sum_i (y_i - \hat{y_i})^2]

Block math：

$$
L = \sum_i (y_i - \hat{y_i})^2
$$

渲染为：

MATH
│ L = \sum_i (y_i - \hat{y_i})^2

5.【图片粘贴】剪贴板图片和终端显示怎么做？

这部分要分清楚两个问题：

1. 图片如何进入 CRUX？
2. 图片如何在终端里展示？
正方：支持图片粘贴很重要

论点1：AI 编程/创意助手需要多模态输入。
用户会贴截图、错误弹窗、UI 设计稿、ComfyUI 节点图、颜色参考、生成图结果。图片入口很有价值。

论点2：比手动保存路径更顺。
用户截图后直接粘贴，比先保存文件再输入路径快很多。

论点3：CRUX 有创意管线，图片是核心资产。
如果 CRUX 要做图片/视频生成工程化，图片引用、预览、附件管理是必要能力。

反方：终端图片粘贴很麻烦

论点1：终端剪贴板图片标准不统一。
不同系统、不同终端、SSH、tmux 对图片粘贴支持差异很大。

论点2：终端内显示图片不可靠。
Kitty graphics、iTerm2 inline image、Sixel、Windows Terminal 支持情况不同。不能依赖某一个协议。

论点3：图片会增加状态管理复杂度。
附件保存、哈希、路径、缓存、清理、隐私、模型上传、导出，都需要设计。

我的结论：图片粘贴要支持，但显示要降级

正确方案不是“终端里强行显示图片”，而是：

图片作为附件进入 CRUX，终端里显示稳定的附件卡片；能预览就预览，不能预览就显示路径和元数据。

图片进入 CRUX 的方案
本地终端

1. 剪贴板图片粘贴
2. 拖入文件路径
3. @image 文件补全
4. /attach path
5. 直接粘贴图片路径

- 保存到项目临时目录
- 计算 hash
- 生成 attachment_id

.crux/attachments/

远程 / SSH 场景

SSH 下不要假设能读取本地剪贴板图片。

推荐降级：

- 提示用户使用 /attach path
- 支持远程文件路径
