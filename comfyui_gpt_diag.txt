下面是简短诊断版，按优先级排序。

问题	根因	修复方案	优先级
1. agent.diagnose() 返回空 / 不深	diagnose 可能只是浅层检查：缺少结构化检查项、没有调用 workflow / ComfyUI / 模型 / 文件状态的真实探测；错误被吞掉或返回默认空结果。	把 diagnose() 改成 诊断管线：env_check → model_check → workflow_check → prompt_check → comfyui_check → execution_check，每项必须返回 {ok, severity, evidence, suggestion}；禁止空返回。	P0
2. agent_flow.py NL→工作流 JSON 经常不合法	自然语言解析、参数提取、节点组装、JSON 校验混在一个大文件里；缺少 schema 验证和自动修复；ComfyUI 节点契约没有显式建模。	拆成：intent_detector.py、param_extractor.py、workflow_builder.py、workflow_schema.py、workflow_validator.py；生成后必须过 JSON Schema + ComfyUI node schema；失败进入 repair loop。	P0
3. chat.py 2802 行中文模式匹配脆弱	正则 / if-else 模式膨胀；中文表达变体太多；解析、路由、会话状态、工具调用耦合。	拆成：chat_session.py、chat_router.py、command_parser.py、intent_classifier.py、conversation_state.py；规则只处理高置信命令，其他交给 BrainProvider；加入 intent confidence。	P1
4. execution_repair.py 错误覆盖不全	错误类型是经验枚举，不是从 ComfyUI 实际错误日志归纳；缺少错误分类层、修复策略表、回归样本。	建立 ErrorClassifier：按 missing_node / bad_input / model_missing / shape_mismatch / oom / timeout / invalid_json / connection_error 分类；每类对应 repair strategy；把真实失败日志沉淀成测试用例。	P1
5. server.py 105 行缺错误处理	Flask API 太薄，直接调用核心逻辑；没有统一异常包装、请求校验、错误码、trace_id、超时控制。	加统一 API 层：validate_request()、handle_exception()、ApiResponse、trace_id；所有接口返回 {success, data, error, trace_id}；核心异常不要直接暴露给前端。	P1
6. 无集成测试	当前只保证语法通过，没验证真实链路：NL→workflow→validate→submit→result→repair。	建 tests/integration/：覆盖 txt2img、img2img、txt2vid、非法 prompt、缺模型、ComfyUI 断连、JSON 损坏、repair 成功/失败；用 fake ComfyUI server 做稳定测试。	P0
最短结论

当前最大问题不是某个函数坏了，而是缺少契约层和闭环验证。

优先修这三个：

P0：workflow JSON schema + validator + repair loop
先保证 agent_flow.py 产物永远可验证、可修复、可解释。

P0：diagnose() 结构化诊断管线
不允许空诊断，每个模块必须给证据和修复建议。

P0：集成测试跑通主链路
只要没有 NL→workflow→ComfyUI→result 的集成测试，后面拆多少文件都不稳。

推荐落地顺序：

P0-1 workflow_schema.py + workflow_validator.py
P0-2 agent.diagnose() 改成结构化诊断报告
P0-3 tests/integration/fake_comfyui_server.py
P1-1 拆 chat.py
P1-2 强化 execution_repair.py 错误分类
P1-3 server.py 加统一错误响应

一句话：先把“生成是否合法、坏了怎么知道、坏了能不能测出来”这三件事补上，再拆大文件。
