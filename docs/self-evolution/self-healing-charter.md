## 铁律正文

**任何运行错误发生时，必须自行诊断并修复。不等待用户指令，不报告除非修复失败。**

## 自愈四层架构

```
第四层 · 代码级     self_audit.py    语法/导入/配置/测试 → 自动修
  ↑ 前三层都失败
第三层 · 会话级     recovery.py      供应商死/磁盘满/配置坏 → 自动切
  ↑ 重试耗尽
第二层 · 请求级     resilience.py    错误分类 → 指数退避重试 → 降级
  ↑ 首次失败
第一层 · 运行时     watchdog.py      每10秒探活，自动切换/清理
```

### 第一层：Watchdog（始终运行）

| 检查项 | 频率 | 动作 |
|--------|------|------|
| Provider 健康 | 每30秒 | ping 失败 → 自动切换供应商 |
| 磁盘空间 | 每120秒 | 低于1GB → 清理 output/images + tmp |
| 上下文内存 | 每60秒 | 超过800K tokens → 自动压缩 |
| 子进程存活 | 持续 | 进程死 → 自动重启 |

### 第二层：请求级自愈（每次API调用）

```
错误发生
  ↓
ErrorClassifier 分类 (10种错误类型)
  ↓
RetryPolicy 决策：
  - Rate Limit (429)     → 指数退避 1s→2s→4s→8s，最多3次
  - Network Error        → 重试3次，间隔2s
  - Auth Error (401/403) → 不重试，立即报告
  - Content Policy       → 自动调整输入，重试2次
  - Timeout              → 重试2次，超时翻倍
  - Validation Error     → 不重试，记录并返回
  - Unknown              → 重试1次
  ↓
3次全部失败 → 熔断（60s冷却）→ 升级到第三层
```

### 第三层：会话级恢复（Playbook）

| 场景 | 触发条件 | 自动动作 |
|------|---------|---------|
| 供应商宕机 | 连续3次超时/连接拒绝 | 按优先级切换：deepseek→crux→local→codebuddy |
| 配置损坏 | JSON解析失败 | 从 snapshots/ 恢复最近备份 |
| 磁盘不足 | 清理后仍<500MB | 清理旧日志、临时文件、cache |
| 模型错误 | 模型返回异常 | 降级参数重试，仍失败切换供应商 |

### 第四层：代码级自愈（按需/定期）

```
触发：每次对话开始 / 用户说"检查一下" / 发现代码异常
  ↓
self_audit.scan()
  ├── 语法检查 (ast.parse 所有 .py)
  ├── 导入检查 (wildcard import)
  ├── 异常检查 (裸 except)
  ├── 配置检查 (JSON 有效性)
  ├── 技能检查 (SKILL.md frontmatter)
  ├── 测试检查 (pytest 冒烟)
  └── Git 检查 (冲突检测)
  ↓
自动修复（安全范围内）：
  ✅ 语法错误 → ast.parse → edit_file 修正
  ✅ JSON 格式 → json.loads → 重写
  ✅ 缺失文件 → 从备份恢复
  ❌ 架构问题 → 报告但不自动改
```

## 自愈边界（不可逾越）

### ✅ 自动修

- API 调用失败 → 重试/切换供应商
- 配置损坏 → 从快照恢复
- 磁盘不足 → 自动清理
- 语法错误 → 自动修正
- 供应商不可用 → 自动切换
- 网络瞬时故障 → 指数退避重试

### ❌ 不动

- 用户数据（绝不删除用户文件）
- 核心架构（不自动重构）
- API Key（不重新生成，仅恢复）
- Git 历史（不 force push）
- 用户明确要求保留的内容


## 故障无法自愈时的上报

当四层全部失败时，执行以下升级路径：

```
1. 明确告知用户：什么错了、试了什么、为什么失败
2. 给出两个选择：
   A. 手动修复（给出具体命令）
   B. 降级使用（用替代方案继续）
3. 记录到 output/health_state.json
4. 标记为"待人工处理"，不再重复尝试
```
