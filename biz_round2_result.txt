2. 最优先解决的 3 个问题
第一优先：把 executor.submit() 真正接上 ComfyUI
这是当前最大硬缺口。
你现在已经有：
workflow registrymanifest 参数模板recommendation engineworkflow_runnerV2 stage mappingCRUX API
但如果 executor 还是 Stub，整个系统只能停留在“会选 workflow、会生成 workflow JSON”，还不能真正产出图片或视频。
技术方案
实现一个真实的 ComfyHTTPExecutor。
核心能力：
1. 提交 workflow 到 ComfyUI2. 获得 prompt_id3. 轮询执行状态4. 读取 history5. 收集输出文件6. 返回统一 RunResult
最小接口：
Python运行class ComfyHTTPExecutor:    def submit(self, workflow: dict, *, client_id: str | None = None) -> SubmitResult:        ...    def get_status(self, prompt_id: str) -> ExecutionStatus:        ...    def wait(self, prompt_id: str, timeout_seconds: int = 900) -> ExecutionStatus:        ...    def collect_outputs(self, prompt_id: str) -> list[WorkflowOutput]:        ...    def cancel(self, prompt_id: str) -> CancelResult:        ...
提交逻辑：
Python运行def submit(self, workflow: dict, *, client_id: str | None = None) -> SubmitResult:    payload = {        "prompt": workflow,        "client_id": client_id or self.client_id,    }    response = self.http.post(        f"{self.base_url}/prompt",        json=payload,        timeout=self.submit_timeout,    )    response.raise_for_status()    data = response.json()    return SubmitResult(        prompt_id=data["prompt_id"],        number=data.get("number"),        node_errors=data.get("node_errors", {}),    )
结果读取逻辑：
POST /prompt↓返回 prompt_id↓GET /history/{prompt_id}↓解析 outputs↓生成结果 URL↓返回给 CRUX / 前端
输出结果结构建议统一成：
JSON{  "job_id": "job_20260706_xxx",  "prompt_id": "comfy_prompt_xxx",  "status": "success",  "workflow_id": "ltxv_img2video_basic",  "outputs": [    {      "type": "image",      "filename": "ComfyUI_00001_.png",      "subfolder": "",      "comfy_type": "output",      "url": "/api/comfyui/view?filename=ComfyUI_00001_.png&type=output",      "local_path": null    }  ],  "error": null,  "duration_ms": 124000}
这一件做完，系统才从“能力库”变成“能干活”。

第三优先：把 V2 死映射改成“阶段 + prompt + assets → 推荐引擎”
现在 V2 写死 8 条规则，短期能跑，但会限制 1351 个工作流的价值。
你现在已经有推荐引擎，所以 V2 不应该再这样：
stage_1 → workflow_Astage_2 → workflow_Bstage_3 → workflow_C
而应该改成：
V2 stage + prompt + 输入资产 + 约束↓WorkflowRecommender↓Top candidates↓选择最佳 workflow↓参数填充↓runner 执行
技术方案
新增一个统一请求对象：
Python运行@dataclassclass StageRunRequest:    stage: str    prompt: str    assets: list[AssetRef]    constraints: dict    user_params: dict = field(default_factory=dict)
推荐请求：
Python运行@dataclassclass WorkflowRecommendRequest:    stage: str    prompt: str    intent: str | None    assets: list[AssetRef]    constraints: dict
推荐器返回：
JSON{  "workflow_id": "ltxv_img2video_basic",  "confidence": 0.87,  "reason": "当前阶段需要图生视频，用户提供了关键帧图片，该工作流支持 image_to_video。",  "missing_params": ["input_image", "motion_prompt"],  "matched_tags": ["video", "image_to_video", "ltxv"]}
最终调用链应该变成：
CRUX autoflow↓comfyui-workflow-adapter.js↓POST /api/crux/comfyui/run-stage↓V2StageRouter↓WorkflowRecommender↓WorkflowRunner↓ComfyHTTPExecutor↓JobManager / ResultManager↓CRUX receives job_id + outputs
死映射保留为 fallback
不要直接删除 8 条死映射。
把它降级为 fallback：
优先：推荐引擎动态选择失败：使用 V2 默认 stage mapping再次失败：返回缺参 / 缺模型 / 无可用工作流
伪代码：

3. 推荐的落地顺序
不要先改前端，也不要先继续扩充工作流数量。
当前正确顺序是：
1. ComfyHTTPExecutor2. JobManager + ResultManager3. V2 动态推荐路由 + CRUX autoflow 接入
原因很简单：
没有 executor，推荐再准也跑不起来。没有状态和结果，跑起来也无法被业务系统使用。没有 CRUX 接入，V2 只能孤立测试，不能进入总流程。

4. 每个问题的完成标准
executor 完成标准
必须通过这些测试：
1. 能 POST /prompt2. 能拿到 prompt_id3. 能 GET /history/{prompt_id}4. 能解析 SaveImage 输出5. 能解析视频输出6. ComfyUI 未启动时返回 ComfyConnectionError7. workflow 错误时返回 ComfyWorkflowError8. 超时时返回 ComfyTimeoutError

状态 / 结果管理完成标准
必须支持：
1. job_id 映射 prompt_id2. 查询 job 状态3. 查询输出结果4. 失败任务有 error_type / message5. 成功任务有 outputs6. 前端可直接用 URL 展示结果7. CRUX 可拿结构化结果继续下一阶段

V2 动态路由完成标准
必须支持：
1. stage + prompt 动态推荐 workflow2. assets 影响推荐结果3. constraints 影响推荐结果4. 静态映射只作为 fallback5. 返回推荐原因6. 返回缺失参数7. run-stage 能直接执行推荐 workflow8. autoflow 真正调用 comfyui-workflow-adapter.js
