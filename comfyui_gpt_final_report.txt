D. API 契约冻结

现在 Showrunner 通过 HTTP API 调 Agent，这个方向是对的。推出前要冻结 API 契约：

GET  /health
POST /classify
POST /diagnose
POST /repair
POST /execute
POST /produce
POST /actor/design

每个接口固定返回：

Python
运行
{
    "ok": true,
    "data": {},
    "error": null,
    "trace_id": "...",
    "duration_ms": 1234
}

不要让每个端点返回格式各不相同。推出后最难维护的就是 API 返回结构漂移。

run_id

status

output_paths

一个小风险：你汇报里的分类修复口径要再核对

你写的是：

classify("状态") → status 而非 diagnose，已修复：将"状态"和"状态检查"加入 diagnose 关键词

这里逻辑上有点冲突。

如果你期望：

"状态" → status

那应该加入 status 关键词，不是 diagnose 关键词。

建议最终语义这样定：

"状态" / "当前状态" / "服务状态" / "health" → status
"检查问题" / "诊断" / "为什么失败" / "环境检查" → diagnose
"状态检查" → 可以映射 diagnose 或 status，但必须固定

我建议：

"状态" → status
"状态检查" → status
"诊断状态" → diagnose
"检查环境" → diagnose

否则后面中文意图会出现边界混乱。
