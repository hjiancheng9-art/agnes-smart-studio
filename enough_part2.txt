3. 数据债：够了的阈值

你们已经有 19 个参数表字段，80% 硬编码搬出去了。

结论：不用追 100%。

该搬进参数表的

必须集中管理：

steps
cfg
width
height
batch_size
frames
fps
denoise
sampler
scheduler
vram_cost_gb
min_vram_gb
recommended_vram_gb
quality_mode
model_family
fallback_policy
upscale_factor
tile_size
seed_policy

这些会影响：

质量
显存
速度
模型兼容
用户体验

必须有单位、reason、适用范围。

可以保留硬编码的

这些不用强行搬：

ComfyUI 节点 class_type 名
固定 input key
内部错误 code
manifest 字段名
默认文件名前缀
测试 fixture 常量
schema 固定字段
数据债停止标准
所有“会影响生成结果、显存、速度、稳定性”的参数已经进参数表，就够了。

不是所有常量都要进参数表。

判断公式：

如果产品经理/用户/硬件条件/模型族会改变它 → 进参数表
如果只是协议字段或代码结构常量 → 留在代码里
4. 认知债：够了的阈值

你们有：

ARCHITECTURE.md
GLOSSARY.md

这已经够 v0.4.0。

不需要 docstring 全覆盖

docstring 全覆盖很容易变成形式主义。

应该只给这些地方写：

公开 API
核心数据结构
复杂策略函数
危险边界函数
容易误用的函数

例如：

compile()
WorkflowValidator.validate()
BlueprintRetriever
