# ComfyUI Agent Methodology v1.0 — CWIM

> **定位**：ComfyUI 工作流智能体的决策执行手册。不是知识库（知识库回答"知道什么"），方法论回答"应该怎么做"。
> **配套**：`COMFYUI_WORLD_KNOWLEDGE.md`（1351 模板 + 1336 参数模板）、`core/comfyui_tools.py`（执行层）、`core/comfyui_pipeline.py`（管线层）
> **状态**：v1.0 · 2026-07-06

---

## 一、核心四问

ComfyUI 方法论回答四个根本问题：

| # | 问题 | 方法论答案 |
|---|------|-----------|
| 1 | 如何理解用户任务？ | 任务拆解 → 类型判断 → 输入/输出/约束分析 |
| 2 | 如何决定推荐/复用/生成？ | 决策树按优先级找最大复用路径 |
| 3 | 如何把失败变成可复用经验？ | 失败捕获 → 保存 → 下次同类问题直接复用修复 |
| 4 | 如何保证每次决策一致、可解释、可验证？ | 10 条原则约束 + Validator 门禁 + 审计日志 |

## 二、10 条原则

### 原则 1：永远不要先生成 Workflow

```
收到用户需求
    ↓
理解任务类型（文生图/图生图/视频/LoRA/修复...）
    ↓
理解输入输出（源素材格式/目标格式/尺寸/风格）
    ↓
理解约束（显存/模型可用性/时间预算）
    ↓
再决策 workflow 方案
```

**禁止**：LLM 根据模糊需求直接猜 JSON。

### 原则 2：优先复用成熟 Workflow

优先级链（从高到低）：

```
已有精确匹配 workflow 模板    → 推荐 + 参数适配
已有相似 motif（子图模式）     → 组合 + 参数适配
已有子图片段                  → 拼接
最后才生成新 workflow          → 从 TaskSpec 编译
```

**原因**：成熟 workflow 的稳定性远高于 LLM 即时生成。

### 原则 3：LLM 不直接生成 ComfyUI JSON

LLM 的输出必须是中间表示（IR），经过编译后才成为可执行的 ComfyUI JSON。

```
LLM / 智能体
    ↓  (理解任务、选择策略)
TaskSpec（任务规范）
    ↓  (输入、输出、约束、推荐参数)
WorkflowIR（工作流中间表示）
    ↓  (节点图、连接、参数语义)
GraphCompiler（图编译器）
    ↓  (拓扑排序、类型检查、优化)
ComfyUI JSON（可执行）
    ↓
Validator（校验器）
    ↓
执行
```

### 原则 4：所有 Workflow 必须经过 Validator

```
生成 → Validator → Patch → 执行
         ↓  失败
         Patch → 再校验 → 执行
           ↓ 反复失败
         记录错误 → 学习 → 返回用户理解的消息
```

Validator 检查项：
- 节点类型存在性
- 必需输入完整性
- 图拓扑连通性（无孤立节点）
- 类型匹配（输出→输入）
- 显存/模型可用性

### 原则 5：失败不是结束，而是学习

每次失败自动保存：

```
{
  "workflow": <失败时的完整工作流>,
  "parameters": <参数快照>,
  "error": <错误信息>,
  "diagnosis": <诊断结果>,
  "patch": <修复补丁>,
  "result": <最终结果>
}
```

下次遇到**相同错误签名**时：
```
查错误库 → 有匹配? → 直接复用修复方案
             无? → 诊断 → 修复 → 存入错误库
```

### 原则 6：参数不是数字，而是语义

参数引擎必须理解每个参数的**语义维度**，而非仅传递数值：

| 参数 | 不要理解成 | 要理解成 |
|------|-----------|---------|
| CFG | 7 | Prompt 遵循程度（1=随意, 15=精确） |
| steps | 20 | 质量/时间权衡（低=快速, 高=精细） |
| denoise | 0.7 | 保留原图程度（0=全重绘, 1=不变） |
| seed | -1 | 随机种子（-1=随机, 固定=复现） |
| batch_size | 4 | 并行生成数 × 显存消耗 |

参数推荐应基于：
- **任务类型**（文生图 vs 图生图 vs 视频）
- **模型特性**（SD1.5 vs SDXL vs FLUX）
- **硬件约束**（显存大小）
- **质量要求**（预览 vs 成品）

### 原则 7：所有生成都必须可解释

推荐 workflow 时，必须告诉用户**为什么**：

```
✅ 好的解释：
  "推荐 workflow #318 (txt2img_basic)
  原因：
  - 输入是文本 → 文生图任务
  - 需要 16:9 宽幅 → #318 支持自定义尺寸
  - 显存 12GB → #318 适合 12GB+
  - 风格为赛博朋克 → 已预载 cyberpunk LoRA 推荐"

❌ 差的解释：
  "推荐 workflow 318"
```

### 原则 8：LoRA 是项目，不是文件

LoRA 应作为完整项目进行生命周期管理：

```
LoRA Project
    ├── Dataset（数据集）
    │   ├── Images（原图）
    │   ├── Captions（标注）
    │   └── Quality Score（质量评分）
    ├── Training（训练）
    │   ├── Config（训练配置）
    │   ├── Checkpoints（检查点）
    │   └── Metrics（损失曲线/评估）
    ├── Evaluation（评估）
    │   ├── Test Prompts（测试提示词）
    │   └── Results（生成结果对比）
    ├── Deployment（部署）
    │   ├── Version（版本号）
    │   └── Workflow Bindings（绑定的工作流）
    └── Usage（使用统计）
        ├── Times Used（使用次数）
        └── Feedback（用户反馈）
```

### 原则 9：Workflow 是图，不是 JSON

JSON 只是存储格式，真正的理解应基于图结构：

```
Graph
    ├── Node（节点）
    │   ├── id, class_type
    │   ├── inputs（输入端口）
    │   └── outputs（输出端口）
    ├── Edge（连接）
    │   ├── source_node → target_node
    │   └── source_port → target_port
    ├── Motif（子图模式）
    │   └── 常见子结构（如 encode+sampler+decode 三件套）
    └── Pipeline（管线）
        └── 多个子图的执行顺序
```

所有推理（推荐、匹配、组合、校验）都应基于图结构操作。

### 原则 10：用户面对任务，不是节点

用户界面层面，术语必须面向任务：

```
用户应该说：
    "生成一张赛博朋克风格的城市夜景"
    "把这张照片转成吉卜力风格"
    "训练一个角色的 LoRA"

而不是：
    "用 KSampler + CLIPTextEncode + VAEDecode..."
```

**节点是智能体的内部语言，不是用户的交互语言。**

## 三、决策树

### 3.1 主决策流程

```
收到用户需求
    ↓
┌─ 类型判断 ──────────────────┐
│ 文生图 | 图生图 | 视频      │
│ LoRA训练 | 修复 | 组合任务  │
└──────────────┬──────────────┘
    ↓
┌─ 输入分析 ──────────────────┐
│ 源素材：文本/图片/视频/无   │
│ 目标格式：jpg/png/mp4/gif  │
│ 尺寸/比例：约束或自由       │
└──────────────┬──────────────┘
    ↓
┌─ 约束检查 ──────────────────┐
│ 显存: ?GB                   │
│ 可用模型: ?                  │
│ 时间预算: 快/中/不限        │
└──────────────┬──────────────┘
    ↓
┌─ 匹配决策 ──────────────────┐
│ 步骤1: 精确匹配模板库       │
│   ├─ 找到 → 推荐 + 参数适配 │
│   └─ 未找到 → 步骤2        │
│ 步骤2: Motif组合匹配        │
│   ├─ 找到 → 组合 + 参数适配 │
│   └─ 未找到 → 步骤3        │
│ 步骤3: TaskSpec → WorkflowIR│
│   → 编译 → Validator        │
└──────────────┬──────────────┘
    ↓
┌─ 执行 ──────────────────────┐
│ 提交 → 监控进度 → 收集结果  │
│   ├─ 成功 → 展示 → 存入库  │
│   └─ 失败 → 诊断 → 修复    │
│        ├─ 修复成功 → 展示   │
│        └─ 修复失败 → 学习   │
└──────────────────────────────┘
```

### 3.2 LoRA 训练决策子流程

```
收到"训练LoRA"需求
    ↓
判断 LoRA 类型
    ├─ 角色（人物/动物）
    ├─ 风格（画风/色调）
    ├─ 物体（特定物品）
    └─ 概念（抽象概念）
    ↓
检查数据集
    ├─ 已提供 → 质量评分
    ├─ 需收集 → 生成采集指南
    └─ 不足 → 建议补充
    ↓
Caption 策略
    ├─ 自动标注（WD14/BLIP）
    └─ 手动优化建议
    ↓
推荐训练参数
    ├─ 基础模型选择
    ├─ 学习率/步数
    ├─ 网络维度
    └─ 优化器选择
    ↓
生成训练计划（用户确认）
    ↓
启动训练 → 监控 → 测试 → 部署
```

### 3.3 错误恢复子流程

```
执行失败
    ↓
┌─ 错误分类 ──────────────────┐
│ 类型A: 连接超时/服务不可用  │ → 重试(3次)
│ 类型B: 模型缺失/类型错误    │ → 安装/切换模型
│ 类型C: 显存不足             │ → 降分辨率/降batch
│ 类型D: 节点配置错误         │ → Patch节点
│ 类型E: 图拓扑错误           │ → 重新编译
│ 类型F: 未知错误             │ → 保存+上报
└──────────────┬──────────────┘
    ↓
应用修复策略
    ↓
验证修复
    ├─ 修复成功 → 继续执行
    └─ 修复失败 → 记录错误库 + 返回用户可读错误
```

## 四、知识库与方法论的边界

| 维度 | 知识库 (World Knowledge) | 方法论 (Methodology) |
|------|-------------------------|---------------------|
| 回答 | 知道什么 | 应该怎么做 |
| 内容 | 模板、参数、模型、节点 | 原则、流程、决策树、标准 |
| 结构 | 数据库/索引/语义搜索 | 文档/规则/决策逻辑 |
| 更新 | 随模板/模型增删而变化 | 稳定，版本化更新 |
| 使用者 | 智能体运行时查询 | 智能体决策时遵循 |

**协作方式**：
```
用户需求
    ↓
方法论：判断任务类型 → 决定策略
    ↓
知识库：查询匹配模板 → 获取参数推荐
    ↓
方法论：校验 → 执行 → 验证
    ↓
知识库：存储结果 → 更新使用统计
```

## 五、审计检查清单

每次 ComfyUI 操作完成后，自查：

- [ ] 是否先理解任务再决定方案？（原则 1）
- [ ] 是否优先复用了已有模板？（原则 2）
- [ ] LLM 输出是否经过 IR → Compiler？（原则 3）
- [ ] 是否经过 Validator？（原则 4）
- [ ] 失败是否已记录并学习？（原则 5）
- [ ] 参数是否按语义维度推荐？（原则 6）
- [ ] 推荐是否附带解释？（原则 7）
- [ ] LoRA 是否按项目管理？（原则 8）
- [ ] Workflow 是否按图结构理解？（原则 9）
- [ ] 用户是否面对任务而非节点？（原则 10）
