# CRUX Studio 实战方法论 v1.1

> 定位：CRUX 专用的 AI 编程执行手册。不是百科，是操作指南。
> 配套文件：`AGENTS.md`（通用铁律）、`CLAUDE.md`（模型路由 + 环境配置）。
> 本文件聚焦：**在 CRUX TUI 里怎么干活最稳。**

---

## 0. 一句话心法

```text
CRUX 不是聊天框，是工程终端。
每次对话 = 一次有边界的 engineering session。
主对话做决策，子 Agent 做执行，验证靠证据。
```

## 1. 任务分级（先看这个）

不是所有任务都走完整流程。

### A 级：微任务（直接干）

| 例子 | 流程 |
|------|------|
| 改 typo、改文案 | 直接改 → 看 diff → 跑相关测试 |
| 解释一段代码 | 主对话直接回答 |
| 补一个简单类型 | 改 → typecheck 通过 |

**不需要**：Plan / TDD / Worktree / PR

---

### B 级：普通开发（精简流程）

| 例子 | 流程 |
|------|------|
| 修一个明确 bug | 复现 → 根因 → 回归测试 → 最小修复 |
| 补一个小功能 | 目标 → 简短计划 → 修改 → 测试 → diff |
| 改一个组件 | 同上 |

**最低要求**：有验证命令、有修改摘要、有风险说明。

---

### C 级：复杂工程（完整流程）

| 例子 | 流程 |
|------|------|
| 多文件功能、重构、接口改动 | 只读分析 → Plan → 确认 → 小步实现 → 测试 → Review |
| 依赖升级、CI 修复 | 同上 |

**必须**：先 Plan、改动分阶段、每阶段可验证。

---

### D 级：高风险（人工门禁）

| 例子 | 流程 |
|------|------|
| 鉴权、支付、数据库迁移、部署 | Spec → 风险评估 → 确认 → Worktree 隔离 → Review → CI → 手动合并 |

**不可**：自动执行、自动合并、跳过人工确认。

---

### 1.5 任务升级规则

任务不是一次定死的。执行中出现以下情况，自动升一级：

| 触发条件 | 升级方向 | 补什么 |
|----------|----------|--------|
| 涉及文件超过 3 个 | A→B / B→C | 补验证命令 / 补 Plan |
| 需要修改公共 API | B→C | 补正式 Plan |
| 需要新增依赖 | B→C | 补 Plan + 影响评估 |
| 测试连续失败 2 次 | B→C | 补根因分析 + Plan |
| 子 Agent 返回不确定或失败 | B→C | 补上下文 / 升级给 pro |
| 涉及 auth / payment / db / deploy | 直接升 D | 补 Spec + 风险评估 + 人工确认 |
| 原任务开始影响其他模块 | B→C / C→D | 补影响范围分析 |

升级后必须暂停执行，按新流程走完再继续。

## 2. 子 Agent 路由（硬约束）

> 模型路由以 `CLAUDE.md` 为唯一真源。以下为执行摘要，如有冲突以 CLAUDE.md 为准。

当前模型：`deepseek-v4-pro`（主对话）+ `deepseek-v4-flash`（子 Agent）。

| 任务 | 路由到 | 模型 |
|------|--------|------|
| grep / glob / 读文件 / 搜代码 | Explore Agent | flash |
| 单文件修改、简单重构、写测试 | general-purpose Agent | flash |
| 调试错误、测试失败 | debugger Agent | flash |
| 实现方案设计 | Plan Agent | flash |
| 架构设计、多文件重构 | **主对话** | pro |
| 复杂调试根因分析 | **主对话** | pro |

**禁则**：
- 不派 flash 做架构决策
- 不派 pro 做 grep
- 子 Agent 失败不重试同一类型，换通路
- 独立子任务一次并行发出，不串行等

### 2.1 子 Agent 失败处理

子 Agent 失败后不要重复派同一 Agent 做同一件事。

```text
1. 主对话读取失败原因。
2. 判断失败类型：
   - 上下文不足 → 补上下文，重试
   - 权限不足 → 缩小范围，换只读 Agent
   - 方案错误 → 主对话重新分析
   - 测试失败 → debugger 分析失败原因
   - 工具失败 → 换等效工具或手动执行
3. 缩小任务范围或补上下文后重试。
4. 必要时升级给主对话 pro。
5. 连续失败 2 次 → 停止执行，重新 Plan。
```

## 3. CRUX 工作流（7 步闭环）

这是 AGENTS.md 七大铁律在 CRUX 里的落地版。

```text
1. 明确目标           → 一句话说清楚要做什么
2. 收集上下文          → 相关文件、错误日志、约束
3. 判断任务级别         → A/B/C/D 分级，选对应流程
4. 写 Plan             → A 级跳过 / B 级 3 行简短计划 / C 级正式 Plan / D 级 Spec + 风险评估
5. 补测试              → A 级跑相关验证即可 / B 级行为变更补测试 / C+D 级完整 TDD
6. 验证 + diff 审查    → 跑测试、查 diff、确认无无关改动
7. 清理资源            → 按安全清理流程检查并清理 pytest 残留（Windows）
```

### 各等级测试要求

| 等级 | 测试要求 |
|------|---------|
| A 级 | 跑相关 smoke / typecheck 即可 |
| B 级 | 行为变更必须有测试证据。Bug 修复补回归测试，新功能补正常路径测试。UI/文案/类型调整可只跑验证 |
| C 级 | 完整 TDD：失败测试 → 最小实现 → 重构 |
| D 级 | TDD + 安全审查 + 人工 Review |

没有测试条件时，必须在完成报告中写明"未验证项"。

## 4. 常用命令速查

### 开发命令

```bash
# 启动 TUI
python crux_studio.py --chat

# 健康检查
python crux_studio.py --check

# 快速生成图片
python crux_studio.py -q "一只猫在太空"

# 生成视频
python crux_studio.py -q "海边日落" -v

# 查询视频任务
python crux_studio.py query <video_id>

# 启动 MCP 服务
python crux_studio.py mcp-serve
```

### 测试命令

```bash
# 冒烟测试（快）
python -m pytest tests/test_smoke.py -q

# 全量测试（限制并发）
python -m pytest tests/ -x -q

# 全量测试后清理残留（Windows 必须）
# ⚠ 执行前确认：没有正在运行的 dev server / 生成任务 / 其他重要 Python 长任务
# 先列进程
tasklist /FI "IMAGENAME eq python.exe"
# 确认无重要任务后，再清理大内存残留
taskkill //F //FI "MEMUSAGE gt 40000" //IM "python.exe"
```

### Git 操作

```bash
# 提交前检查
git status
git diff --stat

# 提交（通过 smart-commit skill）
/commit
```

## 5. 检查/优化闭环（八步法）

> 核心原则：**检查和优化不能混在一起做。** 先查后修，先稳后优。
> 适用范围：Bug 诊断、性能瓶颈、异常行为、回归排查。

```text
┌─────────────────────────────────────────────────────────┐
│  建立基线 → 稳定复现 → 增加观测 → 定位责任层             │
│     ↓                                              ↓    │
│  固化防线 ← 性能优化 ← 回归验证 ← 最小修复              │
└─────────────────────────────────────────────────────────┘
```

### 5.1 建立基线

**目标：在改任何东西之前，量化当前状态。**

```text
必须产出：
  - 正常行为参数（benchmark 数值、响应时间、内存占用）
  - 异常行为参数（差异点在哪，差了多少）
  - 环境快照（OS/Python版本/依赖版本/git commit）

工具链：
  pytest --benchmark-skip  # 先确认测试全过
  python -X importtime     # 导入耗时基线
  memory_profiler / tracemalloc  # 内存基线
  timeit / hyperfine       # 性能基线

禁止：
  - 凭感觉说"慢了"
  - 没基线就开始优化
  - 改代码和测基线混着做
```

### 5.2 稳定复现

**目标：确保每次都能触发问题，消除随机因素。**

```text
必须产出：
  - 最小复现脚本（不依赖项目其他模块，独立可跑）
  - 复现率统计（10次跑，几次触发）
  - 触发条件清单（特定数据？特定顺序？特定时机？）

复现率 < 80% → 不进入下一阶段，先增大样本或加日志

禁止：
  - "偶尔会出现" → 必须量化
  - 在复杂环境中复现 → 必须剥离成最小案例
  - 同时改环境和代码来"排查"
```

### 5.3 增加观测

**目标：在关键路径上插桩，让不可见变成可见。**

```text
观测层次（按成本从低到高）：
  L1: logging — 关键函数入口/出口、参数值、耗时
  L2: metrics — 计数器、直方图（prometheus_client / statsd）
  L3: tracing — 调用链（OpenTelemetry / jaeger）
  L4: profiling — cProfile / py-spy / viztracer

选择原则：
  - 先 L1，不够再加 L2
  - L4 只用于性能问题，不用于逻辑 bug
  - 每加一层观测，先验证不改变问题复现率

禁止：
  - 一上来就开 profiler（噪声大，干扰判断）
  - 观测代码留在生产代码里不清理
```

### 5.4 定位责任层

**目标：二分法缩小范围，确定问题出在哪一层。**

```text
分层模型（从上到下）：
  业务逻辑层  → 你的代码
  接口层      → API 调用、序列化/反序列化
  库层        → 第三方依赖
  运行时层    → Python 版本、GIL、GC
  系统层      → OS、文件系统、网络

定位方法：
  1. git bisect — 哪个 commit 引入的
  2. 依赖回退 — 逐个降级依赖版本
  3. mock 替换 — 把怀疑层换成假实现，问题是否消失
  4. 二分注释 — 注释一半代码，判断问题在哪一半

输出格式：
  责任层: [层名]
  根因: [一句话]
  置信度: [高/中/低]
  证据: [日志摘录 / 测试结果]

禁止：
  - 多假设同时验证（串行排查，一次只动一个变量）
  - 置信度 < 中 就进入修复阶段
```

### 5.5 最小修复

**目标：只改必要的，不动其他。**

```text
修复纪律：
  1. 先写回归测试 — 红（失败）→ 绿（通过）
  2. 最小 diff — 能一行不两行，能一个文件不两个
  3. 只修根因 — 不顺手重构，不优化不相关的
  4. 修复后跑全量测试 — 不只是相关测试

验证清单：
  □ 回归测试通过（证明 bug 已修）
  □ 全量测试通过（证明没引入新问题）
  □ 基线对比（性能/行为回到正常范围）

禁止：
  - "顺便把这里也改一下"
  - 修完不跑全量测试
  - 一行改完发现要改十行 → 回退，重新 Plan
```

### 5.6 回归验证

**目标：确认修复没有引入新问题，且 bug 确定不再复现。**

```text
验证矩阵：
  正常路径    → 原功能不受影响
  边界条件    → 空值/超长/并发/超时
  异常路径    → 网络断开/磁盘满/权限不足
  原 bug 路径 → 复现脚本跑 10 次，0 触发

工具：
  pytest -x --lf        # 先跑上次失败的
  pytest -x --ff        # 失败优先跑
  pytest --repeat=10    # 重复跑可疑测试
  stress-test / locust  # 并发场景

禁止：
  - 只跑"感觉相关"的测试
  - "逻辑上没问题"代替实际跑测试
```

### 5.7 性能优化

**目标：在正确性的基础上，做有数据支撑的优化。**

```text
前置条件（缺一不可）：
  □ 修复已验证（回归测试全绿）
  □ 基线已建立（有优化前的量化数据）
  □ 优化目标明确（响应时间 < 100ms / 内存 < 50MB / ...）

优化阶梯：
  O1: 算法/数据结构 — 换 O(n²) 为 O(n log n)
  O2: 减少分配 — 复用对象、批量操作
  O3: 缓存 — lru_cache / redis / 本地文件缓存
  O4: 并发 — asyncio / 线程池 / 多进程
  O5: C 扩展 — Cython / pyo3 / Numba

每次只动一层，优化后立即跑基准对比。

禁止：
  - 没基线就优化（不知道有没有效）
  - 同时做多个优化（不知道哪个生效）
  - 牺牲正确性换性能
  - "理论上更快"代替实测
```

### 5.8 固化防线

**目标：让同一个问题不可能再漏过去。**

```text
防线层次：
  L1: 单元测试 — 覆盖根因的直接场景
  L2: 回归测试 — 覆盖原 bug 的精确复现路径
  L3: 集成测试 — 覆盖相关调用链
  L4: CI 护栏   — pre-commit hook / CI 必过项
  L5: 运行时断言 — 在生产代码中加 invariant check
  L6: 监控告警 — 对应指标超出阈值自动通知

至少做到 L1+L2，高危害 bug 必须做到 L4 或以上。

输出：
  □ 新增测试用例清单
  □ CI 配置变更（如有）
  □ 监控规则变更（如有）
  □ 知识沉淀（wiki / ADR / 反模式清单更新）

禁止：
  - 修完不补测试
  - "这个问题不会再现了"
```

## 6. 新功能 SOP

```text
1. 写目标和验收标准
2. 让 AI 生成 Plan（涉及文件、实现步骤、测试策略、风险）
3. 人类确认 Plan
4. 拆成小任务，每任务补测试
5. 实现 → 验证 → diff
```

## 7. 完成标准（硬约束）

每次完成必须输出：

```text
1. 修改摘要（做了什么）
2. 修改文件（完整路径）
3. 验证结果（跑了什么命令，exit code 是多少）
4. 风险点（可能影响什么）
5. 未验证项（还有什么没测）
```

**禁止输出**：
```text
应该好了。
看起来没问题。
逻辑上可以。
理论上应该工作。
```

## 8. 反模式（自查清单）

开发反模式：

```text
□ 无上下文直接改代码？
□ 无计划就大改？
□ 无测试修 bug？
□ 无根因就修 bug？
□ 无 diff 审查就提交？
□ 重构和功能混在一个改动？
□ 多个 Agent 同时写同一文件？
```

Prompt 反模式：

```text
□ "帮我优化" — 没有具体目标
□ "随便改一下" — 没有约束边界
□ "不要问我，直接做" — 跳过人类确认
□ "把问题都修了" — 一次塞太多任务
```

自动化反模式：

```text
□ 让 AI 自动部署
□ 让 AI 自动删文件
□ 高风险命令设了 Always Allow
□ 测试 exit 0 但没查残留进程
```

```text

约束：不改无关文件、不新增依赖、
      不改 .env、不改生产配置

输出：修改摘要、文件列表、验证结果、风险、未验证项
```

```text
请先不要修改任何文件。
请阅读 [文件/模块] 并输出：
1. 调用链 2. 数据流 3. 关键函数 4. 风险点
```

### Bug 修复

```text
现象：[错误]
复现：[步骤]
日志：[粘贴]
要求：先找根因，补回归测试，最小修复。
不要猜，不要顺手重构。
```

## 10. 方法论迭代规则

方法论本身不是静态的。每次遇到以下情况，更新本文件或 AGENTS.md：

```text
同一规则说了三次 → 固化到 AGENTS.md
同一 Bug 类型出现两次 → 补充到反模式清单
测试命令变化 → 更新命令速查
新工具接入 → 更新路由表
```
