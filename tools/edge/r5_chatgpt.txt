下面继续按“正反方答辩”评审 CRUX 的 协议层设计：MCP + CDP。

总体判断

CRUX 现在这套设计方向是对的：

MCP 负责连接工具生态，CDP 负责接管网页 AI 和浏览器环境。

但当前风险也很明显：

MCP 是能力入口，CDP 是脆弱执行通道。
如果缺少统一状态层，CRUX 就会变成“很多桥接器 + 很多浏览器动作 + 很少全局可控性”。

所以这不是“架构错误”，而是协议层还没有工程化收敛。

【正方 - 协议层设计合理】
论点1：MCP + CDP 的组合优势在哪

MCP 和 CDP 的组合，其实覆盖了两类完全不同的外部能力。

MCP：连接结构化工具

MCP 更适合连接：

本地工具

文件系统

搜索工具

数据库

代码分析器

图像识别工具

各类专用服务

自定义工具服务器

它的优势是：

模型 / CRUX
  ↓
MCP 协议
  ↓
工具服务器
  ↓
结构化结果

也就是说，MCP 比纯浏览器操作更稳定，因为它走的是工具协议，而不是 UI 页面。

它适合做“可定义、可复用、可验证”的能力。

例如：

读项目文件

调本地模型

调截图分析

查日志

执行脚本

调数据库

调第三方 API

操作本地开发工具

这些如果都通过浏览器做，会非常脆弱。
用 MCP 接，是合理的。

CDP：控制没有稳定 API 的网页 AI

CDP 的价值在于它能接管真实浏览器。

这对 CRUX 很重要，因为很多网页 AI：

没有公开 API

API 费用高

API 能力和网页端不一致

网页端有模型选择、文件上传、图像生成、上下文历史等特性

用户已经登录了网页端账号

所以用 CDP 控制 Edge 操作 ChatGPT / Gemini 等网页 AI，本质上是：

把网页 AI 当成一个可操作的外部智能工具。

这在工程上很实用。

MCP 是“工具协议”，CDP 是“网页执行层”。
两者组合后，CRUX 能同时连接：

结构化工具生态 + 非结构化网页生态

这是它的核心优势。

论点2：这种架构的扩展性强在哪

这个架构扩展性强，主要体现在三点。

1. 能力接入方式多样

不同工具可以用不同方式接入：

本地工具       → MCP
远程工具       → MCP
网页 AI        → CDP
浏览器页面     → CDP
本地脚本       → MCP / subprocess
图像识别       → MCP bridge
代码分析       → MCP bridge

这比只依赖一个 API 强很多。

2. 工具和核心系统解耦

如果 MCP 设计得好，CRUX 主体不需要知道每个工具怎么运行。

CRUX 只需要知道：

这个工具提供什么能力？
需要什么输入？
返回什么输出？
失败怎么处理？

这会让主系统保持稳定。

新增工具时，不必改核心逻辑，只需要新增 MCP server 或 bridge。

3. CDP 能补上 API 生态缺口

很多网页产品没有稳定 API。
如果 CRUX 只依赖 API，就会被限制死。

CDP 让 CRUX 能操作：

网页 AI

账号内已有内容

上传控件

网页按钮

生成结果区域

浏览器上下文

多标签页任务

这使 CRUX 具备“人能操作的网页，我也能操作”的能力。

所以从战略上看，MCP + CDP 是一个很现实的组合。

【反方 - 协议层隐患】
论点1：MCP 桥接服务器可能成为瓶颈

你现在有 9 个 MCP 桥接服务器。这个数量不算夸张，但已经足够产生治理问题。

MCP bridge 的风险主要有这些。

1. 桥接服务器变成单点瓶颈

如果某个 bridge 负责多个高频工具，它会成为瓶颈。

典型问题：

CRUX 请求很多
  ↓
MCP bridge 排队
  ↓
工具响应变慢
  ↓
上层以为模型卡住

用户看到的不是“某个 bridge 慢”，而是：

CRUX 整体变慢、不响应、卡住。

所以 bridge 层必须有：

超时

并发限制

排队策略

取消机制

健康检查

熔断机制

否则它会拖垮整个系统体验。

2. 桥接服务器数量越多，状态越分裂

9 个 MCP server 如果各自维护状态，会出现：

A server 认为任务还在执行
B server 认为资源已释放
CRUX 主控层认为已经完成
浏览器层还在等待

这就是协议层缺乏统一状态管理的典型后果。

多桥接服务器不是问题，无统一状态协议才是问题。

3. 工具 schema 和版本容易漂移

MCP 工具如果升级了参数格式、返回格式或错误格式，但 CRUX 主控层没有同步，就会出现：

工具可用但调用失败

参数名不匹配

返回结构解析失败

错误被吞掉

上层做出错误判断

所以 MCP bridge 需要版本化：

YAML
tool_id: crux.browser.snapshot
version: 1.3.0
input_schema_hash: xxx
output_schema_hash: xxx
capabilities:
  - screenshot
  - dom_snapshot
  - click

否则技能市场、工具路由、多智能体都会被协议不稳定拖累。

4. MCP 错误语义不统一

如果每个 bridge 返回的错误都不同，上层就无法智能恢复。

例如：

timeout
tool_not_found
auth_failed
browser_disconnected
page_not_ready
rate_limited
invalid_schema
permission_denied

这些错误必须标准化。

否则 CRUX 无法判断：

是重试？

是切换工具？

是降级？

是要求用户登录？

是终止任务？

是记录为工具故障？

现在如果没有统一错误语义，MCP bridge 会成为“黑盒故障源”。

【反方 - CDP 的脆弱性如何解决】

CDP 最大的问题不是协议本身，而是它控制的是网页 UI。

网页 UI 天生不稳定。

1. DOM 变化会导致选择器失效

比如 ChatGPT UI 更新后：

旧选择器：textarea[data-id="root"]
新结构：contenteditable div + shadow wrapper

结果就是：

找不到输入框

点击错按钮

读取不到输出

上传文件失败

误判生成是否完成

这类问题会长期存在，因为网页端不是为自动化稳定接口设计的。

2. SPA 页面状态复杂

ChatGPT / Gemini 这类网页通常是 SPA。

会有：

虚拟 DOM

懒加载

流式输出

弹窗

登录态变化

网络状态变化

A/B 测试

多版本 UI

滚动容器变化

shadow DOM 或动态 class

所以 CDP 不能只靠 CSS selector。

3. 页面语义和任务状态不等价

浏览器里看到“按钮可点”，不代表任务真的准备好。

例如：

输入框存在 ≠ 页面已登录
发送按钮存在 ≠ 可以发送
回答区域出现 ≠ 生成完成
停止按钮消失 ≠ 内容完整
DOM 有文本 ≠ 不是中间流式内容

所以 CDP 层必须有自己的状态机，而不是简单 click / type / wait。

【改进建议】协议层的工程化加固

我建议把 CRUX 协议层升级成四层：

CRUX Core
  ↓
Protocol Orchestrator 统一协议编排层
  ↓
MCP Gateway + Browser Automation Gateway
  ↓
MCP Servers + CDP Browser

核心不是继续加 bridge，而是增加一个统一的协议治理层。

1. 增加 Protocol Gateway

不要让 CRUX 主逻辑直接调用 9 个 MCP server 和 CDP。

中间应该有：

Protocol Gateway

职责：

统一工具注册

统一调用入口

统一状态记录

统一错误格式

统一超时控制

统一重试策略

统一日志追踪

统一权限校验

统一降级策略

也就是说：

CRUX 不直接相信任何 bridge。
CRUX 只相信 Protocol Gateway。
2. 建立统一状态模型

这是你观察到的核心问题。

建议设计一个 ProtocolState：

Python
运行
class ProtocolState:
    session_id: str
    task_id: str
    tool_id: str
    protocol: Literal["mcp", "cdp"]
    status: Literal[
        "idle",
        "starting",
        "ready",
        "running",
        "waiting",
        "streaming",
        "completed",
        "failed",
        "cancelled",
        "degraded"
    ]
    started_at: float
    updated_at: float
    last_error: Optional[ProtocolError]
    retry_count: int
    artifacts: list

所有 MCP 调用、CDP 页面操作都必须写入统一状态。

这样 CRUX 才能知道：

哪个工具正在跑

哪个页面卡住了

哪个 bridge 掉线了

哪个任务可以重试

哪个任务应该中断

哪个结果已经完成

3. 给 MCP bridge 加健康检查和熔断

每个 MCP server 应该定期暴露：

health
capabilities
version
current_load
last_error
avg_latency
active_tasks

CRUX 不应该盲目调用 bridge。

推荐加：

health check
timeout
circuit breaker
backpressure
retry budget

例如：

Python
运行
if bridge.error_rate > 0.3 and bridge.last_5min_calls > 20:
    circuit_breaker.open()
    route_to_fallback()

这样某个 bridge 出问题时，不会拖死整个系统。

4. MCP 工具能力注册表

每个 MCP 工具需要声明能力，而不是只暴露工具名。

例如：

YAML
id: browser.snapshot
protocol: mcp
version: 1.2.0

provides:
  - screenshot
  - dom_snapshot
  - accessibility_tree

requires:
  - browser_session

input_schema_version: 3
output_schema_version: 2

risk_level: medium
side_effect: false
timeout_ms: 10000
concurrency_limit: 3

这能让上层路由更可靠。

5. CDP 不要只靠 CSS selector，改成多重定位策略

CDP 自动化应该使用定位优先级：

1. accessibility role / name
2. stable data-testid
3. semantic text
4. DOM structure pattern
5. visual position fallback
6. coordinate fallback

不要只写：

Python
运行
page.click(".send-button")

而应该封装成：

Python
运行
browser_action.send_message(text)

内部自己处理不同 UI 版本。

也就是说，上层不能直接接触 selector。
selector 应该被藏在 adapter 里。

6. 为网页 AI 建立页面状态机

例如 ChatGPT 页面状态可以抽象为：

UNKNOWN
  ↓
PAGE_LOADING
  ↓
LOGIN_REQUIRED / READY
  ↓
COMPOSER_READY
  ↓
MESSAGE_SUBMITTED
  ↓
GENERATING
  ↓
GENERATION_COMPLETE
  ↓
RESULT_EXTRACTED

每个状态都有进入条件和退出条件。

比如 GENERATION_COMPLETE 不应该只判断“停止按钮消失”，而应该组合判断：

stop 按钮消失

assistant message 稳定 N 秒

streaming cursor 消失

network idle

最后一段文本不再变化

没有错误 toast

这样 CDP 的稳定性会高很多。

7. 引入 DOM 适配器版本

对 ChatGPT / Gemini / Claude / Edge 页面，应该分别维护 adapter：

adapters/
  chatgpt/
    v2026_07.py
    selectors.yaml
    state_machine.yaml
  gemini/
    v2026_07.py
    selectors.yaml
    state_machine.yaml

每次 UI 变更，只更新 adapter，不动 CRUX 核心。

同时支持自动检测页面版本：

Python
运行
adapter = detect_adapter(page)

如果检测失败，进入降级模式：

请用户确认页面 / 使用视觉识别 / 切换备用选择器
8. 加入视觉兜底

CDP 的 DOM 会变，但截图视觉通常更稳定。

可以建立：

DOM automation primary
Visual automation fallback

比如：

找不到发送按钮时，截图识别按钮位置

找不到输入框时，识别可输入区域

DOM 读不到回答时，截图 + OCR/视觉模型读取

页面弹窗遮挡时，视觉检测并关闭

这能显著提高网页 AI 自动化的韧性。

不过视觉兜底要谨慎使用，因为成本高、速度慢、误判也可能更高。适合做最后一层 fallback。

9. 增加动作回放和失败诊断

每次协议层操作都应该记录：

task_id
tool_id
protocol
action
input
result
screenshot_before
screenshot_after
dom_snapshot_hash
error
duration

这样当 CDP 失效时，不是只看到：

click failed

而是能知道：

点之前页面是什么状态

点的是哪个元素

页面有没有弹窗

DOM 结构是否变化

是 selector 失败还是状态判断失败

这对修复 CRUX 非常关键。

10. 区分“协议错误”和“业务错误”

这点很重要。

例如：

MCP server disconnected

是协议错误。

ChatGPT 返回无法完成请求

是业务错误。

Gemini 页面要求重新登录

是认证状态错误。

按钮找不到

是页面适配错误。

错误类型建议分层：

Python
运行
ProtocolError
  ├─ TransportError
  ├─ TimeoutError
  ├─ SchemaError
  ├─ AuthError
  ├─ CapabilityError
  ├─ BrowserStateError
  ├─ SelectorError
  ├─ PageChangedError
  └─ ToolExecutionError

不同错误必须走不同恢复策略。

推荐协议层架构

我建议最终收敛成这样：

CRUX Core
  ├─ Task Planner
  ├─ Skill Router
  ├─ Agent Coordinator
  │
  └─ Protocol Orchestrator
        ├─ State Store
        ├─ Capability Registry
        ├─ Error Normalizer
        ├─ Retry / Timeout / Circuit Breaker
        ├─ Audit Log
        │
        ├─ MCP Gateway
        │     ├─ Bridge Health Monitor
        │     ├─ Schema Validator
        │     └─ Load Control
        │
        └─ Browser Gateway
              ├─ CDP Session Manager
              ├─ Page State Machine
              ├─ Site Adapters
              ├─ Selector Resolver
              └─ Visual Fallback

重点是：

MCP 和 CDP 都不要裸奔。
它们都应该被 Protocol Orchestrator 接管。

最小可行修复路线
第一阶段：止血

优先做：

所有 MCP / CDP 调用统一超时

统一错误格式

bridge 健康检查

CDP 操作封装成高层动作

selector 不散落在业务代码里

每次 CDP 失败保存截图和 DOM 摘要

这一阶段能解决大部分“卡死”和“不可诊断”的问题。

第二阶段：稳定

继续做：

ProtocolState 状态表

MCP capability registry

CDP 页面状态机

ChatGPT / Gemini 独立 adapter

重试和降级策略

熔断机制

这一阶段能把协议层从“能用”变成“稳定可维护”。

第三阶段：智能

最后做：

自动选择 MCP / CDP / API

根据历史成功率路由

页面 UI 变化自动检测

bridge 负载均衡

视觉兜底

协议层自我修复建议

这一阶段才是成熟形态。

【最终评分：7 / 10】

我给当前 CRUX 的 MCP + CDP 协议层设计 7 分。

加分点

MCP + CDP 组合方向正确。

MCP 适合结构化工具生态，CDP 适合网页 AI 自动化。

9 个 MCP bridge 说明 CRUX 已具备外部工具扩展能力。

控制 Edge 操作网页 AI 是现实可行的增强路线。

架构扩展性强，未来可以接入更多本地和远程能力。

扣分点

MCP bridge 缺少统一治理，容易成为瓶颈。

CDP 自动化受 DOM 和页面状态变化影响明显。

协议层缺统一状态管理。

错误语义如果不标准化，上层无法自动恢复。

selector 如果散落在业务代码里，维护成本会越来越高。

缺少健康检查、熔断、重试、降级、审计日志这类工程化机制。

最终结论

CRUX 的协议层方向是对的，但现在处于：

可扩展，但还不够可控

MCP + CDP 解决的是“能接很多东西”的问题。
但下一阶段要解决的是：

接入之后怎么稳定、怎么诊断、怎么恢复、怎么统一状态。

一句话总结：

MCP + CDP 是 CRUX 的外部能力骨架，但必须补上 Protocol Orchestrator。
否则工具越多、网页越多、桥越多，系统越容易从“强扩展”变成“强混乱
