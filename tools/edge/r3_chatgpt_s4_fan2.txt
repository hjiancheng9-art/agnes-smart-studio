论点2：是不是还会漏掉真正需要多智能体的场景

会漏，尤其是以下几类。

1. 长链调试任务

用户可能只说：

“还是报错。”

这类任务表面短，实则需要回看历史修复、比较前后状态、分析新错误与旧错误关系。

如果 CRUX 不把“连续失败次数”“已修改文件数”“错误类型变化”纳入判断，就会低估任务复杂度。

2. 隐性跨模块任务

例如：

“把这个功能接上。”

这句话没有明显触发词，但可能涉及：

后端 API

前端按钮

状态管理

数据 schema

测试

错误处理

这类任务真正需要多智能体，尤其需要一个“依赖追踪智能体”辅助判断影响范围。

3. 模糊需求澄清任务

例如：

“这个界面不好用，帮我改到专业一点。”

这不一定命中触发词，但可能需要：

UX 分析

UI 设计

代码实现

交互验收

移动端适配

如果只按关键词判断，会把它当成普通改文案或改样式。

4. 高风险写操作

有些任务虽然短，但风险很高：

“删掉没用的文件。”
“清理根目录。”
“重置配置。”
“改启动逻辑。”
“自动修复所有报错。”

这些可能没有 500 字，也未必有两个触发词，但需要多智能体或至少需要审计模式。因为它们可能造成破坏性后果。
