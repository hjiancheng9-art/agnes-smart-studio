最可能不是 palette 没定义，而是 “Style 已经建好，但没有真正覆盖 PTK 画布/容器背景”。

prompt_toolkit 的 Application(style=...) 只是提供样式表；颜色要通过控件、容器、FormattedText fragment 的 style/class 被引用后才会渲染。官方文档也明确说，Window 等控件接收 style 参数，父控件样式可以向子控件传播；FormattedText 里的 class:xxx 才会去匹配 Style.from_dict() 中的 xxx 规则。
python-prompt-toolkit.readthedocs.io
 
python-prompt-toolkit.readthedocs.io

根因优先级判断
1. 根容器/空白区域没有背景样式，导致终端默认白底露出来

这是最高概率。

你现在改了：

Python
运行
Style.from_dict({
    "": "fg:#E2E8F0 bg:#0B0E14",
    ...
})

但这不等于“整个终端画布都刷成深色”。很多空白区域不是 FormattedText fragment，也不是带 class 的文本，它们只是 layout 中未被 Window/Container 明确绘制的区域。白底终端下，这些区域就会保持白色。

应给根级容器、主 Window、body、padding、frame、input 区域全部显式加背景：

然后在布局根部加：

Python

以及每个占空间的 Window：

Python

官方 full-screen demo 也是 Application(..., style=style, full_screen=True)，但它仍然依赖各 Widget/Container 自己的 class/style 才能显示对应颜色，不是 Application 自动全局涂背景。

3. class 名可能“看起来匹配”，但实际 PTK 匹配规则不是你以为的

FormattedText 中应该是：

Python
运行
("class:crux.title", "text")

Style 表里应该是：

Python
运行
"crux.title": "fg:#E2E8F0 bg:#0B0E14"

不是：

Python
运行
"class:crux.title": ...

另外 dotted class 有特殊展开/组合规则。prompt_toolkit 文档说明，class:left.text.highlighted 会按 dotted class 展开并参与组合计算；复杂 class 组合的顺序也会影响最终样式。
python-prompt-toolkit.readthedocs.io

建议直接在运行时打印真实使用的 style strings：

Python
运行
def after_render(app):
    used = sorted(app.get_used_style_strings())
    with open("ptk_used_styles.log", "w", encoding="utf-8") as f:
        f.write("\n".join(used))

app = Application(
    layout=layout,
    style=style,
    full_screen=True,
    after_render=after_render,
)

如果日志里出现大量你没覆盖的：

class:frame
class:window
class:textarea
class:input-field
class:menu
class:scrollbar

那就是根因。

4. 有局部 inline style 覆盖了全局背景

prompt_toolkit 样式是从父到子拼接计算，后面的样式会覆盖前面的样式。官方文档说明，父级可以设置样式，子级也可以部分覆盖。
python-prompt-toolkit.readthedocs.io

所以即使根部有：

Python
运行
bg:#0B0E14

子控件里只要有：

Python
运行
style="bg:#ffffff"

或：

Python
运行
style="reverse"

或：

Python
运行
style="class:xxx noinherit"

就可能把深色背景干掉。

尤其要查 noinherit，它会让局部控件不继承父样式。

5. 没有强制 full_screen=True，只渲染局部，终端剩余区域仍是白色

如果 Application(full_screen=False)，PTK 不会进入 alternate screen buffer，只占 layout 所需空间。官方文档说，full_screen=True 才会在 alternate screen buffer 中运行；否则应用只消耗布局需要的最小空间。
python-prompt-toolkit.readthedocs.io

确认这里没有被条件分支覆盖：

Python
运行
Application(
    layout=layout,
    style=style,
    full_screen=True,
)

如果 CRUX 有普通模式 / fallback 模式 / debug 模式，很可能你改的是 TUI v2，但实际启动的是另一个分支。

6. 颜色深度被降级不是主因，但仍要固定

如果颜色深度太低，#0B0E14 会被映射成近似色。prompt_toolkit 支持 1-bit、4-bit、8-bit、24-bit 等色深，ColorDepth.TRUE_COLOR 才是 24-bit 真彩色。
python-prompt-toolkit.readthedocs.io

但注意：色深降级通常会变成黑/灰，不太会导致“全白”。所以它不是第一根因，只是加固项：

Python
运行
from prompt_toolkit.output.color_depth import ColorDepth

app = Application(
    layout=layout,
    style=style,
    full_screen=True,
    color_depth=ColorDepth.TRUE_COLOR,
)
最小验证法

先不要查 44 个 class，直接做一个硬编码烟雾测试：

Python
运行
from prompt_toolkit import Application
from prompt_toolkit.layout import Layout
from prompt_toolkit.layout.containers import Window
from prompt_toolkit.layout.controls import FormattedTextControl
from prompt_toolkit.styles import Style
from prompt_toolkit.output.color_depth import ColorDepth

style = Style.from_dict({
    "": "fg:#E2E8F0 bg:#0B0E14",
    "app": "fg:#E2E8F0 bg:#0B0E14",
})

root = Window(
    content=FormattedTextControl([
        ("class:app", "CRUX THEME TEST\n"),
        ("class:app", "If this screen is dark, theme engine is OK."),
    ]),
    style="class:app",
    always_hide_cursor=True,
)

app = Application(
    layout=Layout(root),
    style=style,
    full_screen=True,
    color_depth=ColorDepth.TRUE_COLOR,
)

app.run()

我给你的最终判断

按你描述，“palette 已定义 + 44 个 FormattedText class 已匹配 + 空 class 兜底仍全白”，最可能根因是：

你修的是文本 token 样式，不是 PTK 全屏画布样式。CRUX 的根容器、Window、Frame、TextArea、padding、空白区域没有统一 style="class:app" / bg:#0B0E14，所以白色终端默认背景一直露出来。

第二可能是：

实际启动路径没有走 ui/theme_v2.py → build_style_v2("blade")，而是走了旧 TUI / fallback / 纯文本模式 / 另一个 Application 实例。
