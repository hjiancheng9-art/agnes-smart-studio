【正方 - 按风险触发策略合理】
论点1：80%任务单智能体就够了，为什么对

这个修复方向是对的。多智能体不是默认增强器，而是协调成本很高的复杂任务工具。

CRUX 当前的问题不是“智能体不够多”，而是：

简单任务被过度调度，导致延迟增加、上下文分裂、决策不一致、输出变复杂。

对多数任务来说，单智能体更优：

上下文连续性更强
单智能体能完整理解用户意图、当前文件状态、历史决策和约束。多智能体一旦拆分，容易出现“每个子智能体只看一块”，最后整合时丢失全局意图。

执行路径更短
简单修 bug、改一段 UI、解释错误、生成脚本、调整配置，本质是线性任务。强行并行只会增加规划、分派、汇总、冲突解决成本。

降低幻觉和互相污染
多智能体很容易出现：

A 智能体假设某模块存在

B 智能体给出另一套架构

C 汇总时把两边揉在一起

最后看起来“很系统”，但可执行性下降。

更符合 CRUX 的产品定位
CRUX 如果是面向开发/创作的“智能执行系统”，默认行为应该是稳、快、可控，而不是每次都开一个小型委员会。

所以 should_use_multi_agent() 默认 False 是正确的。它把多智能体从“默认模式”降级为“高风险任务专用能力”，这是架构成熟的表现。

论点2：什么场景真的需要多智能体

多智能体真正有价值的场景，不是“任务听起来高级”，而是任务天然存在多视角、多文件、多目标冲突、并行搜索空间。

比较适合开启的场景：

1. 架构审计 / 系统评审

例如：

审计 CRUX Studio 的 TUI、工具路由、上下文压缩、执行器、错误恢复。

这种任务需要不同角色：

架构师：看模块边界

安全审计员：看风险和权限

测试工程师：看验收标准

性能工程师：看瓶颈

产品视角：看体验是否复杂

这类任务单智能体也能做，但多智能体能提升覆盖率。

2. 大规模重构 / 跨文件改造

例如：

把旧版工具注册系统迁移到 TRM 网格，并同步改 CLI、TUI、API、测试。

这是典型多智能体场景。原因是：

文件多

依赖链长

容易漏改

需要分阶段验证

修改之间可能互相影响

可以拆成：

依赖分析智能体

代码修改智能体

测试补全智能体

回归检查智能体

3. 批量任务 / 可并行搜索任务

例如：

扫描 100 个工具定义，找出 schema 不完整、命名不一致、没有 fallback 的工具。

这类任务适合并行，因为每个工具之间相对独立。多智能体可以分片扫描，再统一汇总。

4. 复杂方案设计

例如：

设计 CRUX v6 的插件协议、权限沙箱、工具调度系统和前端交互。

这种不是单点修复，而是多目标权衡：

易用性

性能

安全

扩展性

兼容性

未来迁移成本

多智能体可以提供互相制衡的视角。

5. 正反方辩论 / 红队审计

你现在这种“正反方答辩”其实就很适合多智能体思维。因为目标不是最快给答案，而是逼出盲点。

【反方 - 这个策略的问题】
论点1：触发词阈值会不会误判

会，而且这是当前策略最大的风险。

你现在的规则大概是：

默认 False；匹配 ≥2 个触发词，或任务 >500 字，才开启；遇到阻断词则跳过。

这个比默认开启好多了，但仍然属于关键词启发式，不是语义判断。

误判主要有两类。

误判 A：关键词命中，但任务其实很简单

例如：

“快速解释一下架构是什么意思。”

命中“快速”和“架构”。如果阻断词优先级合理，会跳过。但如果用户说：

“这个架构文件里某个变量报错，直接帮我看一下。”

这里有“架构”“文件”，但可能只是一个小 bug。

再比如：

“帮我重构这一个函数的变量名。”

命中“重构”，但不需要多智能体。

如果仅靠触发词，很容易把“词的重量”误当成“任务的复杂度”。

误判 B：没有命中足够触发词，但任务其实复杂

例如：

“帮我看看这个项目哪里不对。”

这句话很短，没有“架构/审计/重构/迁移/跨文件”等词，但实际可能需要：

看目录结构

跑测试

查启动入口

查依赖

读错误日志

做修复建议

这是一个典型复杂诊断任务，但触发词可能不够。

再比如：

“它还是不行。”

如果上下文里之前已经连续修了多个文件，这句话本身很短，但任务状态已经复杂。仅看当前输入会误判。

所以这个策略如果只看当前 message，会漏掉上下文累积复杂度。

论点2：是不是还会漏掉真正需要多智能体的场景

会漏，尤其是以下几类。

1. 长链调试任务

用户可能只说：

“还是报错。”

这类任务表面短，实则需要回看历史修复、比较前后状态、分析新错误与旧错误关系。

如果 CRUX 不把“连续失败次数”“已修改文件数”“错误类型变化”纳入判断，就会低估任务复杂度。

2. 隐性跨模块任务

例如：

“把这个功能接上。”

这句话没有明显触发词，但可能涉及：

后端 API

前端按钮

状态管理

数据 schema

测试

错误处理

这类任务真正需要多智能体，尤其需要一个“依赖追踪智能体”辅助判断影响范围。

3. 模糊需求澄清任务

例如：

“这个界面不好用，帮我改到专业一点。”

这不一定命中触发词，但可能需要：

UX 分析

UI 设计

代码实现

交互验收

移动端适配

如果只按关键词判断，会把它当成普通改文案或改样式。

4. 高风险写操作

有些任务虽然短，但风险很高：

“删掉没用的文件。”
“清理根目录。”
“重置配置。”
“改启动逻辑。”
“自动修复所有报错。”

这些可能没有 500 字，也未必有两个触发词，但需要多智能体或至少需要审计模式。因为它们可能造成破坏性后果。

【优化建议】如何让触发更准

我建议把 should_use_multi_agent() 从“关键词判断器”升级为任务风险评分器。

不要直接判断：

Python
运行
trigger_count >= 2

而是计算一个 complexity_score / risk_score。

1. 从关键词阈值改成加权评分

不同词的权重不应该一样。

例如：

Python
运行
TRIGGER_WEIGHTS = {
    "架构": 2,
    "审计": 3,
    "重构": 3,
    "迁移": 4,
    "批量": 3,
    "并行": 3,
    "跨文件": 4,
    "全局": 3,
    "回归": 3,
    "自动修复": 4,
    "删除": 4,
    "清理": 3,
    "启动失败": 3,
}

判断逻辑：

Python
运行
score >= 5 -> use multi-agent
score 3-4 -> use planner/reviewer, not full swarm
score < 3 -> single agent

这样比“命中 2 个词”更稳。

2. 加入阻断词，但不要一票否决

阻断词如“简单/快速/直接/只要/解释一下”是有价值的，但不能直接跳过。

例如：

“快速帮我审计整个项目启动失败原因。”

虽然有“快速”，但任务仍然复杂。

所以阻断词应该是减分，不是一票否决：

Python
运行
BLOCKER_WEIGHTS = {
    "简单": -2,
    "快速": -1,
    "直接": -1,
    "只解释": -3,
    "不用改代码": -2,
}

而不是：

Python
运行
if blocker:
    return False

否则用户一句“快速”就把复杂任务压成单智能体。

3. 加入上下文状态特征

这是最关键的优化。

should_use_multi_agent() 不应该只看当前输入，还要看 session 状态：

Python
运行
context_features = {
    "recent_failures": 2,
    "files_touched": 5,
    "tools_used": 4,
    "error_repeated": True,
    "task_continuation": True,
    "previous_plan_exists": True,
}

建议加入这些信号：

信号	意义
最近连续失败 ≥2 次	需要诊断/审计角色
已涉及文件 ≥3 个	可能跨模块
用户连续说“还是不行”	需要回归分析
上一轮做过大改动	需要 reviewer
错误栈变化	需要比较新旧错误
涉及删除/清理/迁移	高风险写操作

这会显著提升真实准确率。

4. 增加“中间模式”，不要只有 single / swarm

现在的二分法太硬：

Python
运行
False = 单智能体
True = 多智能体

建议改成四档：

Python
运行
AgentMode.SINGLE
AgentMode.SINGLE_WITH_REVIEWER
AgentMode.PLAN_EXECUTE
AgentMode.SWARM

对应关系：

模式	适用任务
SINGLE	简单解释、单文件小修、小脚本
SINGLE_WITH_REVIEWER	有风险但不大，需要自检
PLAN_EXECUTE	中等复杂任务，先计划再执行
SWARM	架构审计、跨文件迁移、批量改造、红队评审

很多任务不需要完整多智能体，但需要一个“审查者”。这比直接开 swarm 更省成本。

5. 让多智能体触发后也要可降级

即使命中多智能体，也不应该一开到底。

建议加入：

Python
运行
if swarm_plan_subtasks <= 1:
    downgrade_to_single()

也就是说，多智能体先做一次轻量规划：

是否真的能拆成多个独立子任务？

是否存在互相依赖？

是否需要并行？

是否需要多角色审查？

如果拆不出来，就自动降级。这样可以避免“为了多智能体而多智能体”。

6. 加入失败反馈学习

你已经在 TRM 里做了 record_call() 成功率统计。多智能体也应该有类似机制：

Python
运行
record_agent_mode_result(
    mode="SWARM",
    task_type="refactor",
    success=True,
    latency=18.2,
    user_correction=False,
)

长期记录：

哪些任务开 swarm 成功率更高

哪些任务开 swarm 反而被用户嫌慢

哪些触发词经常误判

哪些短句其实经常变成长链诊断

最终从硬编码规则过渡到经验型路由。

我建议的最终判断公式

可以这样设计：

Python
运行
def should_use_multi_agent(task, session):
    score = 0

    score += keyword_score(task)
    score += length_score(task)
    score += file_scope_score(session)
    score += failure_score(session)
    score += risk_score(task)
    score += ambiguity_score(task)
    score -= simplicity_score(task)

    if destructive_action(task):
        score += 4

    if score >= 8:
        return AgentMode.SWARM

    if score >= 5:
        return AgentMode.PLAN_EXECUTE

    if score >= 3:
        return AgentMode.SINGLE_WITH_REVIEWER

    return AgentMode.SINGLE

这样比单纯返回 True/False 更适合 CRUX。

【最终评分：7.5 / 10】

我给当前 should_use_multi_agent() 设计 7.5 分。

理由：

优点很明确：

默认关闭多智能体是正确方向。

触发词机制简单、可解释、容易调试。

加入阻断词能防止“简单任务过度调度”。

任务 >500 字 作为复杂度信号也合理。

相比原来默认并行，已经是明显架构进步。

扣分点：

仍然偏关键词匹配，语义理解不够。

阻断词如果一票否决，会误伤复杂任务。

没有利用上下文状态，例如连续失败、涉及文件数、历史修改。

没有中间模式，只有单智能体 / 多智能体二分。

没有把“高风险短任务”纳入判断，例如删除、清理、启动逻辑修改。

没有基于实际成功率持续校准。

我的结论是：

这个策略作为 v1 修复非常合理，但它不是最终形态。
真正成熟的 CRUX 多智能体调度，不应该问“要不要多智能体”，而应该问：
这个任务需要哪种协作强度？是否需要审查？是否需要并行？是否需要降级？

推荐下一步不是继续加触发词，而是把它升级为：

Agent Coordination Router：任务复杂度评分 + 风险评分 + 上下文状态 + 动态降级。

当前版本：能止血，能防止滥用，但还不够聪明。
