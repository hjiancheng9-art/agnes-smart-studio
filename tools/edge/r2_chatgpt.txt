下面聚焦 TRM 工具路由网格评审。我的核心判断是：

你的初步修复方向是对的，已经从“关键词匹配工具列表”迈向“有层级、有分类、有反馈的工具路由系统”。
但目前它仍然更像 Tool Recommender，还没有真正升级成 Tool Router / Tool Planner / Tool Governor。

【正方-TRM设计合理/修复方向对】
论点1：ToolTier + TOOL_CATEGORIES 解决了 100+ 工具平铺的第一层混乱

原始 TRM 最大问题是：

用户意图
  ↓
100+ 工具平铺候选
  ↓
关键词/语义粗匹配
  ↓
模型凭感觉选工具

这会导致两个问题：

工具太多，候选空间过大；

工具之间没有层级，基础工具、高危工具、组合工具混在一起。

你现在加入 ToolTier(1-4) 和 TOOL_CATEGORIES(6类分组)，这是正确的第一步。

它至少可以把工具从“平面表”变成“结构化能力池”：

Tier 1：基础安全工具
Tier 2：常用执行工具
Tier 3：组合/高成本工具
Tier 4：高风险/破坏性/外部控制工具

再加上 6 类分组：

文件类
代码类
ComfyUI类
浏览器类
知识检索类
系统执行类

这样 TRM 不再是“100 个工具里猜一个”，而是：

先判断任务域
  ↓
进入工具类别
  ↓
根据风险和成本筛选 Tier
  ↓
再做加权推荐

这个方向非常对。

尤其是 Tier 机制可以解决一个关键问题：

不是所有工具都应该拥有同等调用机会。

例如：

读取文件 ≠ 修改文件
搜索知识 ≠ 执行 shell
生成建议 ≠ 自动 patch
打开浏览器 ≠ 控制浏览器

如果没有 Tier，高风险工具会被模型误触发。
加入 Tier 后，TRM 至少有了“风险分层”的地基。

论点2：suggest_tools + record_call 开始形成反馈闭环，这是从静态路由走向动态路由的关键

你加入 suggest_tools(加权推荐) 和 record_call(成功率统计)，这非常重要。

因为原来的关键词匹配是静态的：

输入里出现 workflow → 推荐 workflow 工具
输入里出现 browser → 推荐 CDP 工具
输入里出现 file → 推荐文件工具

静态匹配的问题是：
它不知道过去哪个工具真的好用。

现在有 record_call 后，TRM 可以开始记录：

哪个任务用了哪个工具
是否成功
失败原因是什么
耗时多少
是否被用户接受

这会让 TRM 从：

我觉得这个工具相关

升级为：

历史上这个工具在类似任务里成功率更高

这一步非常关键。

例如两个工具都能处理 ComfyUI workflow 检查：

workflow_parser
workflow_validator
workflow_diagnostic_agent

关键词上都相关，但历史数据可能显示：

workflow_parser：成功率 92%，低成本
workflow_diagnostic_agent：成功率 78%，高成本
workflow_validator：成功率 88%，中成本

那么 TRM 就不应该只靠关键词，而应该优先选择：

workflow_parser → workflow_validator → diagnostic_agent

也就是说，你的修复已经开始进入“经验路由”。

这是正确方向。

【反方-TRM问题/修复不足】
论点1：ToolTier 和 Category 只是“工具分类”，还不是“任务理解”

反方最大质疑是：

你现在解决的是工具侧结构化，但没有真正解决用户意图侧结构化。

也就是说，TRM 现在可能变成了：

用户输入
  ↓
关键词提取
  ↓
匹配 category
  ↓
按 tier / weight 推荐工具

但真正应该是：

用户输入
  ↓
TaskSpec
  ↓
任务类型
  ↓
输入资产
  ↓
输出目标
  ↓
风险等级
  ↓
执行模式
  ↓
工具链规划

举例：

用户说：

“帮我看看这个工作流为什么跑不起来。”

关键词可能命中：

工作流
ComfyUI
错误
跑不起来

TRM 推荐：

workflow_validator
model_scanner
node_checker

这看起来没错，但仍不够准。
因为系统必须先知道用户到底是哪类任务：

A. 分析 workflow JSON？
B. 检查 ComfyUI 环境？
C. 修复缺失模型？
D. 修复节点连接？
E. 修改工作流文件？
F. 只是解释错误日志？

这些任务虽然关键词相似，但工具链完全不同。

所以反方认为：
ToolTier / Category 是必要补丁，但它只治理工具池，没有治理任务语义。

TRM 要真正准，必须先构建 TaskSpec。

论点2：record_call 成功率统计如果没有上下文归因，可能会误导路由

record_call 是好方向，但简单成功率有风险。

例如某个工具历史成功率很低：

browser_cdp_control：成功率 42%

这并不一定代表工具差。
它可能只是因为它经常被用于更复杂、更不稳定的任务。

再比如：

file_read：成功率 99%
shell_exec：成功率 60%
patch_file：成功率 70%

这不能说明永远应该优先 file_read。
因为工具难度不同，风险不同，任务阶段不同。

所以成功率必须按上下文切片：

按 task_type 统计
按 category 统计
按 tier 统计
按输入完整度统计
按错误类型统计
按模型/执行环境统计
按是否多步骤任务统计

否则会出现错误学习：

高风险工具因为承担复杂任务，成功率低 → 永远被降权
低风险工具因为任务简单，成功率高 → 被过度推荐

这会导致 TRM 变得保守。

还有一个问题：
“调用成功”不等于“任务成功”。

例如：

工具调用成功：读到了文件
任务成功：修好了 bug
用户满意：问题确实解决

这三者不是一回事。

因此 record_call 至少要区分：

tool_call_success
step_success
task_success
user_acceptance

否则成功率统计只能说明工具有没有报错，不能说明工具有没有帮用户完成目标。

【深层建议】如何让 TRM 真正“准”

TRM 真正变准，不是多加几个权重，而是要从“工具推荐器”升级成“任务路由操作系统”。

我建议按下面 7 层升级。

1. 先做 TaskSpec，不要直接 suggest_tools

当前 TRM 的入口不应该是：

Python
运行
suggest_tools(user_text)

而应该是：

Python
运行
task = build_task_spec(user_text, context)
tools = suggest_tools(task)

推荐 TaskSpec：

JSON
{
  "task_type": "comfyui_workflow_debug",
  "domain": "comfyui",
  "intent": "diagnose_and_repair",
  "input_assets": [
    "workflow_json",
    "error_log"
  ],
  "output_target": "patched_workflow_or_diagnosis",
  "risk_level": "medium",
  "requires_file_write": false,
  "requires_browser": false,
  "requires_shell": false,
  "requires_user_confirmation": false,
  "confidence": 0.82
}

然后 TRM 才根据 TaskSpec 推荐工具。

这一步是分水岭。

没有 TaskSpec，TRM 永远在猜关键词。
有 TaskSpec，TRM 才是在匹配能力。

2. 建立 Tool Manifest，而不是只靠 Category / Tier

每个工具都应该有机器可读的工具契约。

示例：

JSON
{
  "tool_id": "workflow_validator",
  "name": "ComfyUI Workflow Validator",
  "category": "comfyui",
  "tier": 2,
  "task_types": [
    "comfyui_workflow_debug",
    "comfyui_workflow_validate",
    "comfyui_workflow_compile"
  ],
  "inputs_required": [
    "workflow_json"
  ],
  "inputs_optional": [
    "model_inventory",
    "node_registry",
    "error_log"
  ],
  "outputs": [
    "validation_report",
    "error_list",
    "repair_suggestions"
  ],
  "preconditions": [
    "workflow_json exists",
    "workflow_json contains nodes"
  ],
  "anti_patterns": [
    "do not use for pure prompt writing",
    "do not use when user only asks for architectural explanation"
  ],
  "risk_level": "low",
  "cost_level": "low",
  "side_effects": "none"
}

重点是 anti_patterns。
TRM 路由不准，很多时候不是因为不知道该用什么，而是不知道不该用什么。

3. suggest_tools 不应只推荐工具，应推荐“工具链”

单工具推荐不够。

很多任务天然是多步骤的：

代码修复：
read_file → search_symbols → patch_file → run_tests

ComfyUI 修复：
parse_workflow → validate_graph → check_models → suggest_patch

浏览器自动化：
open_page → inspect_dom → click/type → verify_result

所以 suggest_tools 应该升级为：

Python
运行
suggest_tool_plan(task_spec)

输出不只是：

JSON
["workflow_validator", "model_scanner"]

而是：

JSON
{
  "plan": [
    {
      "step": 1,
      "tool": "workflow_parser",
      "reason": "先解析 workflow JSON，确认节点结构。"
    },
    {
      "step": 2,
      "tool": "workflow_validator",
      "reason": "检查拓扑、输入输出类型、节点合法性。"
    },
    {
      "step": 3,
      "tool": "model_inventory_checker",
      "reason": "确认模型文件是否存在且类型匹配。"
    }
  ],
  "fallback": [
    {
      "if": "workflow_json_missing",
      "action": "ask_for_file_or_search_project"
    },
    {
      "if": "model_inventory_unavailable",
      "action": "run_model_scan"
    }
  ]
}

真正的 TRM 不是选工具，而是编排工具。

4. 加入 Fallback Graph，而不是失败后临时想办法

你提到“无失败降级机制”，这是核心缺口。

建议每个工具都定义 fallback：

JSON
{
  "tool_id": "workflow_validator",
  "fallbacks": [
    {
      "on_error": "missing_workflow_json",
      "next": "file_search_workflow"
    },
    {
      "on_error": "unknown_node_class",
      "next": "node_registry_lookup"
    },
    {
      "on_error": "model_missing",
      "next": "model_inventory_scan"
    },
    {
      "on_error": "validation_timeout",
      "next": "lightweight_json_lint"
    }
  ]
}

最终 TRM 应该有一张失败降级图：

主工具失败
  ↓
错误分类
  ↓
低成本 fallback
  ↓
替代工具
  ↓
人工确认 / 停止

这样系统才不会一失败就崩。

5. 成功率统计要从“工具级”升级到“任务-工具-上下文级”

不要只记录：

JSON
{
  "tool": "workflow_validator",
  "success": true
}

要记录：

JSON
{
  "task_type": "comfyui_workflow_debug",
  "tool": "workflow_validator",
  "stage": "validation",
  "input_quality": "complete",
  "error_type": null,
  "tool_call_success": true,
  "step_success": true,
  "task_success": true,
  "user_accepted": true,
  "latency_ms": 820,
  "cost_level": "low"
}

然后评分时按上下文计算：

score =
  semantic_match * 0.35
+ task_type_match * 0.25
+ historical_success_in_context * 0.20
+ input_readiness * 0.10
+ cost_risk_penalty * 0.10

其中历史成功率必须是：

相同 task_type + 相同 stage + 相似输入条件下的成功率

不要用全局成功率。

6. 建立 “No Tool” 路由

这是很多工具系统忽略的点。

TRM 不准的一个表现是：
明明不需要工具，它也推荐工具。

例如用户让你：

评审架构
解释方法论
写一段文案
比较两个方案
给出设计建议

这些任务可能根本不需要工具。

所以 TRM 必须有一个特殊工具：

NO_TOOL

它不是消极选项，而是正式路由结果。

示例：

JSON
{
  "route": "NO_TOOL",
  "reason": "这是架构评审任务，用户已提供足够上下文，不需要外部工具。",
  "confidence": 0.91
}

这会明显降低乱调用。

7. 建立 TRM 路由评测集

最后，TRM 准不准必须可测。

建议先做 200 条黄金样本：

40 条代码修复
40 条 ComfyUI 工作流
30 条图片/视频创作
20 条浏览器自动化
20 条知识检索
20 条文件分析
20 条架构评审 / 不需工具
10 条高风险拒绝 / 需要确认

每条样本包含：

JSON
{
  "case_id": "trm_001",
  "user_input": "帮我检查这个 ComfyUI workflow 为什么报 Missing node",
  "expected_task_type": "comfyui_workflow_debug",
  "expected_category": "comfyui",
  "expected_tools": [
    "workflow_parser",
    "node_registry_lookup",
    "workflow_validator"
  ],
  "forbidden_tools": [
    "browser_cdp",
    "image_generation",
    "shell_exec"
  ],
  "expected_fallback": [
    "extension_installer"
  ]
}

然后每次改 TRM，都跑：

Top-1 route accuracy
Top-3 route accuracy
Forbidden tool rate
No-tool accuracy
Fallback success rate
Average tool count
Unnecessary tool call rate
Task success rate

没有评测集，TRM 的“准”只能靠感觉。

推荐的 TRM v2 架构

我建议你把 TRM 从当前结构：

Tool Registry
  ↓
Category / Tier
  ↓
suggest_tools
  ↓
record_call

升级为：

User Intent
  ↓
TaskSpec Builder
  ↓
Task Classifier
  ↓
Skill / Category Router
  ↓
Tool Manifest Filter
  ↓
Tool Plan Generator
  ↓
Contract Checker
  ↓
Executor
  ↓
Fallback Graph
  ↓
Telemetry / record_call
  ↓
Routing Policy Update

更简洁地说：

先定任务，再选能力；
先选技能，再选工具；
先查契约，再执行；
失败可降级，成功可学习。
【最终评分1-10分】对当前TRM设计评分
当前修复后评分：6.8 / 10

这个分数的含义是：

已经从“原始关键词工具表”进化到“初级工具路由系统”，但还没有达到“可靠的能力路由中枢”。

加分点
1. ToolTier 是正确的

它解决了工具风险和工具能力层级问题。
这是治理 100+ 工具的必要地基。

2. TOOL_CATEGORIES 是正确的

它解决了工具平铺问题，让候选召回空间变小。

3. suggest_tools 是必要入口

它让 TRM 有了统一推荐函数，而不是让各处随意挑工具。

4. record_call 是关键方向

它开始形成历史反馈，为动态路由打基础。

扣分点
1. 仍然偏关键词匹配

只要入口还是 user_text → tools，就很难真正准。
必须变成 user_text → TaskSpec → tools。

2. 没有 Tool Manifest / Tool Contract

Tier 和 Category 太粗。
工具真正能不能用，要看输入、输出、前置条件、禁用场景和副作用。

3. 成功率统计粒度不够

全局成功率会误导路由。
必须按任务类型、阶段、输入质量、错误类型切片。

4. 没有工具链规划

很多任务不是选一个工具，而是选一条工具链。

5. 没有 fallback graph

失败降级不能靠临时 prompt 想，必须工程化。

最终裁决

你的修复方向是对的，但目前只是 TRM 1.5，还不是 TRM 2.0。

现在的 TRM 可以概括为：

分类 + 分层 + 加权推荐 + 成功率记录。

真正准的 TRM 应该是：

任务理解 + 能力召回 + 工具契约 + 工具链规划 + 失败降级 + 上下文成功率学习。

下一步最优先做这三件：

1. TaskSpec Builder：先把用户意图结构化
2. Tool Manifest：给每个工具补输入/输出/前置/禁用/风险
3. Fallback Graph：让工具失败后有固定降级路径

做完这三项，TRM 可以从 6.8 分提升到 8 分以上。
